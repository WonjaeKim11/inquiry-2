# 기능 명세서: 접근 제어 및 데이터 프리필

## 1. 문서 정보

| 항목 | 내용 |
|------|------|
| 문서 번호 | FS-018 |
| 문서 버전 | 1.0 |

| 라이선스 | Community |
| 상태 | 초안 |

## 2. 개요

### 2.1 목적

Link Survey는 URL만으로 누구나 접근 가능하므로, 설문 접근을 제한하고 응답의 신뢰성을 확보하기 위한 메커니즘을 정의한다. 본 문서는 PIN 보호, 이메일 인증, Survey Close Message 등의 접근 제어 기능과, URL 파라미터를 통한 데이터 프리필, 시작 위치 지정, 유입 경로 추적 등의 기능을 상세히 기술한다.

### 2.2 범위

**포함 범위:**
- 4자리 PIN 보호 기능
- 이메일 인증 플로우 (JWT 토큰 기반)
- 단일 이메일 응답 제한
- URL 파라미터 기반 데이터 프리필
- startAt 파라미터로 특정 질문부터 시작
- source 파라미터로 유입 경로 추적
- skipPrefilled 파라미터로 프리필된 질문 건너뛰기
- Hidden Fields URL 파라미터 처리
- Survey Close Message 커스터마이징
- 설문 비활성 상태(paused, completed) 표시

**제외 범위:**
- Single-use 링크 (FS-017 참조)
- Personal Link (FS-017 참조)
- reCAPTCHA 스팸 방지 (FS-020 참조)

### 2.3 대상 사용자

| 역할 | 설명 |
|------|------|
| Survey Creator | PIN 설정, 이메일 인증 활성화, 프리필 URL 구성을 수행하는 설문 생성자 |
| Survey Respondent | PIN 입력, 이메일 인증 후 설문에 응답하는 최종 사용자 |
| 마케팅 담당자 | source 파라미터를 활용하여 유입 경로별 응답을 분석하는 담당자 |
| CRM 통합 시스템 | 프리필 URL을 자동 생성하여 고객에게 발송하는 외부 시스템 |

### 2.4 용어 정의

| 용어 | 정의 |
|------|------|
| PIN | 4자리 숫자로 구성된 설문 접근 보호 코드 |
| 프리필 (Prefill) | URL 파라미터를 통해 설문 질문에 미리 값을 채우는 기능 |
| Hidden Fields | 응답자에게 보이지 않지만 응답 데이터에 포함되는 숨겨진 필드 |
| startAt | 설문 시작 위치를 지정하는 URL 파라미터 |
| skipPrefilled | 프리필된 질문을 건너뛸지 여부를 지정하는 URL 파라미터 |
| source | 유입 경로를 추적하기 위한 URL 파라미터 |
| OTP 입력 컴포넌트 | 각 자릿수를 개별 입력 필드로 분리한 입력 UI 컴포넌트 |
| JWT (JSON Web Token) | 이메일 인증에 사용되는 토큰 형식 |
| Rate Limiting | 특정 시간 내 요청 수를 제한하는 보호 메커니즘 |
| Survey Close Message | 설문이 닫힌 상태에서 표시되는 커스터마이징 가능한 메시지 |
| Element | 설문 내의 개별 질문 또는 구성 요소 |
| Welcome Card | 설문 시작 시 표시되는 환영 카드 화면 |
| fishy | 이메일 인증 토큰이 존재하지만 이메일 추출에 실패한 비정상 상태 |

## 4. 상세 기능 명세

---

### 4.1 PIN 보호

#### 4.1.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-018-01 |
| 기능명 | PIN 보호 (4자리) |
| 관련 요구사항 ID | FR-027-01 |
| 우선순위 | 높음 |
| 기능 설명 | 설문 접근 시 4자리 숫자 PIN 입력을 요구하여 비인가 접근을 차단한다 |

#### 4.1.2 선행 조건

- 설문이 시스템에 존재해야 한다.
- Survey Creator가 해당 설문에 PIN을 설정한 상태여야 한다.
- PIN은 정확히 4자리 숫자로 구성되어야 한다.

#### 4.1.3 후행 조건

- PIN 검증 성공 시: 설문 데이터가 반환되고, 설문이 렌더링된다.
- PIN 검증 실패 시: 에러 메시지가 표시되고, 2초 후 입력란이 초기화된다.

#### 4.1.4 기본 흐름

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | 설문 로드 시 PIN 설정 여부를 확인한다 |
| 2 | 시스템 | PIN이 설정되어 있으면 PIN 입력 화면을 렌더링한다 |
| 3 | 사용자 | OTP 입력 컴포넌트를 통해 4자리 PIN을 입력한다 |
| 4 | 시스템 | 4자리 입력이 완료되면 자동으로 PIN 검증을 요청한다 (수동 제출 불필요) |
| 5 | 시스템 | 설문 ID와 입력된 PIN을 서버에 전달하여 검증한다 |
| 6 | 서버 | 해당 설문의 저장된 PIN과 입력된 PIN을 비교한다 |
| 7 | 서버 | PIN이 일치하면 설문 데이터를 반환한다 |
| 8 | 시스템 | 반환된 설문 데이터로 설문을 렌더링한다 |

#### 4.1.5 대안 흐름

**AF-01: PIN 미설정 설문**

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | 설문 로드 시 PIN 설정 여부를 확인한다 |
| 2 | 시스템 | PIN이 미설정된 설문이면 검증 없이 통과하여 설문을 바로 렌더링한다 |

#### 4.1.6 예외 흐름

**EF-01: 잘못된 PIN 입력**

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 서버 | PIN이 불일치하면 "INVALID_PIN" 에러를 반환한다 |
| 2 | 시스템 | 에러 메시지를 화면에 표시한다 |
| 3 | 시스템 | 2초(2000ms) 후 입력란을 자동 초기화한다 |
| 4 | 사용자 | PIN을 다시 입력할 수 있다 |

**EF-02: 존재하지 않는 설문**

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 서버 | 설문 ID에 해당하는 설문이 없으면 에러를 반환한다 |
| 2 | 시스템 | 설문을 찾을 수 없다는 에러 화면을 표시한다 |

#### 4.1.7 비즈니스 규칙

| 규칙 ID | 규칙 |
|---------|------|
| BR-01-01 | PIN은 정확히 4자리 숫자(0000~9999)만 허용한다 |
| BR-01-02 | 4자리 입력 완료 시 자동으로 검증 요청이 발생한다 (별도 제출 버튼 없음) |
| BR-01-03 | PIN 불일치 시 에러 표시 후 정확히 2초(2000ms) 후 입력란이 자동 초기화된다 |
| BR-01-04 | PIN 재시도 횟수에 제한은 없다 |

#### 4.1.8 데이터 요구사항

**입력 데이터:**

| 필드명 | 타입 | 필수 | 유효성 검증 규칙 |
|--------|------|------|----------------|
| surveyId | string | 필수 | 유효한 설문 ID |
| pin | string | 필수 | 정확히 4자리 숫자 (정규식: `^\d{4}$`) |

**출력 데이터:**

| 필드명 | 타입 | 설명 |
|--------|------|------|
| survey | SurveyObject | PIN 검증 성공 시 반환되는 설문 전체 데이터 |
| error | string | PIN 검증 실패 시 "INVALID_PIN" 에러 코드 |

#### 4.1.9 화면/UI 요구사항

| 항목 | 요구사항 |
|------|---------|
| 입력 컴포넌트 | OTP 스타일 4자리 입력 (각 자릿수별 개별 입력 필드) |
| 자동 포커스 | 첫 번째 입력 필드에 자동 포커스 (단, iframe 내에서는 비활성화) |
| 자동 검증 | 4자리 모두 입력 완료 시 자동으로 검증 요청 |
| 에러 표시 | 입력 필드 하단 또는 근처에 에러 메시지 표시 |
| 자동 초기화 | 에러 표시 2초 후 모든 입력 필드를 비우고 첫 번째 필드에 포커스 |

#### 4.1.10 비기능 요구사항

| ID | 항목 | 기준 |
|----|------|------|
| NFR-01-01 | UX 반응성 | 4자리 입력 완료 즉시 검증 요청 발생 |
| NFR-01-02 | 에러 초기화 타이밍 | 정확히 2초(2000ms) 후 자동 초기화 |
| NFR-01-03 | iframe 호환성 | iframe 내에서는 autofocus 비활성화 |

### 4.2 이메일 인증

#### 4.2.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-018-02 |
| 기능명 | 이메일 인증 |
| 관련 요구사항 ID | FR-027-02 |
| 우선순위 | 높음 |
| 기능 설명 | 설문 응답 전 이메일 인증을 요구하여 응답자 신원을 확인하고, 선택적으로 단일 이메일 응답 제한을 적용한다 |

#### 4.2.2 선행 조건

- 설문이 시스템에 존재해야 한다.
- Survey Creator가 해당 설문에 이메일 인증을 활성화한 상태여야 한다.
- 이메일 발송 시스템이 정상 동작해야 한다.

#### 4.2.3 후행 조건

- 인증 성공 시: 인증된 이메일이 포함된 상태로 설문이 표시된다.
- 인증 성공 시: verifiedEmail 키로 Hidden Fields에 자동 추가된다 (FN-018-08 연계).
- 인증 실패 시: "This looks fishy" 에러 화면이 표시된다.

#### 4.2.4 기본 흐름

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | 설문 로드 시 이메일 인증 활성화 여부를 확인한다 |
| 2 | 시스템 | 이메일 입력 화면을 렌더링한다 (이메일 주소 입력 필드 + "Verify" 버튼) |
| 3 | 사용자 | 이메일 주소를 입력하고 "Verify" 버튼을 클릭한다 |
| 4 | 시스템 | 입력된 이메일의 형식 유효성을 검증한다 |
| 5 | 서버 | 인증 이메일을 발송한다 (설문 ID, 이메일, 설문 이름, Single-use ID, 로케일 포함) |
| 6 | 시스템 | "Survey sent to {email}" 메시지를 표시한다 |
| 7 | 사용자 | 이메일에서 인증 링크를 클릭한다 |
| 8 | 시스템 | URL의 verify 파라미터에서 JWT 토큰을 추출한다 |
| 9 | 서버 | JWT 토큰을 검증하고 이메일을 추출한다 |
| 10 | 서버 | 검증 성공 시 인증 상태를 "verified"로 변경한다 |
| 11 | 시스템 | 인증된 이메일을 포함하여 설문을 표시한다 |

#### 4.2.5 대안 흐름

**AF-01: 질문 미리보기 ("Just Curious?")**

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 사용자 | 이메일 입력 화면에서 "Just Curious?" 옵션을 선택한다 |
| 2 | 시스템 | 인증 없이 질문 목록 미리보기를 표시한다 |
| 3 | 시스템 | 실제 응답은 불가하며, 목록 확인만 가능하다 |

#### 4.2.6 예외 흐름

**EF-01: 단일 이메일 응답 제한 위반**

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | 단일 이메일 응답 제한이 활성화된 경우, 이메일 입력 시 기존 응답 여부를 확인한다 |
| 2 | 서버 | 해당 이메일로 이미 응답이 존재하면 에러를 반환한다 |
| 3 | 시스템 | "Response already received" 에러 메시지를 표시한다 |
| 4 | 시스템 | 설문 접근을 차단한다 |

**EF-02: 인증 토큰 검증 실패 (fishy 상태)**

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 서버 | verify 파라미터의 토큰이 존재하지만 이메일 추출에 실패한다 |
| 2 | 서버 | 인증 상태를 "fishy"로 설정한다 |
| 3 | 시스템 | "This looks fishy" 에러 화면을 표시한다 |
| 4 | 시스템 | 재시도 안내를 표시한다 |

**EF-03: Rate Limiting 초과**

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 서버 | 동일 IP에서 허용 한도를 초과한 인증 이메일 발송 요청이 들어온다 |
| 2 | 서버 | Rate Limit 초과 에러를 반환한다 |
| 3 | 시스템 | 사용자에게 잠시 후 재시도할 것을 안내한다 |

#### 4.2.7 비즈니스 규칙

| 규칙 ID | 규칙 |
|---------|------|
| BR-02-01 | 이메일 인증 상태는 "not-verified", "verified", "fishy" 3가지 중 하나이다 |
| BR-02-02 | 단일 이메일 응답 제한이 활성화된 경우, 동일 이메일로 2회 이상 응답할 수 없다 |
| BR-02-03 | 인증 이메일 발송에는 IP 기반 Rate Limiting이 적용된다 |
| BR-02-04 | 인증 토큰은 JWT 형식이다 |
| BR-02-05 | 인증 성공 시 verifiedEmail이 Hidden Fields에 자동 주입된다 |

### 4.4 데이터 프리필

#### 4.4.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-018-04 |
| 기능명 | 데이터 프리필 |
| 관련 요구사항 ID | FR-028-01, FR-028-02 |
| 우선순위 | 높음 |
| 기능 설명 | URL 파라미터를 통해 설문 질문에 미리 값을 채우고, Element 타입별 검증 규칙에 따라 유효한 값만 적용한다 |

#### 4.4.2 선행 조건

- 설문이 시스템에 존재해야 한다.
- 설문 URL에 프리필 파라미터가 포함되어 있어야 한다.
- 프리필 대상 Element(질문)이 설문 내에 존재해야 한다.

#### 4.4.3 후행 조건

- 유효한 프리필 값은 해당 질문에 미리 입력된 상태로 표시된다.
- 유효하지 않은 프리필 값은 무시되며, 해당 질문은 빈 상태로 표시된다.
- 하나의 프리필 오류가 다른 프리필에 영향을 주지 않는다 (오류 격리).

#### 4.4.4 기본 흐름

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | URL의 모든 쿼리 파라미터를 순회한다 |
| 2 | 시스템 | 예약된 파라미터명(startAt, embed, preview 등)인지 확인하고, 예약된 파라미터는 건너뛴다 |
| 3 | 시스템 | 파라미터 키와 일치하는 설문 Element(질문)을 검색한다 |
| 4 | 시스템 | 일치하는 Element가 없으면 해당 파라미터를 건너뛴다 |
| 5 | 시스템 | 일치하는 Element가 있으면 Element 타입에 따라 값 검증을 수행한다 |
| 6 | 시스템 | 필수 필드에 빈 값이 전달되면 검증 실패로 처리한다 |
| 7 | 시스템 | 검증 통과 시 값을 해당 타입에 맞게 변환한다 |
| 8 | 시스템 | 변환된 값을 프리필 데이터에 추가한다 |
| 9 | 시스템 | 모든 파라미터 처리 완료 후, 프리필 데이터를 설문 렌더링에 적용한다 |

#### 4.4.5 대안 흐름

**AF-01: Hidden Fields 파라미터 처리**

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | URL 파라미터 키가 설문의 Hidden Fields에 정의된 키와 일치하는지 확인한다 |
| 2 | 시스템 | 일치하면 해당 값을 Hidden Fields에 매핑한다 |
| 3 | 시스템 | Hidden Fields에 정의되지 않은 키는 무시한다 |

#### 4.4.6 예외 흐름

**EF-01: 프리필 검증 실패 (오류 격리)**

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | 특정 파라미터의 프리필 검증이 실패한다 |
| 2 | 시스템 | 해당 파라미터만 무시하고, 나머지 파라미터 처리를 계속한다 |
| 3 | 시스템 | 검증 실패한 질문은 빈 상태로 표시된다 |

**EF-02: 알 수 없는 파라미터**

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | URL 파라미터 키에 매칭되는 Element가 없고, Hidden Fields에도 정의되지 않은 키이다 |
| 2 | 시스템 | 해당 파라미터를 조용히 무시한다 (에러 발생 없음) |

#### 4.4.7 비즈니스 규칙

| 규칙 ID | 규칙 |
|---------|------|
| BR-04-01 | 예약된 파라미터명(startAt, embed, preview 등)은 프리필 처리에서 제외한다 |
| BR-04-02 | 하나의 프리필 오류가 다른 프리필에 영향을 주지 않는다 (오류 격리) |
| BR-04-03 | 필수 필드에 빈 값이 전달되면 프리필을 무시한다 |
| BR-04-04 | 알 수 없는 파라미터는 에러 없이 무시한다 |
| BR-04-05 | 라벨 기반 프리필과 Option ID 기반 프리필을 모두 지원한다 (하위 호환성) |

**Element 타입별 검증 규칙:**

| Element Type | 유효 값 | 검증 세부사항 | 프리필 URL 예시 |
|-------------|---------|-------------|----------------|
| OpenText | 임의 문자열 | 항상 유효 | `?questionId=answer` |
| MultipleChoiceSingle | Option ID 또는 라벨 문자열 | Option ID(`option-*` 형식) 또는 라벨로 옵션 매칭, "other" 값 지원 | `?questionId=option-abc123` |

| NPS | 0 ~ 10 정수 | 정수 파싱 후 0 이상 10 이하 범위 검사 | `?npsId=9` |
| Rating | 1 ~ 설정된 범위(기본 5) 정수 | 정수 파싱 후 1 이상 설정 범위 이하 검사 | `?ratingId=4` |

| 우선순위 | 중간 |
| 기능 설명 | URL의 startAt 파라미터를 통해 설문 시작 위치를 특정 질문으로 지정한다 |

| 3 | 시스템 | startAt 값이 Element ID인 경우 해당 Element의 존재를 확인한다 |

### 4.6 skipPrefilled 파라미터

#### 4.6.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-018-06 |
| 기능명 | skipPrefilled 파라미터 |
| 관련 요구사항 ID | FR-028-04 |
| 우선순위 | 중간 |
| 기능 설명 | URL의 skipPrefilled 파라미터를 통해 프리필된 값이 있는 질문을 자동으로 건너뛸지 여부를 제어한다 |

#### 4.6.2 선행 조건

- 설문이 시스템에 존재해야 한다.
- 프리필 데이터가 적용된 상태여야 한다.

#### 4.6.3 후행 조건

- skipPrefilled=true인 경우: 프리필된 질문이 자동으로 건너뛰어지고, 프리필되지 않은 첫 번째 질문부터 표시된다.
- skipPrefilled=false(기본값)인 경우: 프리필된 값이 표시되고 사용자가 확인/수정할 수 있다.

#### 4.6.4 기본 흐름

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | URL에서 skipPrefilled 파라미터 값을 확인한다 |
| 2 | 시스템 | skipPrefilled가 "true" 문자열이면 스킵 모드를 활성화한다 |
| 3 | 시스템 | 프리필 데이터가 적용된 질문을 자동으로 건너뛴다 |
| 4 | 시스템 | 프리필되지 않은 첫 번째 질문을 표시한다 |

#### 4.6.5 대안 흐름

**AF-01: skipPrefilled 미설정 또는 false**

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | skipPrefilled가 설정되지 않았거나 "true"가 아닌 값이면 기본 동작을 수행한다 |
| 2 | 시스템 | 프리필된 값을 해당 질문에 표시하고, 사용자가 확인/수정할 수 있도록 한다 |

#### 4.6.6 예외 흐름

해당 없음.

#### 4.6.7 비즈니스 규칙

| 규칙 ID | 규칙 |
|---------|------|
| BR-06-01 | skipPrefilled는 "true" 문자열로만 활성화된다 (대소문자 구분) |
| BR-06-02 | 기본값은 false (프리필된 값 표시, 사용자 수정 가능)이다 |
| BR-06-03 | skipPrefilled는 프리필 데이터가 존재할 때만 의미가 있다 |

#### 4.6.8 데이터 요구사항

**입력 데이터:**

| 필드명 | 타입 | 필수 | 유효성 검증 규칙 |
|--------|------|------|----------------|
| skipPrefilled | string | 선택 | "true" 문자열로 활성화, 그 외 값은 false로 처리 |

#### 4.6.9 화면/UI 요구사항

| 항목 | 요구사항 |
|------|---------|
| 스킵 동작 | skipPrefilled=true 시, 프리필된 질문은 화면에 표시되지 않고 자동으로 건너뜀 |
| 기본 동작 | skipPrefilled=false 시, 프리필된 값이 미리 입력된 상태로 질문이 표시됨 |

#### 4.6.10 비기능 요구사항

해당 없음.

### 4.7 source 파라미터 (유입 경로 추적)

| 관련 요구사항 ID | FR-028-05 |
| 우선순위 | 중간 |
| 기능 설명 | URL의 source 파라미터를 Hidden Fields를 통해 자동 매핑하여 유입 경로를 추적한다 |

#### 4.7.2 선행 조건

| 2 | 시스템 | 설문의 Hidden Fields에 "source" 키가 정의되어 있는지 확인한다 |
| 3 | 시스템 | Hidden Fields에 "source" 키가 존재하면, 파라미터 값을 해당 필드에 매핑한다 |
| 4 | 시스템 | 응답 제출 시 Hidden Fields 데이터에 source 값이 포함된다 |

| 2 | 시스템 | source 파라미터를 무시한다 (에러 발생 없음) |

| 엔티티 | 설명 | 주요 속성 |

| EmailVerification | 이메일 인증 정보 | email, token, status, surveyId |
| SurveyResponse | 설문 응답 | id, surveyId, data, hiddenFieldData, verifiedEmail |

### 5.2 엔티티 간 관계

```
Survey (1) ──── (N) SurveyElement
Survey (1) ──── (0..1) SurveyCloseMessage
Survey (1) ──── (0..1) HiddenFields
Survey (1) ──── (N) SurveyResponse
Survey (1) ──── (N) EmailVerification
SurveyResponse (1) ──── (0..N) PrefillData
```

### 5.3 데이터 흐름

```
[URL 파라미터]
     |
     ├── startAt ──────────> [시작 위치 결정]
     ├── skipPrefilled ────> [스킵 동작 결정]
     ├── source ───────────> [Hidden Fields 매핑]
     ├── elementId=value ──> [프리필 검증 엔진] ──> [PrefillData]
     └── hiddenKey=value ──> [Hidden Fields 매핑]
                                     |
                                     v
                              [설문 렌더링]
                                     |
                                     v
                              [응답 데이터]
                               ├── data (프리필 포함)
                               ├── hiddenFieldData (source, verifiedEmail 포함)
                               └── meta
```
