## 2. 개요

### 2.1 목적

관리자 UI의 다국어(i18n) 시스템에 대한 상세 기능 명세를 정의한다. react-i18next 기반의 국제화 시스템을 통해 14개 언어를 지원하며, 서버 사이드와 클라이언트 사이드 모두에서 일관된 번역을 제공한다.

### 2.2 범위

- 관리자 UI 14개 언어 번역 체계
- 서버 사이드 및 클라이언트 사이드 번역 처리
- Accept-Language 헤더 기반 로케일 자동 감지
- 사용자 프로필 로케일 설정 및 저장
- ICU Message Format 지원
- 번역 키 명명 규칙 및 번역 파일 관리
- 번역 검증 도구
- 설문 UI 다국어 유틸리티 (i18n-utils, i18nString)

범위 제외:
- 설문 콘텐츠 다국어 지원 (Multi-language Surveys, Enterprise 기능)
- RTL(Right-to-Left) 언어 레이아웃
- 자동 번역 시스템

### 2.3 대상 사용자

| 역할 | 설명 |
|------|------|
| 모든 사용자 | 관리자 UI 언어를 프로필에서 설정할 수 있는 대상 |
| 비로그인 방문자 | 브라우저 Accept-Language 헤더에 따라 자동으로 로케일이 적용되는 대상 |
| 개발자 | 번역 키를 추가하고 번역 파일을 관리하는 대상 |
| 번역 기여자 | 오픈소스 방식으로 번역을 기여하는 대상 |

### 2.4 용어 정의

| 용어 | 정의 |
|------|------|
| 로케일 (Locale) | 언어 및 지역을 식별하는 코드 (예: `en-US`, `de-DE`) |
| i18n | Internationalization의 약어. 소프트웨어의 다국어 지원 체계 |
| i18next | JavaScript용 국제화 프레임워크 라이브러리 |
| react-i18next | i18next의 React 바인딩 라이브러리 |
| ICU Message Format | Unicode 국제화 컴포넌트에서 정의한 메시지 포맷 표준. 복수형, 성별, 변수 보간 등 지원 |
| 폴백 (Fallback) | 현재 로케일에 번역이 없을 때 대체 로케일(en-US)의 번역을 사용하는 메커니즘 |
| Accept-Language | HTTP 요청 헤더. 클라이언트가 선호하는 언어 목록을 서버에 전달 |
| 번역 키 (Translation Key) | 번역 텍스트를 식별하는 고유 문자열 (예: `common.save`) |
| i18nString | 설문 콘텐츠의 다국어 문자열을 표현하는 객체 타입. `default` 키와 언어 코드별 번역 값을 포함 |
| 서버 번역 함수 | 서버 사이드에서 i18next 인스턴스를 생성하여 번역을 처리하는 함수 |
| useTranslation | react-i18next가 제공하는 React hook. 클라이언트 컴포넌트에서 번역을 사용 |

## 4. 상세 기능 명세

### 4.1 지원 언어 관리

#### 4.1.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-030-01 |
| 기능명 | 지원 언어 관리 |
| 관련 요구사항 ID | FR-049-01 |
| 우선순위 | 필수 |
| 기능 설명 | 관리자 UI가 지원하는 14개 언어의 로케일 코드, 번역 파일, 기본 로케일을 정의하고 관리한다 |

#### 4.1.2 선행 조건

- 14개 로케일 각각에 대응하는 JSON 번역 파일이 프로젝트 내에 존재해야 한다
- 지원 로케일 목록이 소스 코드 내 배열로 정의되어야 한다

#### 4.1.3 후행 조건

- 시스템이 14개 로케일 중 하나로 UI를 렌더링할 수 있다
- 기본 로케일(`en-US`)은 모든 번역 키에 대해 100% 번역이 완료된 상태이다

#### 4.1.4 기본 흐름

1. 시스템은 지원 로케일 배열에서 14개 로케일 코드를 관리한다
2. 각 로케일 코드에 대응하는 JSON 번역 파일이 프로젝트 파일 시스템에 존재한다
3. 기본 로케일은 `en-US`로 설정된다
4. 폴백 로케일은 `en-US`로 설정된다

#### 4.1.5 대안 흐름

- 해당 없음

### 4.2 사용자 로케일 저장

#### 4.2.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-030-02 |
| 기능명 | 사용자 로케일 저장 |
| 관련 요구사항 ID | FR-049-02 |
| 우선순위 | 필수 |
| 기능 설명 | 사용자가 선택한 로케일을 User 모델의 locale 필드에 저장하고, 프로필 설정에서 변경할 수 있다 |

#### 4.2.2 선행 조건

- 사용자가 로그인된 상태여야 한다
- 지원 로케일 목록(FN-030-01)이 정의되어 있어야 한다

#### 4.2.3 후행 조건

- User 모델의 `locale` 필드에 선택한 로케일 코드가 저장된다
- 이후 해당 사용자의 모든 요청에서 저장된 로케일이 적용된다

#### 4.2.4 기본 흐름

1. 사용자가 프로필 설정 화면에 접근한다
2. 시스템이 언어 선택 드롭다운에 14개 지원 언어 목록을 표시한다 (FN-030-10 참조)
3. 사용자가 원하는 언어를 선택한다
4. 시스템이 선택된 로케일 코드가 지원 로케일 목록에 포함되는지 검증한다
5. 시스템이 User 모델의 `locale` 필드를 선택된 로케일 코드로 업데이트한다
6. 시스템이 현재 UI 언어를 변경된 로케일로 즉시 전환한다

#### 4.2.5 대안 흐름

| 대안 흐름 | 조건 | 처리 |
|----------|------|------|
| AF-02-01 | 신규 사용자 회원가입 시 | `locale` 필드의 기본값으로 `en-US`가 저장된다 |

#### 4.2.6 예외 흐름

| 예외 | 조건 | 처리 |
|------|------|------|
| EX-02-01 | 지원하지 않는 로케일 코드가 전달된 경우 | 변경을 거부하고 기존 로케일을 유지한다 |
| EX-02-02 | 로케일 저장 중 DB 오류 발생 | 오류 메시지를 사용자에게 표시하고 기존 로케일을 유지한다 |

#### 4.2.7 비즈니스 규칙

| 규칙 ID | 규칙 |
|---------|------|
| BR-02-01 | `locale` 필드의 기본값은 `en-US`이다 |
| BR-02-02 | `locale` 필드에 저장 가능한 값은 14개 지원 로케일 코드로 제한된다 |
| BR-02-03 | 로케일 변경 시 UI 언어가 즉시 전환되어야 한다 (페이지 새로고침 없이 또는 최소한의 갱신으로) |

#### 4.2.8 데이터 요구사항

**출력 데이터**

| 필드명 | 타입 | 설명 |
|--------|------|------|
| locale | String | 업데이트된 사용자 로케일 코드 |

**User 모델 locale 필드**

| 속성 | 값 |
|------|-----|
| 필드명 | locale |

#### 4.2.9 화면/UI 요구사항

- 프로필 설정 화면에 언어 선택 드롭다운이 포함되어야 한다

#### 4.2.10 비기능 요구사항

### 4.3 로케일 결정 로직

#### 4.3.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-030-03 |
| 기능명 | 로케일 결정 로직 |
| 관련 요구사항 ID | FR-049-03 |
| 우선순위 | 필수 |
| 기능 설명 | 요청별로 적용할 로케일을 결정한다. 로그인 사용자는 DB 저장 로케일, 비로그인 사용자는 Accept-Language 헤더 기반 감지, 최종 폴백은 en-US를 사용한다 |

#### 4.3.2 선행 조건

- HTTP 요청이 서버에 도달한 상태이다

#### 4.3.3 후행 조건

- 하나의 유효한 로케일 코드가 결정되어 번역 시스템에 전달된다

#### 4.3.4 기본 흐름 (로그인 사용자)

1. 시스템이 현재 세션에서 인증된 사용자 정보를 조회한다
2. 사용자가 로그인 상태인 경우, User 모델의 `locale` 필드 값을 가져온다
3. 해당 로케일 코드를 번역 시스템에 적용한다

#### 4.3.5 대안 흐름

**AF-03-01: 비로그인 사용자 (Accept-Language 헤더 기반 감지)**

1. 시스템이 현재 세션에 인증된 사용자가 없음을 확인한다
2. HTTP 요청의 `Accept-Language` 헤더 값을 파싱한다
3. 파싱된 언어 목록에서 다음 우선순위로 매칭을 시도한다:
   - **1단계 - 정확한 로케일 코드 매칭**: Accept-Language의 값이 지원 로케일 코드와 정확히 일치하는지 확인한다 (예: `de-DE` -> `de-DE`)
   - **2단계 - 언어 코드 접두사 매칭**: Accept-Language의 언어 코드 부분이 지원 로케일의 언어 코드 부분과 일치하는지 확인한다 (예: `de` -> `de-DE`, `pt` -> `pt-BR`)
   - **3단계 - 폴백**: 매칭되는 로케일이 없으면 `en-US`를 사용한다
4. 결정된 로케일 코드를 번역 시스템에 적용한다

#### 4.3.6 예외 흐름

| 예외 | 조건 | 처리 |
|------|------|------|
| EX-03-01 | Accept-Language 헤더가 없는 경우 | `en-US`를 적용한다 |
| EX-03-02 | Accept-Language에 지원하지 않는 언어만 포함된 경우 | `en-US`를 적용한다 |
| EX-03-03 | 로그인 사용자의 DB locale 값이 현재 지원 로케일에 포함되지 않는 경우 | `en-US`를 적용한다 |

#### 4.3.7 비즈니스 규칙

| 규칙 ID | 규칙 |
|---------|------|
| BR-03-01 | 로케일 결정 우선순위: DB 저장 로케일 > Accept-Language 헤더 > 폴백(`en-US`) |
| BR-03-02 | 접두사 매칭에서 `pt`는 `pt-BR`에 매칭된다 (pt-PT보다 pt-BR 우선) |
| BR-03-03 | 최종 폴백 로케일은 항상 `en-US`이다 |

### 4.9 다국어 설문 콘텐츠 지원 (i18nString)

#### 4.9.1 기능 개요

| 항목 | 내용 |
|------|------|

| 우선순위 | 필수 |
| 기능 설명 | 설문 콘텐츠의 다국어 문자열을 표현하는 i18nString 타입과 관련 유틸리티 함수를 제공한다 |

#### 4.9.2 선행 조건

- 설문이 다국어 지원이 필요한 상태이다

#### 4.9.3 후행 조건

- 설문 콘텐츠가 여러 언어로 저장 및 조회 가능하다

#### 4.9.4 기본 흐름

**i18nString 생성**
1. 시스템이 텍스트와 대상 언어 목록을 입력받는다
2. `default` 키에 기본 언어 텍스트를 저장한다
3. 각 대상 언어 코드를 키로 하여 빈 문자열 또는 번역 텍스트를 저장한다
4. i18nString 객체를 반환한다

**특정 언어 값 추출**
1. i18nString 객체와 언어 코드를 입력받는다
2. 해당 언어 코드의 값이 존재하면 반환한다
3. 존재하지 않으면 `default` 키의 값을 반환한다

## 5. 데이터 모델

### 5.1 주요 엔티티 정의

**User (로케일 관련 필드)**

| 필드명 | 타입 | 필수 | 기본값 | 설명 |
|--------|------|------|-------|------|
| locale | String | Y | `en-US` | 사용자의 선호 로케일 코드 |

**번역 파일 (Translation File)**

| 항목 | 값 |
|------|-----|
| 형식 | JSON |
| 위치 | 프로젝트 내 번역 디렉토리 |
| 파일 수 | 14개 (로케일당 1개) |
| 키 형식 | 소문자 + 점 구분자 (FN-030-06 참조) |
| 값 형식 | 문자열 (ICU Message Format 포함 가능) |

**i18nString (설문 콘텐츠 다국어 문자열)**

| 필드명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| default | String | Y | 기본 언어 텍스트 |
| [languageCode] | String | N | 각 언어별 번역 텍스트 |

**앱 언어 항목 (App Language)**

| 필드명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| code | String | Y | 로케일 코드 (예: `de-DE`) |
| label | String | Y | 영어 표시 이름 (예: "German") |

### 5.2 엔티티 간 관계

```
User.locale ──── 참조 ────► 지원 로케일 목록 (14개 코드 중 하나)
                                │
                                ▼
                         번역 파일 (JSON)
                                │
                                ▼
                         번역 텍스트 출력

i18nString.default ──── 기본 ────► 설문 콘텐츠 표시
i18nString[lang]  ──── 번역 ────► 해당 언어 설문 콘텐츠 표시
```

### 5.3 데이터 흐름

```
[사용자 요청]
     │
     ▼
[로케일 결정] ◄── User.locale (로그인) / Accept-Language (비로그인)
     │
     ▼
[번역 파일 로드] ── 동적 import로 해당 로케일 JSON 파일만 로드
     │
     ▼
[번역 처리] ── i18next + ICU 플러그인
     │
     ├──► [서버 컴포넌트] ── 서버 번역 함수 ── 렌더링
     │
     └──► [클라이언트 컴포넌트] ── useTranslation hook ── 렌더링
```

## 8. 제약사항 및 가정

### 8.1 기술적 제약사항

| 항목 | 내용 |
|------|------|
| 번역 라이브러리 | react-i18next + i18next |
| 메시지 포맷 | ICU Message Format (i18next-icu 플러그인) |
| 번역 파일 형식 | JSON |
| XSS 방지 | React 내장 메커니즘에 의존 (i18next escape 비활성화) |
| 번역 키 형식 | 소문자 + 점 구분자 |

### 8.2 비즈니스 제약사항

| 항목 | 내용 |
|------|------|
| 지원 언어 수 | 14개 (고정) |
| 기본/폴백 로케일 | en-US |
| 번역 관리 방식 | 수동 (자동 번역 미지원) |
| 설문 콘텐츠 다국어 | 별도 Enterprise 기능 (이 문서 범위 외) |
| RTL 레이아웃 | 미지원 |

### 8.3 가정사항

| 항목 | 내용 |
|------|------|
| A-01 | `en-US` 번역 파일은 항상 100% 완전한 상태로 유지된다 |
| A-02 | 사용자 브라우저가 표준 Accept-Language 헤더를 전송한다 |
| A-03 | 모든 React 컴포넌트에서 `dangerouslySetInnerHTML`을 사용하지 않는다 (XSS 방지 전제) |
| A-04 | 번역 기여는 오픈소스 방식으로 관리되며, 기여된 번역의 품질은 리뷰를 통해 검증한다 |
| A-05 | 로케일 저장 위치인 User.locale 필드는 DB 스키마에 이미 존재하거나 마이그레이션으로 추가된다 |
