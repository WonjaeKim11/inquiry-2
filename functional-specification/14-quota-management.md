## 2. 개요

### 2.1 목적

쿼터(Quota) 관리 시스템의 상세 기능 명세를 정의한다. 본 문서는 FSD-014 요구사항 명세서를 기반으로, 설문 응답 수를 조건별로 제한하는 Enterprise 기능의 데이터 모델, 비즈니스 로직, 평가 프로세스, 인터페이스를 구체적으로 기술한다.

### 2.2 범위

**포함 범위:**
- 쿼터 CRUD (생성, 조회, 수정, 삭제)
- 조건 기반 쿼터 평가 (조건부 로직 엔진 재사용)
- 쿼터 초과 시 액션 수행 (endSurvey / continueSurvey)
- 부분 제출(Partial Submission) 카운트 옵션
- 응답-쿼터 연결 테이블을 통한 N:M 관계 추적
- screenedIn / screenedOut 상태 관리
- Ending Card 연계 (쿼터 초과 시 특정 종료 카드 표시)
- Enterprise 라이선스 검증

**제외 범위:**
- 실시간 대시보드 (별도 Summary 컴포넌트로 처리)
- 쿼터 알림/노티피케이션
- 쿼터 일시 정지/재개

### 2.3 대상 사용자

| 역할 | 관심 사항 |
|------|----------|
| 시장 조사자 | 표본 균형 맞추기 (성별, 연령대별 할당) |
| 설문 관리자 | 응답 수 한도 관리, 스크리닝 |
| 데이터 분석가 | 쿼터별 응답 분포 확인 |
| 시스템 관리자 | Enterprise 라이선스 관리 |

### 2.4 용어 정의

| 용어 | 정의 |
|------|------|
| 쿼터 (Quota) | 설문 응답 수를 조건별로 제한하는 규칙 단위. 조건(logic), 한도(limit), 액션(action)으로 구성된다. |
| screenedIn | 쿼터 조건을 충족하며 한도를 초과하지 않은 상태. 정상적으로 카운트되는 응답이다. |
| screenedOut | 쿼터 조건을 충족하였으나 한도를 초과한 상태. 스크리닝 아웃된 응답이다. |
| Ending Card | 설문 종료 시 응답자에게 표시되는 카드. 쿼터 초과로 설문이 종료될 때 지정된 Ending Card가 표시된다. |
| 부분 제출 (Partial Submission) | 설문이 완료(finished=true)되지 않은 상태의 응답. countPartialSubmissions 옵션에 따라 쿼터 카운트에 포함 여부가 결정된다. |
| 조건부 로직 엔진 | FSD-012에서 정의된 로직 평가 시스템. 쿼터의 조건 평가에 재사용된다. |
| Enterprise 라이선스 | 쿼터 기능을 사용하기 위해 필요한 상용 라이선스. quotas 기능 플래그가 활성화되어야 한다. |
| 기능 플래그 (Feature Flag) | Enterprise 라이선스에서 개별 기능의 활성화 여부를 제어하는 설정값. |
| Cascade 삭제 | 상위 엔티티(Survey, Response) 삭제 시 하위 연결 데이터가 자동으로 삭제되는 방식. |
| 복합 기본키 (Composite Primary Key) | 두 개 이상의 컬럼을 조합하여 레코드를 고유하게 식별하는 키. 응답-쿼터 연결 테이블에서 [responseId, quotaId]로 구성된다. |

## 4. 상세 기능 명세

### 4.1 Enterprise 라이선스 검증

#### 4.1.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-014-01 |
| 기능명 | Enterprise 라이선스 검증 |
| 관련 요구사항 ID | FR-022 (4.13), AC-022-01 |
| 우선순위 | 필수 |
| 기능 설명 | 쿼터 기능에 접근하기 전에 Enterprise 라이선스의 quotas 기능 플래그가 활성화되어 있는지 검증한다. |

#### 4.1.2 선행 조건 (Preconditions)

- 사용자가 인증된 상태여야 한다.
- 사용자가 해당 설문이 속한 Organization의 멤버여야 한다.

#### 4.1.3 후행 조건 (Postconditions)

- 라이선스가 유효하면 쿼터 CRUD 및 평가 기능에 접근할 수 있다.
- 라이선스가 유효하지 않으면 쿼터 기능이 비활성화된다.

#### 4.1.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 사용자 | 쿼터 관련 기능(생성, 조회, 수정, 삭제, 평가)에 접근을 시도한다. |
| 2 | 시스템 | Enterprise 라이선스 검증 서비스를 호출한다. |
| 3 | 시스템 | 캐시된 라이선스 정보를 확인한다 (메모리 캐시 TTL: 1분, Redis 캐시 TTL: 24시간). |
| 4 | 시스템 | 라이선스 정보에서 quotas 기능 플래그의 활성화 여부를 확인한다. |
| 5 | 시스템 | quotas 플래그가 활성화되어 있으면 요청을 허용한다. |

#### 4.1.5 대안 흐름 (Alternative Flow)

**AF-01: 캐시 미스 시 라이선스 서버 조회**

| 단계 | 주체 | 동작 |
|------|------|------|
| 3a | 시스템 | 캐시에 라이선스 정보가 없으면 라이선스 서버에 조회 요청을 보낸다. |
| 3b | 시스템 | 라이선스 서버 응답을 캐시에 저장한다 (성공: 24시간 TTL, 실패: 10분 TTL). |
| 3c | 시스템 | 기본 흐름 단계 4로 진행한다. |

#### 4.1.6 예외 흐름 (Exception Flow)

**EF-01: 라이선스 미보유 또는 quotas 플래그 비활성화**

| 단계 | 주체 | 동작 |
|------|------|------|
| 4a | 시스템 | quotas 플래그가 비활성화 상태이면 기능 접근을 차단한다. |
| 4b | 시스템 | 403 Forbidden 또는 기능 비활성화 응답을 반환한다. |

**EF-02: 라이선스 서버 접근 불가**

| 단계 | 주체 | 동작 |
|------|------|------|
| 3a-1 | 시스템 | 라이선스 서버에 접근할 수 없으면 최대 3회 재시도한다 (지수 백오프, 초기 지연 1초). |
| 3a-2 | 시스템 | 재시도 실패 시 Grace Period(3일) 내이면 이전 라이선스 결과를 사용한다 (이전 결과 TTL: 4일). |
| 3a-3 | 시스템 | Grace Period 초과 시 기능을 비활성화한다. |

#### 4.1.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 규칙 | 조건 | 액션 |
|---------|------|------|------|
| BR-01-01 | 쿼터 기능은 Enterprise 전용이다 | quotas 기능 플래그가 false이거나 라이선스가 없는 경우 | 모든 쿼터 관련 API 호출을 차단한다 |
| BR-01-02 | 라이선스 캐시 정책을 적용한다 | 라이선스 조회 성공 시 | 24시간 캐시한다 |
| BR-01-03 | 라이선스 실패 캐시 정책을 적용한다 | 라이선스 조회 실패 시 | 10분 캐시한다 |
| BR-01-04 | Grace Period를 적용한다 | 라이선스 서버 접근 불가 시 | 3일간 이전 결과를 사용한다 |

#### 4.1.8 데이터 요구사항

**입력 데이터:** 없음 (시스템 내부에서 자동으로 수행)

**출력 데이터:**

| 필드명 | 타입 | 설명 |
|--------|------|------|
| isQuotaEnabled | boolean | 쿼터 기능 사용 가능 여부 |

#### 4.1.9 비기능 요구사항 (해당 기능 관련)

- 라이선스 캐시 TTL: 성공 24시간, 실패 10분, 메모리 1분
- 라이선스 서버 재시도: 최대 3회, 지수 백오프
- Grace Period: 3일, 이전 결과 보존 4일

### 4.2 쿼터 생성

#### 4.2.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-014-02 |
| 기능명 | 쿼터 생성 |
| 관련 요구사항 ID | FR-022 (4.1, 4.2, 4.3, 4.4), AC-022-02, AC-022-03, AC-022-04, AC-022-12 |
| 우선순위 | 필수 |
| 기능 설명 | 특정 설문에 새 쿼터를 생성한다. 쿼터 이름, 한도, 조건, 액션, Ending Card 연결 등을 설정한다. |

#### 4.2.2 선행 조건 (Preconditions)

- Enterprise 라이선스의 quotas 기능 플래그가 활성화되어 있어야 한다 (FN-014-01 통과).
- 대상 설문(surveyId)이 존재해야 한다.
- 사용자가 해당 설문에 대한 수정 권한을 보유해야 한다.

#### 4.2.3 후행 조건 (Postconditions)

- 새 쿼터 레코드가 데이터베이스에 생성된다.
- 쿼터 ID(CUID)가 자동 생성된다.
- createdAt, updatedAt이 현재 시간으로 설정된다.

#### 4.2.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 사용자 | 쿼터 생성 요청을 보낸다 (name, limit, logic, action, endingCardId, countPartialSubmissions 포함). |
| 2 | 시스템 | Enterprise 라이선스 검증을 수행한다 (FN-014-01). |
| 3 | 시스템 | 입력 데이터의 유효성을 검증한다 (4.2.8 참조). |
| 4 | 시스템 | 같은 설문 내에서 동일한 이름의 쿼터가 존재하지 않는지 확인한다. |
| 5 | 시스템 | 쿼터 레코드를 데이터베이스에 저장한다. |
| 6 | 시스템 | 생성된 쿼터 정보를 반환한다. |

#### 4.2.5 대안 흐름 (Alternative Flow)

**AF-01: 조건 미지정 시 기본값 적용**

| 단계 | 주체 | 동작 |
|------|------|------|
| 3a | 시스템 | logic 필드가 제공되지 않으면 빈 객체(`{}`)를 기본값으로 설정한다. |

**AF-02: countPartialSubmissions 미지정 시 기본값 적용**

| 단계 | 주체 | 동작 |
|------|------|------|
| 3b | 시스템 | countPartialSubmissions 필드가 제공되지 않으면 false를 기본값으로 설정한다. |

#### 4.2.6 예외 흐름 (Exception Flow)

**EF-01: 라이선스 검증 실패**

| 단계 | 주체 | 동작 |
|------|------|------|
| 2a | 시스템 | Enterprise 라이선스 검증 실패 시 403 응답을 반환한다. |

**EF-02: 필수 필드 누락**

| 단계 | 주체 | 동작 |
|------|------|------|
| 3a | 시스템 | name, limit, action 중 하나라도 누락되면 400 응답과 함께 누락된 필드 정보를 반환한다. |

**EF-03: limit 값이 1 미만**

| 단계 | 주체 | 동작 |
|------|------|------|
| 3b | 시스템 | limit가 0 이하이면 400 응답과 "Limit must be greater than 0" 메시지를 반환한다. |

**EF-04: endSurvey 액션에 endingCardId 미지정**

| 단계 | 주체 | 동작 |
|------|------|------|
| 3c | 시스템 | action이 endSurvey이고 endingCardId가 null이거나 빈 문자열이면 400 응답과 "endingCardId is required when action is endSurvey" 메시지를 반환한다. |

**EF-05: 쿼터 이름 중복**

| 단계 | 주체 | 동작 |
|------|------|------|
| 4a | 시스템 | 같은 설문 내에 동일한 이름의 쿼터가 이미 존재하면 409 응답과 중복 에러 메시지를 반환한다. |

**EF-06: 설문이 존재하지 않음**

| 단계 | 주체 | 동작 |
|------|------|------|
| 4b | 시스템 | surveyId에 해당하는 설문이 존재하지 않으면 404 응답을 반환한다. |

#### 4.2.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 규칙 | 조건 | 액션 |
|---------|------|------|------|
| BR-02-01 | name은 필수이다 | name이 null이거나 빈 문자열인 경우 | 유효성 검증 실패 |
| BR-02-02 | name은 설문 내에서 유일해야 한다 | 동일 surveyId 내 동일 name 존재 시 | 유니크 제약 위반 에러 |
| BR-02-03 | limit는 1 이상이어야 한다 | limit < 1인 경우 | 유효성 검증 실패 |
| BR-02-04 | endSurvey 시 endingCardId는 필수이다 | action=endSurvey이고 endingCardId가 null 또는 빈 문자열인 경우 | 유효성 검증 실패 |
| BR-02-05 | action은 endSurvey 또는 continueSurvey만 허용한다 | action이 두 값 중 하나가 아닌 경우 | 유효성 검증 실패 |
| BR-02-06 | logic 기본값은 빈 객체이다 | logic 미지정 시 | `{}`를 기본값으로 설정 |
| BR-02-07 | countPartialSubmissions 기본값은 false이다 | countPartialSubmissions 미지정 시 | false를 기본값으로 설정 |

#### 4.2.8 데이터 요구사항

**입력 데이터:**

### 4.6 쿼터 조건 평가

#### 4.6.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-014-06 |
| 기능명 | 쿼터 조건 평가 |
| 관련 요구사항 ID | FR-022 (4.7, 4.8), AC-022-05, AC-022-06, AC-022-13, AC-022-14 |
| 우선순위 | 필수 |
| 기능 설명 | 응답 제출/업데이트 시 해당 설문의 모든 쿼터에 대해 조건을 평가하여 조건 충족 쿼터(passedQuotas)와 미충족 쿼터(failedQuotas)로 분류한다. 조건부 로직 엔진(FSD-012)의 로직 평가 처리를 재사용한다. |

#### 4.6.2 선행 조건 (Preconditions)

- 응답 제출 또는 업데이트 이벤트가 발생해야 한다.
- 해당 설문에 1개 이상의 쿼터가 정의되어 있어야 한다.
- 조건부 로직 엔진(FSD-012)이 정상 동작해야 한다.

#### 4.6.3 후행 조건 (Postconditions)

- 각 쿼터가 passedQuotas 또는 failedQuotas로 분류된다.
- 분류 결과가 FN-014-07(한도 확인 및 액션 수행)로 전달된다.

#### 4.6.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | 응답 제출/업데이트 이벤트를 수신한다. |
| 2 | 시스템 | 해당 설문의 모든 쿼터를 조회한다. |
| 3 | 시스템 | 해당 설문의 설문 정보(질문 구조, Ending Card 등)를 조회한다. |
| 4 | 시스템 | 각 쿼터의 조건(logic) 객체에 id를 추가하여 조건 그룹 형식으로 변환한다. |
| 5 | 시스템 | 조건부 로직 엔진의 로직 평가 처리 함수를 호출하여 각 쿼터의 조건을 평가한다. 입력: 설문 데이터, 응답 데이터, 변수 데이터, 조건 그룹. |
| 6 | 시스템 | 평가 결과가 true인 쿼터를 passedQuotas로, false인 쿼터를 failedQuotas로 분류한다. |
| 7 | 시스템 | 분류 결과를 FN-014-07로 전달한다. |

#### 4.6.5 대안 흐름 (Alternative Flow)

**AF-01: 설문에 쿼터가 없는 경우**

| 단계 | 주체 | 동작 |

**AF-02: 부분 제출 필터링**

| 단계 | 주체 | 동작 |
|------|------|------|
| 4a | 시스템 | 응답이 완료되지 않았고(finished=false) endSurvey 액션이 아닌 경우, countPartialSubmissions=true인 쿼터만 평가 대상에 포함한다. |

#### 4.6.6 예외 흐름 (Exception Flow)

**EF-01: 조건 평가 중 에러 발생**

| 단계 | 주체 | 동작 |
|------|------|------|
| 5a | 시스템 | 조건 평가 중 예외가 발생하면 해당 쿼터의 조건을 미충족(false)으로 처리한다. |
| 5b | 시스템 | 에러를 로깅하고 설문 종료 없이 계속 진행한다 (shouldEndSurvey: false). |

### 4.7 한도 확인 및 액션 수행

#### 4.7.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-014-07 |
| 기능명 | 한도 확인 및 액션 수행 |
| 관련 요구사항 ID | FR-022 (4.9, 4.11, 4.12), AC-022-05 ~ AC-022-11, AC-022-13 |
| 우선순위 | 필수 |
| 기능 설명 | 조건을 충족한 쿼터(passedQuotas)에 대해 현재 screenedIn 카운트를 조회하고, 한도 초과 여부를 판단하여 적절한 액션(endSurvey/continueSurvey)을 수행한다. |

#### 4.7.2 선행 조건 (Preconditions)

- FN-014-06(쿼터 조건 평가)의 결과가 전달되어야 한다.
- passedQuotas에 1개 이상의 쿼터가 포함되어 있어야 한다.

#### 4.7.3 후행 조건 (Postconditions)

- 각 passedQuotas에 대해 한도 초과 여부가 판정된다.
- 한도 초과 시 해당 쿼터의 액션이 수행된다.
- 응답-쿼터 연결 레코드가 생성/수정된다 (FN-014-08).

#### 4.7.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | 데이터베이스 트랜잭션을 시작한다. |
| 2 | 시스템 | 조건 미충족 쿼터(failedQuotas)에 대해 기존 응답-쿼터 연결 레코드를 삭제한다. |
| 3 | 시스템 | 각 passedQuota에 대해 현재 screenedIn 상태의 응답 수를 카운트한다 (현재 응답 제외). |
| 4 | 시스템 | countPartialSubmissions=false인 쿼터는 finished=true인 응답만 카운트한다. |
| 5 | 시스템 | countPartialSubmissions=true인 쿼터는 모든 응답(finished 여부 무관)을 카운트한다. |
| 6 | 시스템 | screenedIn 카운트 >= limit이면 해당 쿼터를 한도 초과(full)로 판정한다. |
| 7 | 시스템 | 한도 초과 쿼터에 대해 응답-쿼터 연결을 screenedOut 상태로 생성/업데이트한다. |
| 8 | 시스템 | 한도 미초과 쿼터에 대해 응답-쿼터 연결을 screenedIn 상태로 생성한다 (중복 건너뜀). |
| 9 | 시스템 | 한도 초과된 첫 번째 쿼터의 액션을 적용한다. |
| 10 | 시스템 | 트랜잭션을 커밋한다. |
| 11 | 시스템 | 쿼터 평가 결과를 반환한다. |

#### 4.7.5 대안 흐름 (Alternative Flow)

**AF-01: endSurvey 액션 수행**

| 단계 | 주체 | 동작 |
|------|------|------|
| 9a | 시스템 | 한도 초과된 첫 번째 쿼터의 action이 endSurvey인 경우: |
| 9a-1 | 시스템 | 응답의 finished 상태를 true로 설정한다. |
| 9a-2 | 시스템 | 응답의 endingId를 해당 쿼터의 endingCardId로 설정한다. |
| 9a-3 | 시스템 | 쿼터 초과 응답 구조를 생성한다 (quotaFull: true, quotaId, action: endSurvey, endingCardId). |

**AF-02: continueSurvey 액션 수행**

| 단계 | 주체 | 동작 |
|------|------|------|
| 9b | 시스템 | 한도 초과된 첫 번째 쿼터의 action이 continueSurvey인 경우: |
| 9b-1 | 시스템 | 응답을 screenedOut으로 표시하되 설문은 계속 진행한다. |
| 9b-2 | 시스템 | shouldEndSurvey: false를 반환한다. |

**AF-03: 모든 passedQuotas가 한도 미초과**

| 단계 | 주체 | 동작 |
|------|------|------|
| 9c | 시스템 | 한도를 초과한 쿼터가 없으면 shouldEndSurvey: false를 반환한다. |

### 4.8 응답-쿼터 연결 관리

#### 4.8.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-014-08 |
| 기능명 | 응답-쿼터 연결 관리 |
| 관련 요구사항 ID | FR-022 (4.5, 4.6, 4.10), AC-022-05, AC-022-06, AC-022-14 |
| 우선순위 | 필수 |
| 기능 설명 | 응답과 쿼터 간의 N:M 관계를 관리한다. 쿼터 평가 결과에 따라 연결 레코드를 생성, 수정, 삭제한다. |

#### 4.8.2 선행 조건 (Preconditions)

- FN-014-07(한도 확인 및 액션 수행) 프로세스 내에서 호출되어야 한다.
- 데이터베이스 트랜잭션 컨텍스트 내에서 실행되어야 한다.

#### 4.8.3 후행 조건 (Postconditions)

- 조건 미충족 쿼터의 기존 연결 레코드가 삭제된다.
- 한도 초과 쿼터에 대해 screenedOut 연결 레코드가 생성/업데이트된다.
- 한도 미초과 쿼터에 대해 screenedIn 연결 레코드가 생성된다 (기존 레코드가 있으면 건너뜀).

#### 4.8.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | 조건 미충족 쿼터(failedQuotas)에 대해: 해당 응답과 연결된 기존 연결 레코드를 삭제한다. |
| 2 | 시스템 | 한도 초과 쿼터에 대해: 응답-쿼터 연결 레코드를 screenedOut 상태로 생성한다. 기존에 screenedIn이었다면 screenedOut으로 업데이트한다 (upsert). |
| 3 | 시스템 | 한도 미초과 쿼터에 대해: 응답-쿼터 연결 레코드를 screenedIn 상태로 생성한다. 이미 레코드가 존재하면 건너뛴다 (skipDuplicates). |

#### 4.8.5 대안 흐름 (Alternative Flow)

없음.

#### 4.8.6 예외 흐름 (Exception Flow)

**EF-01: 동시성 충돌 (중복 레코드)**

| 단계 | 주체 | 동작 |
|------|------|------|
| 3a | 시스템 | 동시 요청으로 인해 중복 레코드가 발생하면 skipDuplicates 옵션으로 에러 없이 건너뛴다. |

#### 4.8.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 규칙 | 조건 | 액션 |
|---------|------|------|------|
| BR-08-01 | 연결 레코드의 기본키는 [responseId, quotaId] 복합키이다 | 항상 | 하나의 응답은 하나의 쿼터에 대해 최대 1개의 연결 레코드를 가진다 |
| BR-08-02 | 조건 미충족 시 기존 연결을 삭제한다 | 쿼터 조건이 더 이상 충족되지 않을 때 | 해당 연결 레코드를 삭제한다 |

| BR-08-04 | 한도 미초과 시 screenedIn으로 설정한다 | screenedIn 카운트 < limit | 연결 레코드를 screenedIn으로 생성한다 (중복 건너뜀) |
| BR-08-05 | Cascade 삭제를 적용한다 | 응답 또는 쿼터가 삭제될 때 | 관련 연결 레코드를 자동 삭제한다 |

### 4.9 쿼터 초과 응답 처리

## 5. 데이터 모델

### 5.1 주요 엔티티 정의

**Quota (쿼터)**

| 필드명 | 타입 | 필수 | 기본값 | 제약 | 설명 |
|--------|------|------|--------|------|------|
| id | string (CUID) | Y | 자동생성 | PK | 쿼터 고유 식별자 |

| limit | integer | Y | - | >= 1 | 응답 한도 |
| logic | JSON | Y | `{}` | AND/OR 조건 구조 | 조건 정의 |
| action | enum | Y | - | "endSurvey" 또는 "continueSurvey" | 한도 초과 시 액션 |
| endingCardId | string (CUID) 또는 null | 조건부 | null | action=endSurvey 시 필수 | 종료 시 표시할 Ending Card ID |

**ResponseQuota (응답-쿼터 연결)**

| 필드명 | 타입 | 필수 | 기본값 | 제약 | 설명 |
|--------|------|------|--------|------|------|
| responseId | string (CUID) | Y | - | PK (복합), FK(Response), Cascade 삭제 | 응답 ID |
| quotaId | string (CUID) | Y | - | PK (복합), FK(Quota), Cascade 삭제 | 쿼터 ID |
| status | enum | Y | - | "screenedIn" 또는 "screenedOut" | 연결 상태 |

### 5.2 엔티티 간 관계

```
Survey (1) -----> (N) Quota
  |                      |
  |                      | (N:M via ResponseQuota)
  v                      v
Response (1) ---> (N) ResponseQuota (N) <--- (1) Quota

EndingCard (1) <---- (N) Quota (action=endSurvey)
```

| 관계 | 카디널리티 | 삭제 정책 | 설명 |
|------|-----------|-----------|------|
| Survey -> Quota | 1:N | Cascade | 설문 삭제 시 모든 쿼터 자동 삭제 |
| Response -> ResponseQuota | 1:N | Cascade | 응답 삭제 시 모든 연결 자동 삭제 |
| Quota -> ResponseQuota | 1:N | Cascade | 쿼터 삭제 시 모든 연결 자동 삭제 |
| Quota -> EndingCard | N:1 | - | endSurvey 액션 시 Ending Card 참조 |

### 5.3 데이터 흐름

```
응답 제출/업데이트
    |
    v
[1] 설문의 모든 쿼터 조회 (Quota 테이블)
    |
    v
[2] 설문 정보 조회 (Survey 테이블)
    |
    v
[3] 조건 평가 (조건부 로직 엔진)
    |
    +---> passedQuotas (조건 충족)
    |           |
    |           v
    |     [4] screenedIn 카운트 조회 (ResponseQuota 테이블, [quotaId, status] 인덱스)
    |           |
    |           +---> 한도 미초과: screenedIn 레코드 생성 (ResponseQuota)
    |           |
    |           +---> 한도 초과: screenedOut 레코드 upsert (ResponseQuota)
    |                     |
    |                     +---> endSurvey: Response.finished=true, Response.endingId 설정
    |                     |
    |                     +---> continueSurvey: shouldEndSurvey=false
    |
    +---> failedQuotas (조건 미충족)
                |
                v
          [5] 기존 연결 레코드 삭제 (ResponseQuota)
```

## 6. 인터페이스 명세

### 6.1 외부 시스템 연동

| 연동 대상 | 연동 방식 | 설명 |
|-----------|----------|------|
| 조건부 로직 엔진 (FSD-012) | 내부 함수 호출 | 쿼터 조건 평가 시 로직 평가 처리 함수를 재사용한다 |
| Enterprise 라이선스 서비스 | 내부 API 호출 + 캐시 | quotas 기능 플래그 확인 시 사용한다 |
| 응답 파이프라인 (Response Pipeline) | 이벤트 기반 | 응답 제출/업데이트 이벤트 수신 시 쿼터 평가를 트리거한다 |
| Ending Card 시스템 | 참조 연결 | endSurvey 시 endingCardId를 통해 특정 종료 카드를 표시한다 |

### 6.2 API 명세 (해당 시)

**쿼터 CRUD API:**

| 메서드 | 경로 | 설명 | 인증 |
|--------|------|------|------|
| POST | /api/v2/management/surveys/{surveyId}/quotas | 쿼터 생성 | API Key |
| GET | /api/v2/management/surveys/{surveyId}/quotas | 쿼터 목록 조회 | API Key |
| GET | /api/v2/management/quotas/{quotaId} | 쿼터 단건 조회 | API Key |

**쿼터 평가 (내부 프로세스):**

쿼터 평가는 외부 API로 노출되지 않으며, 응답 제출/업데이트 파이프라인 내부에서 자동으로 실행된다.

## 8. 제약사항 및 가정

### 8.1 기술적 제약사항

| 항목 | 제약 값 | 근거 |
|------|--------|------|
| 쿼터 조건 구조 | 단일 레벨 (중첩 그룹 없음) | 복잡도 제어를 위해 조건부 로직 엔진의 중첩 그룹 기능을 사용하지 않음 |
| 응답-쿼터 기본키 | [responseId, quotaId] 복합키 | 동일 응답이 동일 쿼터에 대해 중복 레코드를 가질 수 없음 |
| 카운트 인덱스 | [quotaId, status] | 카운트 쿼리 성능 최적화를 위한 필수 인덱스 |
| Cascade 삭제 | Survey -> Quota, Response -> ResponseQuota, Quota -> ResponseQuota | 데이터 정합성 유지를 위한 자동 삭제 |

### 8.2 비즈니스 제약사항

| 항목 | 제약 값 | 근거 |
|------|--------|------|
| 최소 limit | 1 이상 | 0 이하의 limit는 의미가 없음 |
| 쿼터 이름 최소 길이 | 1자 | 빈 이름은 식별에 부적합 |
| 쿼터 이름 유일성 범위 | 동일 설문 내 | 서로 다른 설문의 쿼터 이름은 중복 가능 |
| endSurvey 시 endingCardId | 필수 (not null, not empty) | 설문 종료 시 반드시 종료 카드를 표시해야 함 |
| 부분 제출 카운트 기본값 | false | 기본적으로 완료된 응답만 카운트 |
| 조건 기본값 | 빈 객체 `{}` | 조건 미설정 시 기본값 |
| 라이선스 기능 플래그 | quotas | Enterprise 라이선스에서 해당 플래그가 활성화되어야 함 |
| 라이선스 캐시 TTL | 성공: 24시간, 실패: 10분 | 라이선스 서버 부하 최소화 |

### 8.3 가정사항

| 항목 | 가정 | 영향 |
|------|------|------|
| 조건부 로직 엔진 가용성 | FSD-012의 로직 평가 처리 함수가 안정적으로 동작한다고 가정 | 해당 엔진에 장애 발생 시 쿼터 평가가 실패하며, 안전 처리(shouldEndSurvey: false)로 대응 |
| Ending Card 존재 | endSurvey 설정 시 유효한 Ending Card가 사전에 생성되어 있다고 가정 | Ending Card가 삭제된 경우에 대한 무결성 검증은 별도 고려 필요 |
| 응답 파이프라인 연동 | 응답 제출/업데이트 파이프라인에서 쿼터 평가 함수를 호출한다고 가정 | 파이프라인 연동이 누락되면 쿼터가 평가되지 않음 |
| 데이터베이스 트랜잭션 지원 | PostgreSQL의 트랜잭션 기능이 정상 동작한다고 가정 | 트랜잭션 실패 시 데이터 불일치 가능 |
