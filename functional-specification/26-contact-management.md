## 2. 개요

### 2.1 목적

Inquiry 플랫폼에서 설문 대상이 되는 사용자/고객(연락처, Contact)을 체계적으로 관리하기 위한 기능을 정의한다. CSV 대량 업로드, SDK를 통한 자동 생성, API v2를 통한 CRUD, 속성(Attribute) 기반의 타이핑 시스템을 제공하여 설문 타겟팅과 개인화된 설문 링크 생성을 가능하게 한다.

### 2.2 범위

**포함 범위 (In-scope)**
- CSV 파일을 통한 연락처 대량 가져오기 (최대 10,000건)
- API v2를 통한 단건/대량(250건) 연락처 생성
- SDK 사용자 식별 호출을 통한 자동 연락처 생성
- 연락처 속성(Attribute) CRUD 및 typed storage (string, number, date)
- 중복 처리 전략 3종 (skip, update, overwrite)
- 연락처 삭제 (hard delete)
- 속성 키(Attribute Key) 관리 (default vs custom)
- 개인화된 설문 링크 생성

**제외 범위 (Out-of-scope)**
- 관리자 UI에서의 개별 연락처 수동 생성 (CSV 또는 API만 지원)
- 연락처 병합(merge)
- 연락처 소프트 삭제 (hard delete만 지원)

### 2.3 대상 사용자

| 역할 | 설명 | 주요 기능 |
|------|------|----------|
| Organization Owner/Manager | 연락처를 관리하고 설문 타겟팅에 활용 | CSV 가져오기, 연락처 조회/삭제, 속성 키 관리 |
| Team Member | 할당된 프로젝트의 연락처 조회 | 연락처 조회/검색 |
| SDK/API Consumer | 프로그래밍 방식으로 연락처 생성/업데이트 | API v2 CRUD, SDK identify |
| System | SDK identify 시 자동 연락처 생성 | 자동 연락처/속성 생성 |

### 2.4 용어 정의

| 용어 | 정의 |
|------|------|
| Contact (연락처) | 설문 대상이 되는 사용자/고객. Environment 단위로 격리되며 속성(Attribute)을 가진다. Enterprise 라이선스 필요. |
| Contact Attribute (연락처 속성) | 연락처에 연결된 속성 값. string, number, date 데이터 타입 지원. |
| Contact Attribute Key (속성 키) | 속성의 정의(스키마). default(시스템 정의, 수정/삭제 불가)와 custom(사용자 정의, 수정/삭제 가능) 두 유형. |
| Typed Storage | 속성 값을 데이터 타입(string, number, date)에 따라 네이티브 컬럼에 저장하는 방식. |
| Safe Identifier | 속성 키 이름 규칙. 소문자, 숫자, 언더스코어만 허용하며 문자로 시작해야 한다. 정규식: `^[a-z][a-z0-9_]*$` |
| Duplicate Strategy (중복 처리 전략) | CSV Import 시 기존 연락처와 email이 중복될 때의 처리 방법. skip, update, overwrite 3종. |
| Environment | 프로젝트 내 production과 development 두 환경을 제공. 데이터가 완전히 격리된다. |
| Hard Delete | 데이터베이스에서 레코드를 완전히 삭제하는 방식. 소프트 삭제(논리 삭제)와 대비된다. |
| Cascade Delete | 부모 레코드 삭제 시 관련 자식 레코드도 함께 삭제되는 방식. |

### 4.2 Contact Attribute 시스템

#### 4.2.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-026-02 |
| 기능명 | Contact Attribute 시스템 |
| 관련 요구사항 ID | FR-045-02 |
| 우선순위 | 필수 |
| 기능 설명 | 연락처에 속성을 부여하기 위한 키-값 시스템을 제공한다. Attribute Key(속성 정의)와 Attribute Value(속성 값)로 구성되며, string/number/date 3종의 데이터 타입을 지원한다. |

#### 4.2.2 선행 조건 (Preconditions)

1. Enterprise 라이선스가 활성화되어 있어야 한다.
2. 유효한 Environment가 존재해야 한다.
3. (Attribute Value 생성 시) 대상 Contact가 존재해야 한다.
4. (Attribute Value 생성 시) 대상 Attribute Key가 존재해야 한다.

#### 4.2.3 후행 조건 (Postconditions)

1. Attribute Key가 Environment 내에서 고유하게 생성된다.
2. Attribute Value가 해당 Contact-Key 조합으로 고유하게 저장된다.
3. 데이터 타입에 따라 적절한 네이티브 컬럼(문자열/숫자/날짜)에 값이 저장된다.

#### 4.2.4 기본 흐름 (Basic Flow)

**Attribute Key 생성 흐름**

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 사용자/시스템 | 새 속성 키 생성을 요청한다 (키 이름, 데이터 타입, 표시 이름, 설명 등 전달). |
| 2 | 시스템 | 환경당 속성 키 수가 150개 미만인지 확인한다. |
| 3 | 시스템 | 키 이름이 safe identifier 규칙(`^[a-z][a-z0-9_]*$`)을 준수하는지 검증한다. |
| 4 | 시스템 | 동일 Environment 내에 동일 키 이름이 존재하지 않는지 확인한다. |
| 5 | 시스템 | 속성 키를 `custom` 종류로 생성하고 지정된 데이터 타입을 설정한다. |
| 6 | 시스템 | 생성된 속성 키 정보를 반환한다. |

**Attribute Value 설정 흐름**

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 사용자/시스템 | 연락처의 속성 값 설정을 요청한다 (연락처 ID, 속성 키 ID, 값 전달). |
| 2 | 시스템 | 해당 속성 키의 데이터 타입을 확인한다. |
| 3 | 시스템 | 전달된 값이 해당 데이터 타입에 유효한지 검증한다. |
| 4 | 시스템 | 값을 데이터 타입에 따라 적절한 컬럼에 저장한다. (string -> 문자열 값 컬럼, number -> 숫자 값 컬럼, date -> 날짜 값 컬럼. 하위 호환성을 위해 문자열 값 컬럼에도 항상 문자열 변환 값을 저장한다.) |
| 5 | 시스템 | 동일 Contact-Key 조합이 이미 존재하면 upsert로 업데이트한다. |
| 6 | 시스템 | 저장된 속성 값 정보를 반환한다. |

#### 4.2.5 대안 흐름 (Alternative Flow)

**AF-02-01: Attribute Key 수정**

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 사용자 | 기존 속성 키의 표시 이름 또는 설명 수정을 요청한다. |
| 2 | 시스템 | 해당 속성 키의 종류가 `custom`인지 확인한다. |
| 3 | 시스템 | `custom` 종류인 경우 수정을 적용하고 결과를 반환한다. |

**AF-02-02: Attribute Key 삭제**

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 사용자 | 속성 키 삭제를 요청한다. |
| 2 | 시스템 | 해당 속성 키의 종류가 `custom`인지 확인한다. |
| 3 | 시스템 | `custom` 종류인 경우 해당 키와 관련된 모든 Attribute Value를 cascade 삭제한 후 키를 삭제한다. |

#### 4.2.6 예외 흐름 (Exception Flow)

| 예외 ID | 조건 | 시스템 응답 |
|---------|------|-----------|
| EX-02-01 | default 타입 속성 키 수정 시도 | "허용되지 않는 작업" 에러 반환 (HTTP 403) |
| EX-02-02 | default 타입 속성 키 삭제 시도 | "허용되지 않는 작업" 에러 반환 (HTTP 403) |
| EX-02-03 | 키 이름이 safe identifier 규칙 위반 | 검증 에러 반환 (HTTP 400). 메시지: "속성 키 이름은 소문자로 시작하며, 소문자/숫자/언더스코어만 사용할 수 있습니다." |
| EX-02-04 | 환경당 속성 키 수 150개 초과 | 검증 에러 반환 (HTTP 400). 메시지: "환경당 최대 속성 키 수(150개)를 초과했습니다." |
| EX-02-05 | 동일 Environment 내 키 이름 중복 | 검증 에러 반환 (HTTP 409). 메시지: "이미 존재하는 속성 키 이름입니다." |
| EX-02-06 | 속성 값이 데이터 타입과 불일치 | 검증 에러 반환 (HTTP 400). 메시지: "'{값}'은(는) {타입} 형식에 맞지 않습니다." |

#### 4.2.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 규칙 | 조건 | 결과 |
|---------|------|------|------|
| BR-02-01 | default 속성 보호 | 속성 키의 종류가 `default`일 때 | 수정/삭제 불가 |
| BR-02-02 | 환경당 속성 키 상한 | 환경 내 속성 키 수가 150개 이상일 때 | 신규 속성 키 생성 차단 |
| BR-02-03 | 키 이름 유니크 제약 | 동일 Environment 내 | 키 이름 + 환경 ID 조합 유니크 |
| BR-02-04 | 속성 값 유니크 제약 | 동일 Contact-Key 조합 | 연락처 ID + 속성 키 ID 조합 유니크 |
| BR-02-05 | 하위 호환성 문자열 저장 | number/date 타입 값 저장 시 | 문자열 값 컬럼에도 항상 문자열로 변환하여 저장 |
| BR-02-06 | default 속성 키 목록 | 시스템 초기화 시 | userId, email, firstName, lastName이 default로 생성됨 |
| BR-02-07 | 고유성 속성 | 속성 키의 `isUnique`가 true일 때 | 동일 Environment 내에서 해당 속성 값이 유일해야 함 |

#### 4.2.8 데이터 요구사항

**Attribute Key 엔티티 필드 정의**

| 필드명 | 타입 | 필수 | 기본값 | 설명 | 유효성 검증 규칙 |
|--------|------|------|--------|------|-----------------|
| id | String (CUID) | Y | 자동 생성 | 고유 식별자 | 시스템 자동 생성 |
| isUnique | Boolean | Y | false | 해당 속성 값의 유니크 여부 | - |
| key | String | Y | - | 속성의 고유 키 이름 | `^[a-z][a-z0-9_]*$` (safe identifier) |
| name | String | N | null | 사용자에게 보이는 표시 이름 | - |
| description | String | N | null | 속성 설명 | - |
| type | Enum | Y | - | 속성 종류: `default` 또는 `custom` | `default`, `custom` 중 하나 |
| dataType | Enum | Y | `string` | 데이터 타입 | `string`, `number`, `date` 중 하나 |
| environmentId | String | Y | - | 소속 환경 식별자 | 유효한 Environment ID 참조 |

**유니크 제약**: `key` + `environmentId` 조합

**Attribute Value 엔티티 필드 정의**

| 필드명 | 타입 | 필수 | 기본값 | 설명 | 유효성 검증 규칙 |
|--------|------|------|--------|------|-----------------|
| id | String (CUID) | Y | 자동 생성 | 고유 식별자 | 시스템 자동 생성 |
| contactAttributeKeyId | String | Y | - | 연결된 속성 키 식별자 | 유효한 Attribute Key ID 참조 |
| contactId | String | Y | - | 연결된 연락처 식별자 | 유효한 Contact ID 참조 |
| value | String | Y | - | 문자열 형태의 값 (하위 호환성) | - |
| numberValue | Float | N | null | number 타입 전용 네이티브 저장 | number 타입일 때만 설정 |
| dateValue | DateTime | N | null | date 타입 전용 네이티브 저장 | date 타입일 때만 설정 |

**유니크 제약**: `contactId` + `contactAttributeKeyId` 조합

**인덱스**

| 인덱스 대상 | 용도 |
|------------|------|
| contactAttributeKeyId + value | 문자열 속성 값 기반 검색 |
| contactAttributeKeyId + numberValue | 숫자 속성 값 기반 검색/비교 |
| contactAttributeKeyId + dateValue | 날짜 속성 값 기반 검색/비교 |

#### 4.2.9 화면/UI 요구사항

해당 없음 - 속성 관리는 API 및 시스템 내부 처리 위주.

#### 4.2.10 비기능 요구사항

| 항목 | 내용 |
|------|------|
| 데이터 무결성 | Foreign Key 및 Unique 제약으로 참조/중복 무결성 보장 |

### 4.3 CSV Import

#### 4.3.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-026-03 |
| 기능명 | CSV Import |
| 관련 요구사항 ID | FR-045-03 |
| 우선순위 | 필수 |
| 기능 설명 | CSV 파일을 통해 연락처를 대량으로 가져온다. 최대 10,000건까지 지원하며, 중복 처리 전략(skip, update, overwrite)을 선택할 수 있다. 속성 데이터 타입은 자동 탐지되며, 누락된 속성 키는 자동 생성된다. |

#### 4.3.2 선행 조건 (Preconditions)

1. Enterprise 라이선스가 활성화되어 있어야 한다.
2. 유효한 Environment가 존재해야 한다.
3. 사용자가 Organization Owner 또는 Manager 역할이어야 한다.
4. 업로드할 CSV 파일이 준비되어 있어야 한다.
5. CSV 파일의 모든 레코드에 email 필드가 포함되어 있어야 한다.

#### 4.3.3 후행 조건 (Postconditions)

1. CSV 파일의 유효한 레코드에 대해 연락처가 생성 또는 업데이트된다.
2. 누락된 속성 키가 자동으로 `custom` 종류로 생성된다.
3. 속성 값이 탐지된 데이터 타입에 따라 적절한 컬럼에 저장된다.
4. 중복 처리 전략에 따라 기존 연락처가 처리된다.

#### 4.3.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 사용자 | CSV 파일과 중복 처리 전략(skip/update/overwrite)을 선택하여 업로드를 요청한다. |
| 2 | 시스템 | CSV 파일을 파싱하여 레코드 수가 10,000건 이하인지 확인한다. |
| 3 | 시스템 | 모든 레코드에 email 필드가 존재하는지 확인한다. |
| 4 | 시스템 | CSV 내 email 중복 여부를 확인한다. |
| 5 | 시스템 | CSV 내 userId 중복 여부를 확인한다. |
| 6 | 시스템 | 모든 속성 키 이름이 safe identifier 규칙을 준수하는지 확인한다. |
| 7 | 시스템 | CSV 데이터에서 메타데이터를 추출한다 (email 목록, userId 목록, 속성 키 목록). |
| 8 | 시스템 | DB에서 기존 데이터를 병렬로 조회한다: (a) email로 기존 연락처 조회, (b) userId로 기존 연락처 조회, (c) 환경의 기존 속성 키 조회. |
| 9 | 시스템 | Lookup Map을 구성한다: email-to-contact 매핑, attributeKey 매핑. |
| 10 | 시스템 | 속성 타입 자동 탐지 및 검증을 수행한다: (a) 기존 typed 속성: CSV 값의 타입이 기존 타입과 일치하는지 검증 -- 불일치 시 validation error 반환, (b) 새 속성: 모든 값의 타입을 탐지하여 Date > Number > String 우선순위로 결정 -- 값이 혼재되면 string으로 downgrade. |
| 11 | 시스템 | 누락된 속성 키를 `custom` 종류, 탐지된 데이터 타입으로 자동 생성한다. |
| 12 | 시스템 | 각 CSV 레코드를 순회하며 중복 처리 전략에 따라 처리한다 (상세는 대안 흐름 참조). |
| 13 | 시스템 | 처리 결과(생성 건수, 업데이트 건수, 스킵 건수, 에러 건수)를 반환한다. |

#### 4.3.5 대안 흐름 (Alternative Flow)

**AF-03-01: 중복 처리 전략 - skip**

| 단계 | 주체 | 동작 |
|------|------|------|
| 12a-1 | 시스템 | CSV 레코드의 email로 Lookup Map에서 기존 연락처를 검색한다. |
| 12a-2 | 시스템 | 기존 연락처가 존재하면 해당 레코드를 무시하고 스킵 카운트를 증가시킨다. |
| 12a-3 | 시스템 | 기존 연락처가 없으면 새 연락처를 생성하고 속성 값을 설정한다. |

**AF-03-02: 중복 처리 전략 - update**

| 단계 | 주체 | 동작 |
|------|------|------|
| 12b-1 | 시스템 | CSV 레코드의 email로 Lookup Map에서 기존 연락처를 검색한다. |
| 12b-2 | 시스템 | 기존 연락처가 존재하면 기존 속성은 유지하면서 CSV의 속성 값으로 upsert한다 (새 속성 추가, 기존 속성 갱신). |
| 12b-3 | 시스템 | 기존 연락처가 없으면 새 연락처를 생성하고 속성 값을 설정한다. |

**AF-03-03: 중복 처리 전략 - overwrite**

| 단계 | 주체 | 동작 |
|------|------|------|
| 12c-1 | 시스템 | CSV 레코드의 email로 Lookup Map에서 기존 연락처를 검색한다. |
| 12c-2 | 시스템 | 기존 연락처가 존재하면 해당 연락처의 모든 기존 속성 값을 삭제한다. |
| 12c-3 | 시스템 | CSV의 속성 값으로 새로 속성 값을 생성한다. |
| 12c-4 | 시스템 | 기존 연락처가 없으면 새 연락처를 생성하고 속성 값을 설정한다. |

**AF-03-04: userId 충돌 해결**

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | CSV 레코드의 userId가 다른 기존 연락처에 이미 할당되어 있는지 확인한다. |
| 2 | 시스템 | 이미 할당된 경우, CSV 레코드에서 userId를 제거한다. |
| 3 | 시스템 | 기존 연락처의 userId는 그대로 유지한다. |

#### 4.3.6 예외 흐름 (Exception Flow)

| 예외 ID | 조건 | 시스템 응답 |
|---------|------|-----------|
| EX-03-01 | CSV 레코드 수 10,000건 초과 | 검증 에러 반환 (HTTP 400). 메시지: "CSV 파일의 레코드 수는 10,000건을 초과할 수 없습니다." |
| EX-03-02 | email 필드 누락 레코드 존재 | 검증 에러 반환 (HTTP 400). 메시지: "모든 레코드에 email 필드가 필요합니다. {N}번째 행에 email이 없습니다." |
| EX-03-03 | CSV 내 email 중복 | 검증 에러 반환 (HTTP 400). 메시지: "CSV 파일 내에 중복된 email이 있습니다: {email}" |
| EX-03-04 | CSV 내 userId 중복 | 검증 에러 반환 (HTTP 400). 메시지: "CSV 파일 내에 중복된 userId가 있습니다: {userId}" |

#### 4.3.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 규칙 | 조건 | 결과 |
|---------|------|------|------|
| BR-03-01 | CSV 최대 레코드 수 | 레코드 수 > 10,000 | 업로드 거부 |
| BR-03-02 | email 필수 | CSV 레코드에 email이 없을 때 | 검증 에러 |
| BR-03-03 | email 유니크 (CSV 내) | CSV 내 동일 email 존재 | 검증 에러 |
| BR-03-04 | userId 유니크 (CSV 내) | CSV 내 동일 userId 존재 | 검증 에러 |
| BR-03-05 | 타입 자동 탐지 우선순위 | 새 속성의 값 타입 결정 시 | Date > Number > String 순 |
| BR-03-06 | 혼재 타입 downgrade | 새 속성의 값이 여러 타입으로 혼재될 때 | string으로 downgrade |
| BR-03-07 | 기존 속성 타입 엄격 검증 | 기존 typed 속성에 다른 타입 값 입력 시 | validation error |
| BR-03-08 | userId 충돌 해결 | CSV의 userId가 다른 연락처에 할당된 경우 | CSV의 userId를 제거하고 기존 연락처의 userId 유지 |
| BR-03-09 | 누락 속성 키 자동 생성 | CSV에 존재하지만 DB에 없는 속성 키 | `custom` 종류로 자동 생성 |

### 4.4 API v2 Bulk Upload

#### 4.4.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-026-04 |
| 기능명 | API v2 Bulk Upload |
| 관련 요구사항 ID | FR-045-04 |
| 우선순위 | 필수 |
| 기능 설명 | API v2를 통해 최대 250건의 연락처를 한 번에 생성한다. 각 연락처에 email이 필수이며, 연락처 간 email/userId 중복이 불가하다. |

#### 4.4.2 선행 조건 (Preconditions)

1. Enterprise 라이선스가 활성화되어 있어야 한다.
2. 유효한 API Key가 제공되어야 한다.
3. 유효한 Environment가 존재해야 한다.

#### 4.4.3 후행 조건 (Postconditions)

1. 요청된 연락처가 Environment에 생성된다.
2. 각 연락처의 속성 값이 저장된다.

#### 4.4.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | API Consumer | API v2 bulk upload 엔드포인트에 연락처 배열을 전송한다 (각 연락처에 email 필수, 속성 선택). |

| 3 | 시스템 | 연락처 배열의 크기가 250건 이하인지 확인한다. |
| 4 | 시스템 | 각 연락처의 email이 유효한 이메일 형식인지 확인한다. |
| 5 | 시스템 | 요청 내 연락처 간 email 중복 여부를 확인한다. |
| 6 | 시스템 | 요청 내 연락처 간 userId 중복 여부를 확인한다. |
| 7 | 시스템 | 각 연락처 내에서 속성 키 중복 여부를 확인한다. |
| 8 | 시스템 | 검증 통과 후 연락처를 생성하고 속성 값을 저장한다. |
| 9 | 시스템 | 생성 결과를 반환한다. |

#### 4.4.5 대안 흐름 (Alternative Flow)

**AF-04-01: 단건 연락처 생성**

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | API Consumer | 단건 연락처 생성 엔드포인트에 연락처 정보를 전송한다. |
| 2 | 시스템 | email 유효성 검증 후 연락처를 생성한다. |
| 3 | 시스템 | 생성된 연락처 정보를 반환한다. |

#### 4.4.6 예외 흐름 (Exception Flow)

### 4.6 Contact 삭제

#### 4.6.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-026-06 |
| 기능명 | Contact 삭제 |
| 관련 요구사항 ID | FR-045-06 |
| 우선순위 | 필수 |
| 기능 설명 | 연락처를 데이터베이스에서 완전히 삭제(hard delete)한다. 관련된 속성 값, 응답 참조, 표시 기록이 cascade로 함께 삭제된다. |

#### 4.6.2 선행 조건 (Preconditions)

1. Enterprise 라이선스가 활성화되어 있어야 한다.
2. 삭제 대상 연락처가 존재해야 한다.
3. 사용자가 Organization Owner 또는 Manager 역할이어야 한다.

#### 4.6.3 후행 조건 (Postconditions)

1. 연락처 레코드가 데이터베이스에서 완전히 삭제된다.
2. 해당 연락처의 모든 속성 값(ContactAttribute)이 cascade 삭제된다.
3. 해당 연락처의 응답 참조, 표시 기록이 정리된다.

#### 4.6.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 사용자 | 연락처 목록에서 삭제 대상 연락처를 선택하고 삭제를 요청한다. |
| 2 | 시스템 | 삭제 확인 다이얼로그를 표시한다. |
| 3 | 사용자 | 삭제를 확인한다. |
| 4 | 시스템 | 해당 연락처와 관련된 모든 속성 값을 cascade 삭제한다. |
| 5 | 시스템 | 연락처 레코드를 데이터베이스에서 삭제한다. |
| 6 | 시스템 | 삭제 완료 메시지를 표시하고 연락처 목록을 갱신한다. |

#### 4.6.5 대안 흐름 (Alternative Flow)

**AF-06-01: 삭제 취소**

| 단계 | 주체 | 동작 |
|------|------|------|
| 3a | 사용자 | 삭제 확인 다이얼로그에서 취소를 선택한다. |
| 4a | 시스템 | 삭제를 중단하고 연락처 목록으로 돌아간다. |

#### 4.6.6 예외 흐름 (Exception Flow)

| 예외 ID | 조건 | 시스템 응답 |
|---------|------|-----------|
| EX-06-01 | 삭제 대상 연락처가 존재하지 않음 | 에러 반환 (HTTP 404). 메시지: "연락처를 찾을 수 없습니다." |

#### 4.6.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 규칙 | 조건 | 결과 |
|---------|------|------|------|
| BR-06-01 | Hard Delete Only | 연락처 삭제 시 항상 | 데이터베이스에서 완전 삭제 (소프트 삭제 미지원) |
| BR-06-02 | Cascade 삭제 | 연락처 삭제 시 항상 | 관련 속성 값, 응답 참조, 표시 기록 함께 삭제 |
| BR-06-03 | GDPR 준수 | 연락처 삭제 요청 시 | 연락처의 모든 개인 데이터 완전 삭제 |

#### 4.6.8 데이터 요구사항

**출력 데이터**

### 4.8 SDK 자동 연락처 생성

#### 4.8.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-026-08 |
| 기능명 | SDK 자동 연락처 생성 |
| 관련 요구사항 ID | FR-045-08 |
| 우선순위 | 필수 |
| 기능 설명 | JavaScript SDK에서 사용자 식별(identify) 호출 시, 해당 userId의 연락처가 존재하지 않으면 자동으로 생성한다. SDK의 타입 탐지는 CSV보다 엄격한 규칙을 따른다. |

#### 4.8.2 선행 조건 (Preconditions)

1. Enterprise 라이선스가 활성화되어 있어야 한다.
2. SDK가 올바른 Environment ID로 초기화되어 있어야 한다.
3. SDK에서 identify 호출 시 userId가 전달되어야 한다.

#### 4.8.3 후행 조건 (Postconditions)

1. 해당 userId의 연락처가 존재하지 않았다면 새로 생성된다.
2. 전달된 속성 값이 타입 탐지 규칙에 따라 저장된다.
3. 해당 userId의 연락처가 이미 존재했다면 속성이 업데이트된다.

#### 4.8.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | SDK Consumer | SDK의 `identify` 메서드를 호출한다 (userId, 속성 객체 전달). |
| 2 | SDK | Client API에 사용자 식별 요청을 전송한다. |
| 3 | 시스템 | 해당 Environment에서 전달된 userId로 기존 연락처를 조회한다. |
| 4 | 시스템 | 기존 연락처가 없으면 새 연락처를 생성하고 userId를 설정한다. |
| 5 | 시스템 | 전달된 속성 값을 SDK 타입 탐지 규칙에 따라 분류한다. |
| 6 | 시스템 | 속성 키가 존재하지 않으면 탐지된 타입으로 자동 생성한다. |
| 7 | 시스템 | 속성 값을 연락처에 upsert한다. |
| 8 | 시스템 | 처리 결과를 SDK에 반환한다. |

#### 4.8.5 대안 흐름 (Alternative Flow)

**AF-08-01: 기존 연락처 업데이트**

| 단계 | 주체 | 동작 |
|------|------|------|
| 4a | 시스템 | 해당 userId의 기존 연락처가 존재하면 새 연락처를 생성하지 않는다. |
| 5a | 시스템 | 전달된 속성 값을 기존 연락처에 upsert한다. |

#### 4.8.6 예외 흐름 (Exception Flow)

| 예외 ID | 조건 | 시스템 응답 |
|---------|------|-----------|
| EX-08-01 | Enterprise 라이선스 미활성 | SDK에 에러 응답, 연락처 생성/업데이트 미수행 |
| EX-08-02 | userId 미전달 | SDK에 에러 응답: "userId가 필요합니다." |

#### 4.8.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 규칙 | 조건 | 결과 |
|---------|------|------|------|
| BR-08-01 | 자동 생성 | userId에 해당하는 연락처가 없을 때 | 연락처 자동 생성 |
| BR-08-02 | SDK 타입 탐지 - number | JavaScript number 타입인 경우 | `number`로 분류 |
| BR-08-03 | SDK 타입 탐지 - date | ISO 8601 형식 문자열인 경우 | `date`로 분류 |
| BR-08-04 | SDK 타입 탐지 - string (기본) | 위 두 조건에 해당하지 않는 모든 값 | `string`으로 분류 (숫자처럼 보이는 문자열 포함) |
| BR-08-05 | SDK vs CSV 타입 탐지 차이 | 숫자처럼 보이는 문자열 (예: "123") | CSV: `number`로 탐지 / SDK: `string`으로 탐지 |

**SDK 타입 탐지 vs CSV 타입 탐지 비교**

| 입력 값 | CSV 타입 탐지 결과 | SDK 타입 탐지 결과 |
|---------|------------------|------------------|

| `"42"` (문자열) | number | string |
| `"2026-02-21T00:00:00Z"` (ISO 8601) | date | date |
| `"21-02-2026"` (DD-MM-YYYY) | date | string |
| `"hello"` | string | string |
