## 2. 개요

### 2.1 목적

본 문서는 Inquiry JS SDK의 초기화, 위젯 배치/표시, No-Code Action 감지, GTM 통합에 대한 상세 기능 명세를 정의한다. 요구사항 명세서 FSD-007(FR-010, FR-042)을 기반으로 클라이언트 사이드 SDK 런타임의 모든 동작을 구체적으로 기술하여 개발, 테스트, 검증의 기준 문서로 활용한다.

### 2.2 범위

**포함 범위 (In-scope)**:
- JS SDK 초기화 및 설정 흐름
- 위젯 배치(Placement) 5가지 유형
- 오버레이(Overlay) 3가지 유형
- 설문 표시 모드 (modal, inline/fullwidth)
- 외부 클릭 닫기 옵션
- 설문별 프로젝트 설정 오버라이드 (Per-Survey Override)
- No-Code Action 감지 메커니즘 (4가지 이벤트 유형)
- GTM(Google Tag Manager) 통합 방식
- 상태 관리 (Environment State, User State)
- 에러 상태 관리 및 정리 처리

**제외 범위 (Out-of-scope)**:
- 서버 사이드 API 엔드포인트 구현 상세
- 설문 응답 수집/저장 로직
- Contact/Segment 서버 로직

### 2.3 대상 사용자

| 역할 | 설명 |
|------|------|
| 프론트엔드 개발자 | SDK를 자사 웹 애플리케이션에 통합하는 개발자 |
| 마케팅 담당자 | GTM을 통해 코드 변경 없이 설문을 배포하고 관리하는 담당자 |
| SDK 런타임 | 자동 환경 동기화 및 설문 렌더링을 수행하는 시스템 컴포넌트 |

### 2.4 용어 정의

| 용어 | 정의 |
|------|------|
| Environment ID | Inquiry Environment의 CUID 형식 고유 식별자. SDK 초기화 시 필수 |
| App URL | Inquiry 서버의 기본 URL |
| Environment State | 환경 관련 데이터(설문 목록, Action Class, 프로젝트 설정 등)를 포함하는 상태 객체 |
| User State | 사용자 관련 데이터(사용자 ID, 세그먼트, 표시 이력, 응답 목록 등)를 포함하는 상태 객체 |
| Placement | 인앱 설문 위젯의 화면 배치 위치. bottomLeft, bottomRight, topLeft, topRight, center |
| Overlay | 인앱 설문 표시 시 배경 오버레이 스타일. none, light, dark |
| Action Class | 사용자 행동을 정의하는 클래스. 설문 트리거 조건에 사용 |
| No-Code Action | 코드 작성 없이 설정만으로 감지하는 사용자 행동(페이지 방문, 클릭, 이탈 의도, 스크롤) |
| GTM | Google Tag Manager. 웹사이트에 태그를 관리/배포하는 도구 |
| Per-Survey Override | 개별 설문에서 프로젝트 기본 설정을 덮어쓰는 설정 |
| Hidden Fields | 설문 URL의 쿼리 파라미터로 전달되는 숨겨진 필드. 응답 데이터에 포함되지만 응답자에게 표시되지 않음 |
| Display | 설문이 사용자에게 표시된 기록 |
| SPA | Single Page Application. 페이지 전체를 새로 로드하지 않고 동적으로 콘텐츠를 갱신하는 웹 애플리케이션 |
| UMD | Universal Module Definition. 여러 모듈 시스템에서 사용 가능한 JavaScript 번들 형식 |
| ESM | ECMAScript Module. 표준 JavaScript 모듈 시스템 |
| Shadow DOM | 호스트 페이지의 스타일과 격리된 DOM 트리를 생성하는 웹 표준 기술 |

## 3. 시스템 개요

### 3.1 시스템 구성도

```
+---------------------+       +----------------------+       +-------------------+
|   호스트 웹 앱       |       |  Inquiry JS SDK   |       | Inquiry Server |
|                     |       |                      |       |                   |
| - SDK 초기화 코드    | ----> | - 설정 관리 (Config) |<----->| - Environment API |
| - 또는 GTM 태그     |       | - 상태 관리          |       | - User API        |
|                     |       | - Action 감지        |       | - Survey Data     |
|                     |       | - 위젯 렌더링        |       |                   |
+---------------------+       +-------+--------------+       +-------------------+
                                      |
                              +-------v--------------+
                              |   LocalStorage        |
                              |   (formbricks-js)     |
                              +----------------------+
```

SDK는 호스트 웹 애플리케이션에 삽입되어 Inquiry 서버와 통신하며, LocalStorage를 통해 설정 상태를 영속화한다.

### 3.3 기능 간 관계도

```
FN-07-01 SDK 초기화
    |
    +---> FN-07-08 에러 상태 관리
    |         |
    |         +---> FN-07-09 정리 처리 (tearDown)
    |
    +---> FN-07-06 No-Code Action 감지
    |         |
    |         +---> FN-07-05 위젯 렌더링
    |                   |
    |                   +---> FN-07-02 위젯 배치 및 표시
    |                   |         |
    |                   |         +---> FN-07-03 외부 클릭 닫기
    |                   |
    |                   +---> FN-07-04 설문별 오버라이드
    |                   |
    |                   +---> FN-07-10 설문 닫기
    |
    +---> FN-07-07 GTM 통합 (SDK 초기화의 대체 진입점)
```

- SDK 초기화(FN-07-01)가 모든 기능의 전제 조건

- No-Code Action 감지(FN-07-06)가 위젯 렌더링(FN-07-05)을 트리거
- 위젯 렌더링 시 배치(FN-07-02)와 오버라이드(FN-07-04) 설정이 적용됨
- 에러 상태(FN-07-08) 진입 시 정리 처리(FN-07-09)가 수행됨

## 4. 상세 기능 명세

### 4.1 SDK 초기화 및 구성 (FN-07-01)

#### 4.1.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-07-01 |
| 기능명 | SDK 초기화 및 구성 |
| 관련 요구사항 ID | FR-010 (4.1~4.6), AC-010-01, AC-010-02 |
| 우선순위 | 필수 |
| 기능 설명 | Environment ID와 App URL을 입력받아 SDK를 초기화하고, 서버에서 환경/사용자 상태를 조회하여 설정을 구성한 뒤 LocalStorage에 영속화한다 |

#### 4.1.2 선행 조건 (Preconditions)

1. 호스트 웹 페이지에 Inquiry JS SDK 스크립트가 로드되어 있어야 한다
2. Inquiry 서버가 접근 가능한 상태여야 한다 (최초 초기화 시)
3. 유효한 Environment ID가 사전에 Inquiry 시스템에 등록되어 있어야 한다

#### 4.1.3 후행 조건 (Postconditions)

1. SDK 내부 설정 상태가 구성 완료된다
2. LocalStorage `formbricks-js` 키에 설정이 저장된다
3. 초기화 완료 플래그가 true로 설정된다
4. 이벤트 리스너가 등록된다 (No-Code Action 감지용)
5. 필터링된 설문 목록이 계산된다

#### 4.1.4 기본 흐름 (Basic Flow)

| 단계 | 행위자 | 동작 |
|------|--------|------|
| 1 | 호스트 앱 | SDK 초기화 함수를 호출하며 Environment ID, App URL을 전달한다 |
| 2 | SDK | 디버그 모드 여부를 확인한다. 디버그 모드이면 로그 레벨을 `debug`로 설정한다 |
| 3 | SDK | LocalStorage에서 Legacy 포맷 데이터를 확인한다. Legacy 포맷(환경 상태/개인 상태)이 존재하면 새 포맷(환경/사용자)으로 자동 마이그레이션한다 |
| 4 | SDK | 이미 초기화 완료 상태인지 확인한다. 초기화 완료 상태이면 초기화를 스킵한다 |
| 5 | SDK | 에러 상태인지 확인한다. 에러 상태이면 만료 시간을 확인하고, 만료 전이면 초기화를 스킵한다 |
| 6 | SDK | Environment ID와 App URL의 존재 여부를 검증한다 |
| 7 | SDK | LocalStorage에서 기존 설정을 로드한다. 유효한 설정이 존재하면 만료 여부를 확인하고, 만료되지 않았으면 증분 동기화를 수행한다 |
| 8 | SDK | 서버에 환경 상태(Environment State) 조회 요청을 보낸다 |
| 9 | 서버 | 해당 Environment ID에 대한 환경 상태를 응답한다 |
| 10 | SDK | 사용자 ID가 존재하면 서버에 사용자 상태 업데이트를 전송한다 |
| 11 | SDK | 환경 상태와 사용자 상태를 기반으로 설문 필터링을 실행한다 |
| 12 | SDK | No-Code Action 이벤트 리스너 및 정리(cleanup) 이벤트 리스너를 등록한다 |
| 13 | SDK | 초기화 완료 플래그를 true로 설정한다 |
| 14 | SDK | 설정을 LocalStorage `formbricks-js` 키에 저장한다 |

#### 4.1.5 대안 흐름 (Alternative Flow)

**AF-01-01: 이미 초기화 완료 상태**
- 단계 4에서 초기화 완료 플래그가 true인 경우
- 초기화 과정을 스킵하고 기존 설정을 유지한다
- 후속 단계로 진행하지 않는다

**AF-01-02: 에러 상태이지만 만료됨**
- 단계 5에서 에러 상태의 만료 시간이 경과한 경우
- 에러 상태를 해제하고 단계 6부터 초기화를 재시도한다

**AF-01-03: 디버그 모드에서 에러 상태**
- 단계 5에서 에러 상태이지만 디버그 모드가 활성화된 경우
- 에러 상태를 무시하고 단계 6부터 초기화를 진행한다

**AF-01-04: 기존 설정 존재 및 유효**
- 단계 7에서 LocalStorage에 유효하고 만료되지 않은 설정이 존재하는 경우
- 전체 동기화 대신 증분 동기화만 수행한다
- 서버에서 변경된 데이터만 조회하여 기존 설정을 업데이트한다

**AF-01-05: 고급 초기화 (식별된 사용자)**
- 단계 1에서 사용자 ID와 속성(키-값 쌍)이 함께 전달되는 경우
- 단계 10에서 사용자 ID 및 속성 데이터를 서버에 전송한다
- 서버에서 해당 사용자의 세그먼트, 표시 이력 등을 포함한 User State를 수신한다

#### 4.1.6 예외 흐름 (Exception Flow)

**EX-01-01: 필수 필드 누락**
- 단계 6에서 Environment ID 또는 App URL이 누락된 경우
- SDK가 "필수 필드 누락" 에러를 반환한다
- 초기화가 중단되며, SDK는 비활성 상태로 유지된다

**EX-01-02: 서버 통신 실패**
- 단계 8 또는 10에서 서버 응답이 실패하는 경우
- SDK가 에러 상태로 전환된다 (FN-07-08 참조)
- 에러 상태 지속 시간 600,000ms (10분) 설정

**EX-01-03: LocalStorage 접근 불가**
- LocalStorage가 비활성화되었거나 접근 권한이 없는 경우
- SDK가 메모리 내에서만 설정을 관리하며, 페이지 새로고침 시 설정이 초기화된다

#### 4.1.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 조건 | 동작 |
|---------|------|------|
| BR-01-01 | Environment ID가 빈 문자열이거나 null/undefined | 초기화를 중단하고 필수 필드 누락 에러 반환 |
| BR-01-02 | App URL이 빈 문자열이거나 null/undefined | 초기화를 중단하고 필수 필드 누락 에러 반환 |
| BR-01-03 | 에러 상태이고 현재 시각 < 에러 만료 시각 | 초기화를 스킵 |
| BR-01-04 | 에러 상태이고 현재 시각 >= 에러 만료 시각 | 에러 상태를 해제하고 초기화 재시도 |
| BR-01-05 | 에러 상태이고 디버그 모드 활성화 | 에러 상태를 무시하고 초기화 진행 |
| BR-01-06 | 설정이 Singleton 패턴으로 관리됨 | 하나의 SDK 인스턴스만 존재 |
| BR-01-07 | Legacy 포맷 데이터가 LocalStorage에 존재 | 새 포맷으로 자동 마이그레이션 후 초기화 진행 |

#### 4.1.8 데이터 요구사항

**입력 데이터**:

| 필드명 | 타입 | 필수 여부 | 유효성 검증 규칙 | 설명 |
|--------|------|-----------|-----------------|------|
| environmentId | string | 필수 | 빈 문자열 불가, CUID 형식 | Inquiry Environment 고유 식별자 |
| appUrl | string | 필수 | 빈 문자열 불가, 유효한 URL 형식 | Inquiry 서버 URL |
| userId | string | 선택 | 빈 문자열 불가 (전달 시) | 식별된 사용자의 고유 ID |
| attributes | Record<string, string> | 선택 | 키-값 쌍, 키와 값 모두 문자열 | 사용자 속성 |

**출력 데이터 (SDK 내부 설정 상태)**:

| 필드명 | 타입 | 설명 |
|--------|------|------|
| environmentId | string | 초기화에 사용된 환경 ID |
| appUrl | string | Inquiry 서버 URL |
| environmentState | EnvironmentState | 환경 관련 데이터 |
| userState | UserState | 사용자 관련 데이터 |
| filteredSurveys | Survey[] | 조건에 맞는 설문 목록 |
| status | "success" \| { type: "error"; expiresAt: Date } | SDK 상태 정보 |

**Environment State 구조**:

| 필드명 | 타입 | 설명 |
|--------|------|------|
| expiresAt | Date | 상태 캐시 만료 시점 |
| surveys | Survey[] | 현재 환경의 설문 데이터 |
| actionClasses | ActionClass[] | 현재 환경의 액션 정의 |
| project | ProjectConfig | 프로젝트 정보 (아래 상세) |
| recaptchaSiteKey | string \| null | 스팸 방지용 reCAPTCHA 사이트 키 |

**ProjectConfig 구조**:

| 필드명 | 타입 | 설명 |
|--------|------|------|
| id | string | 프로젝트 고유 ID |
| recontactDays | number | 재접촉 대기 일수 |
| clickOutsideClose | boolean | 외부 클릭 닫기 기본값 |
| overlay | "none" \| "light" \| "dark" | 오버레이 유형 기본값 |
| placement | "bottomLeft" \| "bottomRight" \| "topLeft" \| "topRight" \| "center" | 위젯 배치 기본값 |
| inAppSurveyBranding | boolean | In-app 설문 브랜딩 표시 여부 |
| styling | object | 스타일링 설정 |

**User State 구조**:

### 4.2 위젯 배치 및 표시 (FN-07-02)

#### 4.2.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-07-02 |
| 기능명 | 위젯 배치 및 표시 |
| 관련 요구사항 ID | FR-010 (4.7~4.9), AC-010-03, AC-010-04 |
| 우선순위 | 필수 |
| 기능 설명 | 설문 위젯을 5가지 배치 위치 중 하나에, 3가지 오버레이 유형 중 하나와 함께, 2가지 표시 모드(modal/inline) 중 하나로 렌더링한다 |

#### 4.2.2 선행 조건 (Preconditions)

1. SDK가 성공적으로 초기화되어 있어야 한다 (FN-07-01 완료)
2. 표시할 설문 데이터가 존재해야 한다
3. 현재 다른 설문이 표시 중이 아니어야 한다

#### 4.2.3 후행 조건 (Postconditions)

1. 위젯 컨테이너가 DOM의 body에 추가된다
2. 설문이 지정된 배치 위치에 표시된다
3. 지정된 오버레이가 적용된다
4. 설문 실행 중 플래그가 true로 설정된다

#### 4.2.4 기본 흐름 (Basic Flow)

| 단계 | 행위자 | 동작 |
|------|--------|------|
| 1 | SDK | 설문의 표시 모드를 결정한다. app 타입 설문은 modal, link 타입 설문은 fullwidth(inline)로 설정한다 |
| 2 | SDK | 위젯 배치 위치를 결정한다. 설문별 오버라이드가 있으면 적용하고, 없으면 프로젝트 기본값(bottomRight)을 사용한다 |
| 3 | SDK | 오버레이 유형을 결정한다. 설문별 오버라이드가 있으면 적용하고, 없으면 프로젝트 기본값(none)을 사용한다 |
| 4 | SDK | 위젯 컨테이너 div 요소(id: `formbricks-modal-container`)를 body에 추가한다 |
| 5 | SDK | 결정된 배치 위치, 오버레이, 표시 모드에 따라 설문을 렌더링한다 |
| 6 | 사용자 | 설문을 확인하고 상호작용한다 |

#### 4.2.5 대안 흐름 (Alternative Flow)

**AF-02-01: center 배치 + 오버레이 적용**
- 단계 2에서 배치 위치가 center인 경우
- 화면 중앙에 설문을 배치하며, 오버레이가 none이 아닌 경우 배경을 함께 표시한다

**AF-02-02: inline/fullwidth 표시 모드**
- 단계 1에서 link 타입 설문인 경우
- modal 대신 페이지에 직접 삽입되는 fullwidth 모드로 표시한다
- 오버레이는 적용되지 않는다

#### 4.2.6 예외 흐름 (Exception Flow)

**EX-02-01: 위젯 컨테이너 중복**
- 단계 4에서 이미 `formbricks-modal-container` 요소가 DOM에 존재하는 경우
- 기존 컨테이너를 제거한 후 새로운 컨테이너를 추가한다

#### 4.2.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 조건 | 동작 |
|---------|------|------|
| BR-02-01 | 설문 타입이 app | modal 모드로 표시 |
| BR-02-02 | 설문 타입이 link | fullwidth(inline) 모드로 표시 |
| BR-02-03 | Placement 미지정 | bottomRight(기본값) 적용 |
| BR-02-04 | Overlay 미지정 | none(기본값) 적용 |
| BR-02-05 | 설문별 오버라이드 존재 | 프로젝트 기본값보다 우선 적용 |

#### 4.2.8 데이터 요구사항

**Placement 열거 값**:

| 값 | 설명 |
|-----|------|
| bottomLeft | 화면 좌하단 |
| bottomRight | 화면 우하단 (기본값) |
| topLeft | 화면 좌상단 |
| topRight | 화면 우상단 |
| center | 화면 중앙 |

**Overlay 열거 값**:

| 값 | 설명 |
|-----|------|
| none | 배경 오버레이 없음 (기본값) |
| light | 밝은 반투명 배경 오버레이 |
| dark | 어두운 반투명 배경 오버레이 |

**표시 모드**:

| 값 | 설명 | 적용 조건 |
|-----|------|-----------|
| modal | 오버레이와 함께 팝업 형태 | app 타입 설문 |
| fullwidth (inline) | 페이지에 직접 삽입 | link 타입 설문 |

#### 4.2.9 화면/UI 요구사항

- 위젯 컨테이너 ID: `formbricks-modal-container`
- 위젯은 Shadow DOM 기반으로 호스트 페이지 스타일과 격리되어야 한다
- 5가지 Placement 각각에 대해 화면 내 정확한 위치에 배치되어야 한다
- Overlay가 light일 때 밝은 반투명 배경, dark일 때 어두운 반투명 배경이 적용된다
- modal 모드에서 오버레이는 전체 뷰포트를 덮는다

#### 4.2.10 비기능 요구사항

| 항목 | 요구사항 |
|------|----------|
| 스타일 격리 | Shadow DOM 기반으로 호스트 페이지 CSS와 완전 격리 |
| CSP 호환 | nonce 지원으로 Content Security Policy와 호환 |

### 4.5 위젯 렌더링 (FN-07-05)

1. 설문 위젯이 화면에 표시된다

| 단계 | 행위자 | 동작 |

| 5 | SDK | 설문에 딜레이(초 단위)가 설정되어 있으면 해당 시간만큼 대기한다 |
| 6 | SDK | 설문 패키지가 이미 로드되어 있는지 확인한다 |
| 7a | SDK | (이미 로드된 경우) 기존 설문 패키지를 재사용한다 |
| 7b | SDK | (로드되지 않은 경우) 설문 스크립트를 동적으로 삽입하여 로드한다 |
| 8 | SDK | 위젯 컨테이너 div 요소를 body에 추가한다 |
| 9 | SDK | 설문 렌더링 함수를 호출한다. 다음 파라미터를 전달한다: App URL, Environment ID, 연락처 ID, 액션, 설문 데이터, 브랜딩 활성화 여부, 외부 클릭 닫기, 오버레이, 언어 코드, 위젯 배치, 스타일링, Hidden Fields 데이터, reCAPTCHA 설정, 콜백 함수(표시 생성/응답 생성/닫기) |
| 10 | SDK | 설문 실행 중 플래그를 true로 설정한다 |

#### 4.5.5 대안 흐름 (Alternative Flow)

### 4.6 No-Code Action 감지 (FN-07-06)

#### 4.6.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-07-06 |
| 기능명 | No-Code Action 감지 |
| 관련 요구사항 ID | FR-010 (4.14), AC-010-07, AC-010-08, AC-010-09, AC-010-10 |
| 우선순위 | 필수 |
| 기능 설명 | 4가지 유형의 No-Code Action 이벤트(Page URL, Click, Exit Intent, 50% Scroll)를 자동 감지하고, 조건 매칭 시 설문 트리거 흐름을 시작한다 |

#### 4.6.2 선행 조건 (Preconditions)

1. SDK가 성공적으로 초기화되어 있어야 한다 (FN-07-01 완료)
2. 이벤트 리스너가 등록되어 있어야 한다 (초기화 단계 12 완료)
3. 해당 Action Class가 Environment State에 정의되어 있어야 한다

#### 4.6.3 후행 조건 (Postconditions)

1. 조건 매칭된 Action에 연결된 설문의 트리거 흐름이 시작된다
2. 위젯 렌더링 (FN-07-05)이 호출된다

#### 4.6.4 기본 흐름 (Basic Flow) -- Page URL 이벤트

| 단계 | 행위자 | 동작 |
|------|--------|------|
| 1 | SDK | 다음 5가지 이벤트에 대한 리스너를 등록한다: `hashchange`, `popstate`, `pushstate`, `replacestate`, `load` |
| 2 | SDK | SPA 라우팅 변경 감지를 위해 `history.pushState`와 `history.replaceState`를 래핑한다 |
| 3 | 사용자/앱 | 페이지 URL이 변경된다 (네비게이션, SPA 라우팅 등) |
| 4 | SDK | 등록된 이벤트 리스너가 URL 변경을 감지한다 |
| 5 | SDK | 현재 URL을 Action Class에 정의된 URL 필터와 비교한다. 커넥터(or/and)를 사용하여 다중 필터 매칭 여부를 판단한다 |
| 6 | SDK | 매칭 성공 시 해당 Action에 연결된 설문 트리거 흐름을 시작한다 |

#### 4.6.5 대안 흐름 (Alternative Flow)

**AF-06-01: Click 이벤트**

| 단계 | 행위자 | 동작 |
|------|--------|------|
| 1 | SDK | 문서(document) 레벨에 `click` 이벤트 리스너를 등록한다 |
| 2 | 사용자 | 페이지 내 요소를 클릭한다 |
| 3 | SDK | 클릭된 대상 요소의 CSS 선택자 또는 내부 HTML(innerHTML)을 확인한다 |
| 4 | SDK | Action Class에 정의된 CSS 선택자 또는 내부 HTML 패턴과 비교한다 |
| 5 | SDK | 매칭 성공 시 해당 Action에 연결된 설문 트리거 흐름을 시작한다 |

**AF-06-02: Exit Intent 이벤트**

| 단계 | 행위자 | 동작 |
|------|--------|------|
| 1 | SDK | body 레벨에 `mouseleave` 이벤트 리스너를 등록한다 |
| 2 | 사용자 | 마우스를 뷰포트 상단 밖으로 이동한다 |
| 3 | SDK | 마우스 Y좌표가 0 이하인지 확인한다 |
| 4 | SDK | Y좌표 <= 0이면 URL 필터를 추가로 검증한다 |
| 5 | SDK | 모든 조건 매칭 시 해당 Action에 연결된 설문 트리거 흐름을 시작한다 |

**AF-06-03: 50% Scroll 이벤트**

| 단계 | 행위자 | 동작 |
|------|--------|------|
| 1 | SDK | window 레벨에 `scroll` 이벤트 리스너를 등록한다 |
| 2 | 사용자 | 페이지를 스크롤한다 |
| 3 | SDK | 스크롤 비율을 계산한다: `scrollPosition / (documentHeight - viewportHeight)` |
| 4 | SDK | 스크롤 비율 >= 0.5이면 트리거한다 |
| 5 | SDK | 트리거 후 스크롤 위치가 0으로 돌아가기 전까지 재트리거를 차단한다 |
| 6 | SDK | 스크롤 위치가 0으로 돌아온 후에야 다시 트리거가 가능해진다 |

#### 4.6.6 예외 흐름 (Exception Flow)

**EX-06-01: URL 필터 매칭 실패**
- 단계 5 (Page URL) 또는 단계 4 (Exit Intent)에서 URL 필터가 매칭되지 않는 경우
- 트리거를 발생시키지 않고 이벤트를 무시한다

**EX-06-02: 클릭 대상 매칭 실패**
- 단계 4 (Click 이벤트)에서 CSS 선택자와 내부 HTML 모두 매칭되지 않는 경우
- 트리거를 발생시키지 않고 이벤트를 무시한다

#### 4.6.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 조건 | 동작 |
|---------|------|------|
| BR-06-01 | Page URL 필터에 다중 조건이 있고 커넥터가 "or" | 하나 이상의 필터 매칭 시 트리거 |
| BR-06-02 | Page URL 필터에 다중 조건이 있고 커넥터가 "and" | 모든 필터가 매칭되어야 트리거 |
| BR-06-03 | Exit Intent에서 마우스 Y좌표 <= 0 | Exit Intent 트리거 발생 |
| BR-06-04 | Exit Intent에서 마우스 Y좌표 > 0 | Exit Intent 트리거 미발생 |
| BR-06-05 | 50% Scroll 트리거 후 스크롤 위치 > 0 | 재트리거 차단 |
| BR-06-06 | 50% Scroll 트리거 후 스크롤 위치가 0으로 복귀 | 재트리거 가능 |
| BR-06-07 | Click 이벤트에서 CSS 선택자 또는 innerHTML 매칭 | 트리거 발생 |
| BR-06-08 | SPA 라우팅 변경 | history.pushState 래핑을 통해 URL 변경 감지 |

#### 4.6.8 데이터 요구사항

**Action Class 구조**:

| 필드명 | 타입 | 설명 |
|--------|------|------|
| type | "pageUrl" \| "click" \| "exitIntent" \| "fiftyPercentScroll" | Action 유형 |
| urlFilters | URLFilter[] | URL 필터 목록 (pageUrl, exitIntent에서 사용) |
| connector | "or" \| "and" | URL 필터 간 논리 연산자 |
| cssSelector | string | CSS 선택자 (click에서 사용) |
| innerHtml | string | 내부 HTML 패턴 (click에서 사용) |

**이벤트 리스너 등록 대상**:

| Action 유형 | 이벤트 | 등록 대상 | 설명 |
|-------------|--------|-----------|------|
| pageUrl | hashchange, popstate, pushstate, replacestate, load | window | SPA 라우팅 포함 URL 변경 감지 |
| click | click | document | 문서 레벨 클릭 감지 |
| exitIntent | mouseleave | body | 마우스 뷰포트 이탈 감지 |
| fiftyPercentScroll | scroll | window | 스크롤 위치 감지 |

#### 4.6.9 화면/UI 요구사항

해당 없음. No-Code Action 감지는 백그라운드에서 동작하며 UI를 제공하지 않는다.

#### 4.6.10 비기능 요구사항

| 항목 | 요구사항 |
|------|----------|
| SPA 지원 | `history.pushState`와 `history.replaceState`를 래핑하여 SPA 라우팅 변경을 감지한다 |
| 성능 | 이벤트 리스너는 필요한 이벤트에만 등록하여 불필요한 오버헤드를 방지한다 |

### 4.7 GTM 통합 (FN-07-07)

#### 4.7.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-07-07 |
| 기능명 | GTM (Google Tag Manager) 통합 |
| 관련 요구사항 ID | FR-042 (4.15), AC-042-01, AC-042-02 |
| 우선순위 | 필수 |
| 기능 설명 | Google Tag Manager의 Custom HTML 태그를 통해 Inquiry SDK를 삽입하고 초기화한다. 코드 변경 없이 태그 매니저 기반으로 설문을 배포할 수 있다 |

#### 4.7.2 선행 조건 (Preconditions)

1. 호스트 웹 페이지에 GTM 컨테이너 스니펫이 설치되어 있어야 한다
2. GTM 컨테이너에 Inquiry Custom HTML 태그가 설정되어 있어야 한다
3. 유효한 Environment ID와 App URL이 태그에 구성되어 있어야 한다

#### 4.7.3 후행 조건 (Postconditions)

1. Inquiry SDK가 페이지에 로드되고 초기화된다
2. 해당 Environment에 속한 설문만 로드된다
3. 이후 동작은 일반 SDK 초기화와 동일하다 (FN-07-01의 후행 조건과 동일)

#### 4.7.4 기본 흐름 (Basic Flow)

| 단계 | 행위자 | 동작 |
|------|--------|------|
| 1 | 마케팅 담당자 | GTM 콘솔에서 Custom HTML 태그를 생성한다 |
| 2 | 마케팅 담당자 | 태그에 Inquiry SDK 스크립트를 삽입하고, Environment ID와 App URL을 설정한다 |
| 3 | 마케팅 담당자 | 적절한 트리거(예: All Pages)를 설정하고 태그를 게시한다 |
| 4 | GTM | 사용자가 웹 페이지를 방문하면 GTM이 Custom HTML 태그를 실행한다 |
| 5 | SDK | Custom HTML 태그 내의 스크립트가 Inquiry SDK를 로드한다 |
| 6 | SDK | Environment ID와 App URL로 SDK를 초기화한다 (FN-07-01 기본 흐름 수행) |
| 7 | SDK | 해당 Environment의 설문만 로드하고 트리거 조건에 따라 표시한다 |

#### 4.7.5 대안 흐름 (Alternative Flow)

해당 없음. GTM 통합의 SDK 초기화 이후 동작은 FN-07-01과 동일하다.

#### 4.7.6 예외 흐름 (Exception Flow)

**EX-07-01: 잘못된 Environment ID**
- 단계 6에서 유효하지 않은 Environment ID가 설정된 경우
- SDK 초기화가 실패하고, EX-01-02 (서버 통신 실패)와 동일한 에러 처리가 수행된다

**EX-07-02: GTM 태그 실행 지연**
- GTM 태그 실행이 페이지 로드 후 지연되는 경우
- SDK 초기화가 지연될 수 있으나, 초기화 완료 후 정상 동작한다
- 이미 발생한 No-Code Action 이벤트는 감지하지 못할 수 있다

#### 4.7.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 조건 | 동작 |
|---------|------|------|
| BR-07-01 | GTM Custom HTML 태그에 Environment ID가 설정됨 | 해당 Environment의 설문만 로드 |
| BR-07-02 | GTM 통합으로 SDK 초기화됨 | 이후 동작은 일반 SDK 초기화와 동일 |

#### 4.7.8 데이터 요구사항

**GTM Custom HTML 태그에 포함되어야 하는 데이터**:

| 필드명 | 타입 | 필수 여부 | 설명 |
|--------|------|-----------|------|
| SDK Script URL | string | 필수 | Inquiry SDK 스크립트 URL |
| environmentId | string | 필수 | Inquiry Environment의 CUID |
| appUrl | string | 필수 | Inquiry 서버 URL |

#### 4.7.9 화면/UI 요구사항

해당 없음. GTM 태그 설정은 Google Tag Manager 인터페이스에서 수행된다.

#### 4.7.10 비기능 요구사항

### 4.8 에러 상태 관리 (FN-07-08)

#### 4.8.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-07-08 |
| 기능명 | 에러 상태 관리 |
| 관련 요구사항 ID | FR-010 (4.16), AC-010-12 |
| 우선순위 | 필수 |
| 기능 설명 | SDK에서 에러가 발생하면 에러 상태로 전환하고, 10분(600,000ms) 동안 SDK 재초기화를 차단한다. 디버그 모드에서는 에러 상태를 무시하고 즉시 재초기화할 수 있다 |

#### 4.8.2 선행 조건 (Preconditions)

1. SDK에서 에러가 발생해야 한다 (서버 통신 실패, 초기화 실패 등)

#### 4.8.3 후행 조건 (Postconditions)

1. SDK 상태가 error로 변경된다
2. 에러 만료 시각이 현재 시각 + 600,000ms로 설정된다
3. 정리 처리(tearDown)가 수행된다 (FN-07-09)
4. 에러 만료 전까지 SDK 재초기화가 차단된다

#### 4.8.4 기본 흐름 (Basic Flow)

| 단계 | 행위자 | 동작 |
|------|--------|------|
| 1 | SDK | 에러를 감지한다 (서버 통신 실패, 데이터 파싱 오류 등) |
| 2 | SDK | SDK 상태를 error로 전환한다 |
| 3 | SDK | 에러 만료 시각을 현재 시각 + 600,000ms로 설정한다 |
| 4 | SDK | 설정을 업데이트하고 LocalStorage에 저장한다 |
| 5 | SDK | 정리 처리(tearDown)를 수행한다 (FN-07-09) |
| 6 | SDK | 에러 만료 시각까지 SDK 재초기화 요청을 차단한다 |
| 7 | SDK | 에러 만료 시각이 경과하면 자동으로 재시도가 가능해진다 |

#### 4.8.5 대안 흐름 (Alternative Flow)

**AF-08-01: 디버그 모드에서 에러 상태**
- 디버그 모드가 활성화된 상태에서 에러 상태에 진입하는 경우
- 에러 상태를 무시하고 즉시 재초기화를 허용한다

#### 4.8.6 예외 흐름 (Exception Flow)

해당 없음.

## 5. 데이터 모델

### 5.1 주요 엔티티 정의

| 엔티티 | 설명 | 저장 위치 |
|--------|------|-----------|
| SDKConfig | SDK 전체 설정 상태. Singleton 패턴으로 관리 | LocalStorage (`formbricks-js`) |
| EnvironmentState | 환경 관련 데이터 (설문, Action Class, 프로젝트 설정) | SDKConfig 내 포함 |
| UserState | 사용자 관련 데이터 (ID, 세그먼트, 표시 이력, 응답) | SDKConfig 내 포함 |
| Survey | 설문 데이터 (질문, 로직, 설정) | EnvironmentState 내 포함 |
| ActionClass | No-Code Action 정의 (유형, 필터, 선택자) | EnvironmentState 내 포함 |
| ProjectConfig | 프로젝트 설정 (배치, 오버레이, 외부 클릭, 스타일링) | EnvironmentState 내 포함 |
| SurveyOverride | 설문별 프로젝트 설정 오버라이드 | Survey 내 포함 |

### 5.2 엔티티 간 관계

```
SDKConfig (1)
├── EnvironmentState (1)
│   ├── Survey (N)
│   │   └── SurveyOverride (0..1)
│   ├── ActionClass (N)
│   └── ProjectConfig (1)
└── UserState (1)
```

- SDKConfig는 하나의 EnvironmentState와 하나의 UserState를 포함한다
- EnvironmentState는 다수의 Survey와 ActionClass를 포함하며, 하나의 ProjectConfig를 포함한다
- 각 Survey는 선택적으로 하나의 SurveyOverride를 가질 수 있다

### 5.3 데이터 흐름

```
[Inquiry Server]
       |
       | (1) 초기화 시 환경/사용자 상태 조회
       v
[SDK Runtime] <----> [LocalStorage: formbricks-js]
       |                    (2) 설정 캐시/로드
       |
       | (3) Action 감지 시 설문 트리거
       v
[위젯 렌더링]
       |
       | (4) 표시/응답/닫기 콜백
       v
[Inquiry Server]
```

1. SDK 초기화 시 서버에서 Environment State와 User State를 조회한다
2. 조회된 데이터를 LocalStorage에 캐시하고, 이후 초기화 시 캐시에서 로드한다
3. No-Code Action이 감지되면 조건 매칭 후 설문 트리거 및 위젯 렌더링을 수행한다
4. 설문 표시/응답 생성/닫기 이벤트는 콜백을 통해 서버에 전달된다
