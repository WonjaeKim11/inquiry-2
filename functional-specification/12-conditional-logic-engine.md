## 2. 개요

### 2.1 목적

본 문서는 설문 응답 흐름을 동적으로 제어하는 조건부 로직 엔진의 상세 기능 명세를 정의한다. Block 단위의 로직 정의, 조건 평가, 액션 수행의 전체 동작을 구체적으로 기술하여 개발 및 테스트의 기준 문서로 활용한다.

### 2.2 범위

**포함 범위 (In-scope)**
- 31개 조건 비교 연산자의 평가 로직
- 7개 변수 계산 연산자 (텍스트 2개 + 숫자 5개)
- 3가지 액션 타입: calculate, requireAnswer, jumpToBlock
- 중첩 AND/OR 조건 그룹 (재귀적 구조)
- Block 레벨 로직 정의 및 logicFallback 처리
- element, variable, hiddenField 피연산자 타입 지원
- 로직 평가 엔진 (조건 평가 + 액션 수행)
- 로직 유틸리티 기능 (복제, 추가, 삭제, 업데이트, 토글)
- 순환 로직(Cyclic Logic) 검증

**제외 범위 (Out-of-scope)**
- Question 레벨 로직 (deprecated, v1 API 호환용으로만 유지)
- 외부 API 호출 기반 조건부 로직
- 타이머/시간 기반 자동 분기

### 2.3 대상 사용자

| 역할 | 관심 사항 |
|------|----------|
| 설문 작성자 | 응답에 따른 동적 분기 설정, 변수 계산을 통한 스코어링 |
| 데이터 분석가 | 변수 계산을 통한 스코어링 및 가중치 부여 |
| 프론트엔드 개발자 | 로직 평가 엔진 구현 및 유지보수 |
| API 사용자 | v1에서 v2로의 마이그레이션 (question -> element 타입 변경) |

### 2.4 용어 정의

| 용어 | 정의 |
|------|------|
| Block | 설문을 구성하는 최소 단위 컨테이너. 1개 이상의 Element를 포함하며 로직 아이템을 가질 수 있다 |
| Element | Block 내에 배치되는 개별 설문 질문 요소 |
| Logic Item | 조건 그룹(conditions)과 액션 배열(actions)로 구성된 하나의 로직 규칙 |
| Condition Group | AND 또는 OR connector로 연결된 조건들의 집합. 재귀적 중첩 가능 |
| Single Condition | 좌측 피연산자, 연산자, 우측 피연산자로 구성된 단일 비교 조건 |
| Connector | 조건 그룹 내 조건들의 연결 방식. "and"(모든 조건 충족) 또는 "or"(하나 이상 충족) |
| Dynamic Field | element, variable, hiddenField 중 하나의 동적 참조 필드 |
| Static Value | 고정 값 (문자열, 숫자, 문자열 배열) |
| logicFallback | Block의 모든 로직 조건이 불일치할 때 이동할 대상 Block ID |
| Hidden Field | 설문 URL 파라미터 등으로 전달되는 숨겨진 필드 값 |
| Variable | 설문 내에서 계산 및 참조에 사용되는 변수 (텍스트 또는 숫자 타입) |
| CUID / CUID2 | Collision-resistant Unique Identifier. Block ID는 CUID, Variable ID는 CUID2 형식 |
| Cyclic Logic | 블록 간 이동 경로가 순환 구조를 형성하는 로직 오류 |

### 4.1 조건 비교 연산자 평가

#### 4.1.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-012-01 |
| 기능명 | 조건 비교 연산자 평가 |
| 관련 요구사항 ID | FR-018 (4.1) |
| 우선순위 | 상 |
| 기능 설명 | 31개의 조건 비교 연산자를 사용하여 좌측 피연산자와 우측 피연산자 간의 비교를 수행하고 boolean 결과를 반환한다 |

#### 4.1.2 선행 조건

- 좌측 피연산자 값이 추출 완료되어야 한다
- 우측 피연산자가 필요한 연산자의 경우 우측 피연산자 값이 제공되어야 한다

#### 4.1.3 후행 조건

- 비교 결과(true/false)가 반환된다
- 에러 발생 시 false가 반환된다

#### 4.1.4 기본 흐름

1. 시스템이 연산자 종류를 식별한다
2. 우측 피연산자 필요 여부를 확인한다
3. 좌측 피연산자 값과 우측 피연산자 값(필요 시)을 연산자에 전달한다
4. 연산자별 비교 로직을 수행한다
5. boolean 결과를 반환한다

#### 4.1.5 대안 흐름

- 없음

#### 4.1.6 예외 흐름

- **EX-01-01**: 비교 수행 중 에러가 발생하면 false를 반환한다 (NFR-012-01)

### 4.2 변수 계산 연산자 수행

#### 4.2.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-012-02 |
| 기능명 | 변수 계산 연산자 수행 |
| 관련 요구사항 ID | FR-018 (4.2) |
| 우선순위 | 상 |
| 기능 설명 | 7개의 변수 계산 연산자를 사용하여 텍스트 또는 숫자 변수의 값을 계산하고 업데이트한다 |

#### 4.2.2 선행 조건

- 대상 변수가 설문 내에 정의되어 있어야 한다
- 대상 변수의 타입(텍스트/숫자)이 결정되어 있어야 한다
- 계산에 사용할 값(static 또는 dynamic)이 제공되어야 한다

#### 4.2.3 후행 조건

- 변수 값이 계산 결과로 업데이트된다
- 0으로 나누기 시도 시 undefined가 반환된다

#### 4.2.4 기본 흐름

1. 시스템이 대상 변수의 타입(텍스트/숫자)을 확인한다
2. 변수 타입에 맞는 계산 연산자를 적용한다
3. 값 소스(static 또는 dynamic field)에서 계산에 사용할 값을 추출한다
4. 연산자에 따라 계산을 수행한다

- 좌측 피연산자 값이 연산자 비교에 사용 가능한 형태로 추출된다

2. 타입에 따라 값 추출 소스를 결정한다
   - element: 설문 질문 요소의 응답값 참조
   - variable: 설문 변수값 참조

4. 변환된 값을 반환한다

**우측 피연산자 값 추출**

1. 우측 피연산자 타입을 확인한다 (static / dynamic)
2. static인 경우 고정 값(문자열, 숫자, 문자열 배열)을 그대로 사용한다
3. dynamic인 경우 element, variable, hiddenField 중 해당하는 소스에서 값을 추출한다

### 4.4 조건 그룹 평가 (중첩 AND/OR)

#### 4.4.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-012-04 |
| 기능명 | 조건 그룹 평가 (중첩 AND/OR) |
| 관련 요구사항 ID | FR-018 (4.5) |
| 우선순위 | 상 |
| 기능 설명 | 재귀적 구조의 조건 그룹을 AND/OR connector에 따라 평가하고 최종 boolean 결과를 반환한다 |

#### 4.4.2 선행 조건

- 조건 그룹이 올바른 재귀 구조로 정의되어 있어야 한다
- 각 단일 조건의 피연산자 값이 추출 가능한 상태여야 한다

#### 4.4.3 후행 조건

- 조건 그룹 전체의 평가 결과(true/false)가 반환된다

#### 4.4.4 기본 흐름

1. 시스템이 조건 그룹의 connector 타입(AND/OR)을 확인한다
2. 그룹 내의 각 항목을 순회한다
3. 항목이 단일 조건(Single Condition)인 경우:
   a. 좌측 피연산자 값을 추출한다 (FN-012-03)
   b. 우측 피연산자 값을 추출한다 (해당 시) (FN-012-03)
   c. 연산자로 비교를 수행한다 (FN-012-01)
   d. 비교 결과(boolean)를 반환한다
4. 항목이 하위 조건 그룹(Condition Group)인 경우:
   a. 재귀적으로 하위 그룹을 평가한다 (본 흐름 반복)
   b. 하위 그룹의 평가 결과(boolean)를 반환한다
5. connector에 따라 전체 결과를 결정한다:
   - AND: 모든 항목이 true일 때만 전체 true
   - OR: 하나라도 true이면 전체 true

#### 4.4.5 대안 흐름

#### 4.4.6 예외 흐름

- **EX-04-02**: 재귀 평가 중 에러 발생 시 false를 반환한다 (NFR-012-01)

#### 4.4.7 비즈니스 규칙

| 규칙 ID | 규칙 |
|---------|------|
| BR-04-01 | connector가 "and"인 경우: 그룹 내 모든 조건/하위 그룹이 true여야 전체 true |
| BR-04-02 | connector가 "or"인 경우: 그룹 내 하나라도 true이면 전체 true |
| BR-04-03 | 조건 그룹은 무한 중첩이 가능하다 (재귀적 구조) |
| BR-04-04 | 조건 그룹 내부에 단일 조건 또는 하위 조건 그룹이 혼합 가능하다 |

### 4.6 Block 레벨 로직 정의

#### 4.6.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-012-06 |
| 기능명 | Block 레벨 로직 정의 |
| 관련 요구사항 ID | FR-018 (4.7) |
| 우선순위 | 상 |
| 기능 설명 | 각 Block에 여러 로직 아이템을 정의하고, 로직 조건 불일치 시의 fallback 동작을 설정한다 |

#### 4.6.2 선행 조건

- Block이 생성되어 있어야 한다
- Block 내에 최소 1개의 Element가 포함되어 있어야 한다
- Block 이름이 최소 1자 이상이어야 한다

#### 4.6.3 후행 조건

- Block에 로직 아이템이 정의된다
- logicFallback이 설정된다 (선택)
- 설문 저장 시 유효성 검증이 수행된다

#### 4.6.4 기본 흐름

1. 설문 작성자가 Block을 선택한다
2. 시스템이 해당 Block의 로직 설정 영역을 표시한다
3. 설문 작성자가 로직 아이템을 추가한다
4. 로직 아이템에 조건 그룹(conditions)을 정의한다
5. 로직 아이템에 액션 배열(actions)을 정의한다
6. (선택) logicFallback으로 모든 조건 불일치 시 이동할 Block ID를 설정한다
7. 시스템이 설문 저장 시 유효성 검증을 수행한다

#### 4.6.5 대안 흐름

- **AF-06-01**: logicFallback을 설정하지 않은 경우 모든 로직 조건 불일치 시 기본 순서(다음 Block)로 진행한다

#### 4.6.6 예외 흐름

- **EX-06-01**: Block 내 Element가 0개인 경우 유효성 검증 실패
- **EX-06-02**: Block 이름이 빈 문자열인 경우 유효성 검증 실패
- **EX-06-03**: Element ID가 Block 내에서 중복되는 경우 유효성 검증 실패 (NFR-012-03)

#### 4.6.7 비즈니스 규칙

| 규칙 ID | 규칙 |
|---------|------|
| BR-06-01 | 각 Block은 0개 이상의 로직 아이템을 가질 수 있다 |
| BR-06-02 | 각 로직 아이템은 1개의 조건 그룹(conditions)과 1개 이상의 액션(actions)으로 구성된다 |
| BR-06-03 | Block ID는 CUID 형식이어야 한다 |
| BR-06-04 | Block 이름은 필수이며 최소 1자 이상이어야 한다 |
| BR-06-05 | Block 내에 최소 1개의 Element가 포함되어야 한다 |
| BR-06-06 | logicFallback은 유효한 Block ID를 참조해야 한다 |
| BR-06-07 | 로직 아이템은 정의된 순서대로 평가된다. 첫 번째로 조건이 일치하는 로직 아이템의 액션이 수행된다 |

#### 4.6.8 데이터 요구사항

**Block 로직 데이터 구조**

| 필드명 | 타입 | 필수 | 유효성 검증 규칙 |
|--------|------|------|----------------|
| blockId | string (CUID) | Y | CUID 형식 |
| blockName | string | Y | 최소 1자 |
| elements | array | Y | 최소 1개 이상의 Element. Element ID는 Block 내 유일 |
| logicItems | array | N | 로직 아이템 배열 |
| logicFallback | string (CUID) / null | N | 이동할 Block ID 또는 null |

- logicFallback 설정을 위한 Block 선택 UI가 제공되어야 한다

#### 4.7.2 선행 조건

- 로직 아이템의 조건 평가가 true로 완료되어야 한다
- 액션에 필요한 참조 대상(변수, Element, Block)이 존재해야 한다

#### 4.7.3 후행 조건

4. 변수 계산 연산자를 수행한다 (FN-012-02)
5. 계산 결과를 calculations 객체에 저장한다

2. 해당 Element ID를 requiredElementIds 목록에 추가한다

**JumpToBlock 액션**
1. 시스템이 대상 Block ID를 확인한다
2. jumpTarget에 해당 Block ID를 설정한다

#### 4.8.1 기능 개요

| 항목 | 내용 |

| 기능 ID | FN-012-08 |
| 기능명 | 로직 평가 엔진 |

#### 4.8.2 선행 조건

- 설문 데이터가 로드되어 있어야 한다
- 현재 Block의 응답 데이터가 수집되어 있어야 한다
- 변수 데이터가 초기화되어 있어야 한다
- 선택 언어가 결정되어 있어야 한다

#### 4.8.3 후행 조건

- 조건 그룹에 대한 boolean 결과가 반환된다
- 에러 발생 시 false가 반환된다

#### 4.8.4 기본 흐름

1. 시스템이 현재 Block의 로직 아이템 배열을 순서대로 순회한다
2. 각 로직 아이템의 조건 그룹을 평가한다 (FN-012-04)
   a. 조건 그룹 내 각 조건에 대해 좌측 피연산자 값을 추출한다 (FN-012-03)
   b. 필요한 경우 우측 피연산자 값을 추출한다 (FN-012-03)
   c. 연산자로 비교를 수행한다 (FN-012-01)
   d. connector(AND/OR)에 따라 결과를 조합한다
3. 조건이 true로 평가된 첫 번째 로직 아이템의 액션을 수행한다 (FN-012-09)
4. 모든 로직 아이템의 조건이 false인 경우 logicFallback을 확인한다
   a. logicFallback이 설정되어 있으면 해당 Block으로 이동한다
   b. logicFallback이 없으면 다음 Block으로 진행한다

#### 4.8.5 대안 흐름

- **AF-08-01**: 현재 Block에 로직 아이템이 없는 경우 다음 Block으로 진행한다

#### 4.8.6 예외 흐름

| BR-08-04 | logicFallback이 없고 모든 로직 조건 불일치 시 기본 순서(다음 Block)로 진행한다 |

#### 4.10.1 기능 개요

| 항목 | 내용 |

| 기능명 | 로직 유틸리티 기능 |

#### 4.10.2 선행 조건

- 설문 에디터가 열려 있어야 한다
- 대상 Block이 선택되어 있어야 한다

#### 4.10.3 후행 조건

- 편집 결과가 설문 데이터에 반영된다
- UI가 변경된 로직 구조를 반영하여 갱신된다

#### 4.10.4 기본 흐름

각 유틸리티 기능별 기본 흐름:

**로직 아이템 복제**
1. 사용자가 로직 아이템의 복제 버튼을 클릭한다
2. 시스템이 해당 로직 아이템의 전체 구조(조건 그룹 + 액션)를 깊은 복사(deep copy)한다
3. 복제된 아이템에 새 ID를 생성한다
4. 원본 아이템 바로 아래에 복제된 아이템을 삽입한다

**조건 추가**
1. 사용자가 특정 조건 그룹 내에서 조건 추가를 요청한다
2. 시스템이 기본 구조의 새 단일 조건을 생성한다
3. 해당 조건 그룹에 새 조건을 추가한다

**그룹 connector 토글**
1. 사용자가 조건 그룹의 connector를 클릭한다
2. 시스템이 AND와 OR 사이를 토글한다 (AND -> OR, OR -> AND)

**조건 삭제**
1. 사용자가 특정 조건의 삭제 버튼을 클릭한다
2. 시스템이 해당 조건을 제거한다
3. 조건 삭제 후 부모 그룹이 비어있게 되면 빈 그룹을 정리(제거)한다

**조건 복제**
1. 사용자가 특정 조건의 복제 버튼을 클릭한다
2. 시스템이 해당 단일 조건을 깊은 복사한다
3. 원본 조건 바로 아래에 복제된 조건을 삽입한다

**그룹 생성**
1. 사용자가 단일 조건에 대해 그룹 생성을 요청한다
2. 시스템이 해당 단일 조건을 새 조건 그룹으로 감싼다
3. 새 그룹의 기본 connector는 "and"로 설정한다

**조건 업데이트**
1. 사용자가 조건의 좌측 피연산자, 연산자, 우측 피연산자 중 하나를 변경한다
2. 시스템이 해당 조건을 업데이트한다
3. 연산자 변경 시 우측 피연산자의 필요 여부가 변경되면 우측 피연산자를 초기화한다

**액션 본문 갱신**
1. 사용자가 액션의 objective(타입)를 변경한다
2. 시스템이 변경된 objective에 맞는 기본 본문(default body)을 생성한다
3. 기존 본문을 새 기본 본문으로 교체한다

|---------|------|
| BR-10-01 | 로직 아이템 복제 시 모든 내부 ID는 새로 생성된다 |
| BR-10-02 | 조건 삭제 후 부모 그룹이 비어있으면 해당 그룹도 자동 제거된다 |
| BR-10-03 | 그룹 생성 시 기본 connector는 "and"이다 |
| BR-10-04 | 액션 objective 변경 시 기본 본문이 자동 생성된다 |

#### 4.10.8 데이터 요구사항

해당 기능은 기존 로직 아이템, 조건 그룹, 액션 데이터 구조를 수정한다. 새로 생성되는 데이터의 형식은 4.4, 4.5, 4.6, 4.7 절의 데이터 구조를 따른다.

#### 4.10.9 화면/UI 요구사항

- 각 로직 아이템, 조건, 그룹에 복제/삭제/추가 버튼을 제공한다
- connector(AND/OR) 클릭 시 즉시 토글되어야 한다
- 조건 삭제 후 빈 그룹이 자동 정리되는 과정이 시각적으로 반영되어야 한다
- 액션 objective 변경 시 하위 입력 필드가 동적으로 갱신되어야 한다

#### 4.10.10 비기능 요구사항

해당 없음

### 4.11 순환 로직 검증

#### 4.11.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-012-11 |
| 기능명 | 순환 로직 검증 |
| 관련 요구사항 ID | NFR-012-02 |
| 우선순위 | 중 |
| 기능 설명 | 설문 내 Block 간 jumpToBlock 및 logicFallback 참조가 순환 구조를 형성하는지 검출한다 |

#### 4.11.2 선행 조건

- 설문의 전체 Block 구조와 로직 아이템이 정의되어 있어야 한다

#### 4.11.3 후행 조건

- 순환 로직이 감지되면 경고/에러를 반환한다
- 순환 로직이 없으면 검증 통과를 반환한다

#### 4.11.4 기본 흐름

1. 시스템이 설문의 모든 Block을 순회한다
2. 각 Block의 로직 아이템에서 jumpToBlock 액션의 대상 Block ID를 수집한다
3. 각 Block의 logicFallback 대상 Block ID를 수집한다
4. Block 간 이동 관계를 방향 그래프로 구성한다
5. 그래프에서 순환(cycle)이 존재하는지 검사한다 (DFS 또는 유사 알고리즘)
6. 순환이 발견되면 해당 경로 정보와 함께 에러를 반환한다
7. 순환이 없으면 검증 통과를 반환한다

#### 4.11.5 대안 흐름

- 없음

#### 4.11.6 예외 흐름

- **EX-11-01**: 순환 로직이 감지되면 설문 저장을 차단하고 순환 경로를 표시한다

#### 4.11.7 비즈니스 규칙

| 규칙 ID | 규칙 |
|---------|------|
| BR-11-01 | jumpToBlock과 logicFallback 모두 순환 검증 대상이다 |
| BR-11-02 | 순환 로직이 감지되면 설문을 저장할 수 없다 |

#### 4.11.8 데이터 요구사항

**입력 데이터**

| 필드명 | 타입 | 필수 | 유효성 검증 규칙 |
|--------|------|------|----------------|
| blocks | array | Y | 전체 Block 배열 (각 Block의 로직 아이템 및 logicFallback 포함) |

**출력 데이터**

| 필드명 | 타입 | 설명 |
|--------|------|------|
| hasCycle | boolean | 순환 존재 여부 |
| cyclePath | string[] / null | 순환이 발견된 경우 순환 경로의 Block ID 배열 |

#### 4.11.9 화면/UI 요구사항

- 순환 로직 감지 시 에디터에서 관련 Block 및 로직 아이템을 시각적으로 강조 표시한다
- 순환 경로 정보를 사용자가 이해할 수 있는 형태로 표시한다

#### 4.11.10 비기능 요구사항

- 설문 저장 시 자동으로 순환 로직 검증이 수행되어야 한다 (NFR-012-02)

## 5. 데이터 모델

### 5.1 주요 엔티티 정의

**ConditionGroup (조건 그룹)**

| 필드명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| connector | "and" / "or" | Y | 조건 연결 방식 |
| conditions | (SingleCondition / ConditionGroup)[] | Y | 조건 또는 하위 그룹 배열 |

**SingleCondition (단일 조건)**

| 필드명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| id | string | Y | 조건 고유 식별자 |
| leftOperand | DynamicField | Y | 좌측 피연산자 |
| operator | ConditionOperator (enum, 31개) | Y | 비교 연산자 |
| rightOperand | RightOperand / null | 조건부 | 우측 피연산자 |

**DynamicField (동적 필드 - 좌측 피연산자)**

| 필드명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| type | "element" / "variable" / "hiddenField" / "question"(deprecated) | Y | 필드 타입 |
| id | string | Y | 참조 대상 ID |
| meta | object / null | N | 추가 메타 정보 (matrix row 등) |

**RightOperand (우측 피연산자)**

| 필드명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| type | "static" / "dynamic" | Y | 피연산자 유형 |
| value | string / number / string[] | 조건부 | static인 경우 필수. 빈 문자열 불가 |
| fieldType | "element" / "variable" / "hiddenField" | 조건부 | dynamic인 경우 필수 |
| fieldId | string | 조건부 | dynamic인 경우 필수 |

**LogicItem (로직 아이템)**

| 필드명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| id | string | Y | 로직 아이템 고유 식별자 |
| conditions | ConditionGroup | Y | 조건 그룹 |
| actions | Action[] | Y | 액션 배열 (1개 이상) |

**Action (액션)**

| 필드명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| objective | "calculate" / "requireAnswer" / "jumpToBlock" | Y | 액션 타입 |
| variableId | string (CUID2) | 조건부 | calculate 시 필수. 대상 변수 ID |
| operator | CalculateOperator (enum, 7개) | 조건부 | calculate 시 필수 |
| value | object | 조건부 | calculate 시 필수. static/dynamic 값 |
| elementId | string | 조건부 | requireAnswer 시 필수 |
| blockId | string (CUID) | 조건부 | jumpToBlock 시 필수 |

**Block (블록, 로직 관련 필드)**

| 필드명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| id | string (CUID) | Y | Block 고유 식별자 |
| name | string | Y | Block 이름 (최소 1자) |
| elements | Element[] | Y | 요소 배열 (최소 1개) |
| logicItems | LogicItem[] | N | 로직 아이템 배열 |
| logicFallback | string (CUID) / null | N | 모든 로직 불일치 시 이동할 Block ID |

### 5.2 엔티티 간 관계

```
Block (1) ──── (0..*) LogicItem
  │                      │
  │                      ├── (1) ConditionGroup
  │                      │        │
  │                      │        └── (1..*) SingleCondition / ConditionGroup (재귀)
  │                      │                    │
  │                      │                    ├── (1) DynamicField (좌측 피연산자)
  │                      │                    └── (0..1) RightOperand (우측 피연산자)
  │                      │
  │                      └── (1..*) Action
  │                                  │
  │                                  ├── Calculate → Variable
  │                                  ├── RequireAnswer → Element
  │                                  └── JumpToBlock → Block
  │
  ├── (1..*) Element
  └── (0..1) logicFallback → Block
```

### 5.3 데이터 흐름

```
[설문 응답 시작]
       │
       v
[현재 Block 진입]
       │
       v
[Block 내 로직 아이템 존재?] ──No──> [다음 Block으로 이동]
       │
      Yes
       │
       v
[로직 아이템 순서대로 조건 평가]
       │
       ├── 조건 true ──> [액션 배열 수행]
       │                      │
       │                      ├── calculate → [변수 값 업데이트]
       │                      ├── requireAnswer → [필수 응답 설정]
       │                      └── jumpToBlock → [대상 Block으로 이동]
       │
       └── 모든 조건 false ──> [logicFallback 존재?]
                                      │
                                      ├── Yes → [fallback Block으로 이동]
                                      └── No  → [다음 Block으로 이동]
```

### 8.1 기술적 제약사항

| 항목 | 제약 |

| Block 최소 요소 수 | 1개 |
