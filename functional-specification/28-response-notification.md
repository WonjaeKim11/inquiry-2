## 2. 개요

### 2.1 목적

본 문서는 FSD-028 요구사항 명세서(FR-046)를 기반으로, 설문 응답 완료 시 관련 사용자에게 이메일 알림을 발송하는 기능의 상세 설계를 정의한다. 알림 설정 데이터 모델, 알림 설정 UI, 알림 토글 컴포넌트, 설문 생성 시 자동 구독, 알림 이메일 발송, 구독 해제 등 전체 알림 라이프사이클을 다룬다.

### 2.2 범위

**포함 범위 (In-scope)**:
- 설문 응답 완료(responseFinished) 시 이메일 알림 전송
- 설문별 알림 토글 (on/off) 관리
- Organization 단위 자동 구독 설정
- 설문 생성 시 생성자 자동 구독 처리
- 이메일 내 구독 해제 링크를 통한 자동 해제
- Production 환경 설문만 알림 대상 필터링

**제외 범위 (Out-of-scope)**:
- 인앱(in-app) 알림
- 응답 생성(responseCreated) 트리거 알림
- Slack/Discord 등 외부 채널 알림 (Webhook으로 대체, FSD-023 참조)
- 알림 빈도 제한 (매 응답마다 1회 발송)
- 알림 이메일 내용 커스터마이징

### 2.3 대상 사용자

| 역할 | 설명 | 알림 설정 접근 |
|------|------|---------------|
| Organization Owner | 모든 프로젝트의 설문 알림 관리 | 가능 (전체 프로젝트) |
| Organization Manager | 모든 프로젝트의 설문 알림 관리 | 가능 (전체 프로젝트) |
| Team Member (member) | 팀을 통해 접근 가능한 프로젝트의 설문 알림 관리 | 가능 (팀 기반 프로젝트만) |
| Billing Role | 알림 설정 접근 불가 | 불가 |
| Survey Creator | 설문 생성 시 자동 구독 대상 (Organization 정책에 따라) | 해당 역할의 접근 권한에 따름 |

### 2.4 용어 정의

| 용어 | 정의 |
|------|------|
| **알림 설정 (Notification Settings)** | User 모델의 JSON 필드에 저장되는 사용자별 알림 구성. alert 레코드와 자동 구독 해제 Organization 목록으로 구성된다. |
| **alert 레코드** | 설문 ID를 키(key)로, 알림 활성화 여부(boolean)를 값(value)으로 가지는 Record 구조. 예: `{ "survey_id_1": true, "survey_id_2": false }` |
| **자동 구독 해제 목록 (Unsubscribed Organizations)** | 사용자가 자동 구독을 해제한 Organization의 ID 배열. 이 목록에 포함된 Organization의 새 설문에 대해서는 자동 구독되지 않는다. |
| **responseFinished** | 설문 응답이 완료(finished = true)된 이벤트. 알림 이메일 발송의 유일한 트리거이다. |
| **Pipeline** | 설문 이벤트 발생 시 Webhook, 이메일, 통합 처리를 수행하는 내부 파이프라인 시스템 (FSD-022 참조). |
| **CRON_SECRET** | Pipeline API 내부 인증에 사용되는 환경변수 기반 비밀 키. |
| **Production 환경** | 프로젝트 내 실제 운영 데이터를 다루는 환경. Development 환경과 분리되며, 알림은 Production 환경에서만 발송된다. |
| **locale** | 사용자의 언어/지역 설정. 이메일 발송 시 해당 locale에 맞는 언어로 렌더링된다. |

## 3. 시스템 개요

### 3.1 시스템 구성도

```
[설문 응답자] --응답 완료--> [Client API]
                                |
                                v
                     [Pipeline API] <-- CRON_SECRET 인증
                                |
                 +------+-------+--------+
                 |      |       |        |
                 v      v       v        v
            [Webhook] [통합] [알림 이메일] [Follow-Up]
                                |
                                v
                    [알림 수신자 조회]
                    - Organization 멤버 확인
                    - 역할 기반 접근 권한 확인
                    - 설문별 알림 활성화 확인
                                |
                                v
                    [이메일 병렬 발송]
                    - 수신자별 locale 적용
                    - 응답 수 정보 포함
                    - 구독 해제 링크 포함

[사용자] --알림 설정 변경--> [알림 설정 UI]
                                |
                                v
                    [User 모델 JSON 필드 업데이트]
                    - alert 레코드 갱신
                    - 자동 구독 해제 목록 갱신

[설문 생성자] --설문 생성--> [자동 구독 처리]
                                |
                                v
                    [자동 구독 조건 확인]
                    - 해당 Organization 자동 구독 해제 여부 확인
                    - 조건 충족 시 alert 레코드에 새 설문 ID 추가
```

| FN-028-03 | 알림 스위치 컴포넌트 | FR-046-03 | 높음 |
| FN-028-04 | 설문 생성 시 자동 구독 | FR-046-04 | 높음 |

### 3.3 기능 간 관계도

```
FN-028-01 알림 설정 데이터 모델
    ^           ^           ^
    |           |           |
FN-028-02   FN-028-04   FN-028-05
알림 설정     자동 구독     알림 이메일
페이지                     발송
    ^                       |
    |                       v
FN-028-03               FN-028-06
알림 스위치              구독 해제
컴포넌트              (이메일 링크 → 설정 페이지)
```

- **FN-028-01 (데이터 모델)**: 모든 기능의 기반. 알림 설정 읽기/쓰기의 대상이 되는 데이터 구조를 정의한다.
- **FN-028-02 (설정 페이지)**: FN-028-03(스위치 컴포넌트)을 포함하며, FN-028-01(데이터 모델)을 조회/갱신한다.
- **FN-028-04 (자동 구독)**: 설문 생성 이벤트에서 FN-028-01(데이터 모델)에 새 항목을 추가한다.
- **FN-028-05 (알림 이메일 발송)**: Pipeline에서 FN-028-01(데이터 모델)을 조회하여 수신자를 결정한다.
- **FN-028-06 (구독 해제)**: 이메일 링크를 통해 FN-028-02(설정 페이지)로 이동하고, FN-028-01(데이터 모델)을 갱신한다.

## 4. 상세 기능 명세

### 4.1 알림 설정 데이터 모델 (FN-028-01)

#### 4.1.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-028-01 |
| 기능명 | 알림 설정 데이터 모델 |
| 관련 요구사항 ID | FR-046-01 |
| 우선순위 | 높음 |
| 기능 설명 | 사용자의 알림 설정을 저장하는 데이터 구조를 정의한다. User 모델 내 JSON 필드에 저장되며, 설문별 알림 활성화 레코드와 자동 구독 해제 Organization 목록으로 구성된다. |

#### 4.1.2 선행 조건

- User 모델이 존재하고, JSON 타입의 알림 설정 필드를 지원해야 한다.
- Organization, Project, Survey 모델이 정의되어 있어야 한다.

#### 4.1.3 후행 조건

- 알림 설정 데이터가 User 레코드의 JSON 필드에 영속적으로 저장된다.
- 모든 알림 관련 기능(설정 UI, 자동 구독, 이메일 발송, 구독 해제)이 이 데이터 구조를 참조한다.

#### 4.1.4 기본 흐름

해당 사항 없음 (데이터 모델 정의).

#### 4.1.5 대안 흐름

해당 사항 없음.

#### 4.1.6 예외 흐름

해당 사항 없음.

#### 4.1.7 비즈니스 규칙

| 규칙 ID | 규칙 | 조건 | 결과 |
|---------|------|------|------|
| BR-028-01 | alert 기본값 | 기존에 알림 설정이 없는 설문 | 비활성(false)으로 초기화 |
| BR-028-02 | 레거시 호환 | 레거시 형식의 알림 설정이 존재하는 경우 | 현재 형식으로 변환하여 처리 |
| BR-028-03 | 자동 구독 해제 목록 기본값 | 사용자의 자동 구독 해제 목록이 미설정인 경우 | 빈 배열([])로 초기화 |

#### 4.1.8 데이터 요구사항

**알림 설정 JSON 구조**:

```typescript
interface NotificationSettings {
  alert: Record<string, boolean>;  // 설문 ID → 알림 활성화 여부
  unsubscribedOrganizationIds?: string[];  // 자동 구독을 해제한 Organization ID 배열
}
```

**입력 데이터**:

| 필드명 | 타입 | 필수 여부 | 유효성 검증 규칙 |
|--------|------|----------|----------------|
| alert | `Record<string, boolean>` | 필수 | 키는 유효한 설문 ID (문자열), 값은 boolean |
| unsubscribedOrganizationIds | `string[]` | 선택 | 각 요소는 유효한 Organization ID (문자열) |

**출력 데이터**:

| 필드명 | 타입 | 설명 |
|--------|------|------|
| alert | `Record<string, boolean>` | 현재 사용자의 설문별 알림 활성화 상태 |
| unsubscribedOrganizationIds | `string[]` | 자동 구독 해제된 Organization ID 목록 |

#### 4.1.9 화면/UI 요구사항

해당 사항 없음 (데이터 모델).

#### 4.1.10 비기능 요구사항

| 항목 | 내용 |
|------|------|
| 하위 호환성 | 레거시 알림 설정 형식을 현재 형식으로 정상 변환해야 한다 (NFR-N05) |
| 데이터 무결성 | alert 레코드의 키는 실제 존재하는 설문 ID만 허용한다 |

### 4.2 알림 설정 페이지 (FN-028-02)

#### 4.2.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-028-02 |
| 기능명 | 알림 설정 페이지 |
| 관련 요구사항 ID | FR-046-02 |
| 우선순위 | 높음 |
| 기능 설명 | 사용자가 Organization 단위 자동 구독 설정과 설문별 알림 토글을 관리할 수 있는 페이지를 제공한다. Production 환경의 설문만 표시하며, 역할 기반 접근 제어를 적용한다. |

#### 4.2.2 선행 조건

- 사용자가 인증(로그인)되어 있어야 한다.
- 사용자가 billing 역할이 아니어야 한다.
- 사용자가 1개 이상의 Organization에 소속되어 있어야 한다.

#### 4.2.3 후행 조건

- 사용자의 알림 설정 변경 사항이 User 모델의 JSON 필드에 즉시 반영된다.

#### 4.2.4 기본 흐름

| 단계 | 행위자 | 동작 |
|------|--------|------|
| 1 | 사용자 | 알림 설정 페이지에 접근한다. |
| 2 | 시스템 | 사용자의 역할을 확인한다. billing 역할이면 접근을 차단한다. |
| 3 | 시스템 | 사용자가 접근 가능한 프로젝트 목록을 조회한다. owner/manager는 모든 프로젝트, 그 외 멤버는 팀을 통해 배정된 프로젝트만 포함한다. |
| 4 | 시스템 | 각 프로젝트의 Production 환경에 속한 설문 목록을 조회한다. Development 환경의 설문은 제외한다. |
| 5 | 시스템 | 현재 사용자의 알림 설정(alert 레코드)을 조회한다. |
| 6 | 시스템 | 각 설문에 대해 알림 활성화 상태를 초기화한다. 기존에 설정이 없는 설문은 비활성(false)으로 초기화한다. 레거시 형식은 현재 형식으로 변환한다. |
| 7 | 시스템 | Organization별 자동 구독 토글 상태를 표시한다. 자동 구독 해제 목록에 포함된 Organization은 비활성 상태로 표시한다. |
| 8 | 시스템 | 알림 설정 페이지를 렌더링한다. Organization 단위 자동 구독 토글과 설문별 알림 토글을 표시한다. |

#### 4.2.5 대안 흐름

| 조건 | 동작 |
|------|------|
| AF-01: 사용자에게 접근 가능한 설문이 없는 경우 | 빈 상태 메시지를 표시한다. 알림 토글 목록은 비어있는 상태로 렌더링한다. |
| AF-02: URL 파라미터를 통한 자동 해제 접근 | FN-028-06(구독 해제)의 기본 흐름을 따른다. 페이지 로드 시 URL 파라미터에 지정된 설문의 알림을 자동으로 비활성화한다. |

#### 4.2.6 예외 흐름

| 조건 | 동작 |
|------|------|
| EF-01: billing 역할 사용자 접근 | 알림 설정 페이지 접근을 거부한다. 설문 목록이 표시되지 않는다. |
| EF-02: 알림 설정 조회 실패 | 에러 메시지를 표시하고, 재시도 안내를 제공한다. |

#### 4.2.7 비즈니스 규칙

| 규칙 ID | 규칙 | 조건 | 결과 |
|---------|------|------|------|
| BR-028-04 | Production 환경 필터 | 알림 설정 페이지 로드 시 | Production 환경의 설문만 표시한다. Development 환경 설문은 노출하지 않는다. |
| BR-028-05 | 역할 기반 프로젝트 접근 | owner 또는 manager 역할 | 해당 Organization의 모든 프로젝트 접근 가능 |
| BR-028-06 | 역할 기반 프로젝트 접근 | member 역할 | 팀을 통해 배정된 프로젝트만 접근 가능 |
| BR-028-07 | billing 역할 제외 | billing 역할 사용자 | 알림 설정 페이지에서 설문을 볼 수 없음 |
| BR-028-08 | 설정 초기화 | 기존에 알림 설정이 없는 설문 | 비활성(false)으로 자동 초기화 |

#### 4.2.8 데이터 요구사항

**입력 데이터 (페이지 로드 시 조회)**:

| 필드명 | 타입 | 필수 여부 | 설명 |
|--------|------|----------|------|
| userId | string | 필수 | 현재 로그인한 사용자의 ID |
| organizationId | string | 필수 | 대상 Organization ID |

**출력 데이터 (페이지 렌더링)**:

| 필드명 | 타입 | 설명 |
|--------|------|------|
| organizations | `Organization[]` | 사용자 소속 Organization 목록 |
| surveys | `Survey[]` | 접근 가능한 Production 환경 설문 목록 |
| alertSettings | `Record<string, boolean>` | 설문별 알림 활성화 상태 |
| autoSubscribeEnabled | `boolean` | Organization 자동 구독 활성화 여부 |

#### 4.2.9 화면/UI 요구사항

- **레이아웃**: Organization 선택 영역과 설문별 알림 토글 목록으로 구성한다.
- **Organization 자동 구독 토글**: 페이지 상단에 "Auto-subscribe to new surveys" 토글 스위치를 배치한다.
- **설문별 알림 토글**: 각 설문 항목에 "Every Response" 알림 on/off 스위치를 배치한다.
- **설문 표시 정보**: 설문명, 설문 상태 등 식별 가능한 정보를 함께 표시한다.

### 4.5 알림 이메일 발송 (FN-028-05)

#### 4.5.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-028-05 |
| 기능명 | 알림 이메일 발송 |
| 관련 요구사항 ID | FR-046-05, FR-046-06 |
| 우선순위 | 높음 |
| 기능 설명 | 응답 완료(responseFinished) 이벤트 발생 시, 해당 설문에 알림이 활성화된 적격 사용자에게 이메일 알림을 발송한다. 이메일은 병렬로 발송되며 개별 실패가 전체에 영향을 주지 않는다. |

#### 4.5.2 선행 조건

- Pipeline API가 responseFinished 이벤트를 수신해야 한다.
- Pipeline API는 CRON_SECRET 기반 내부 인증을 통과해야 한다.
- 해당 설문이 Production 환경에 속해야 한다.

#### 4.5.3 후행 조건

- 적격 수신자 전원에게 이메일 발송이 시도된다.
- 개별 이메일 발송 실패는 로깅되며, 다른 수신자의 발송에 영향을 주지 않는다.

#### 4.5.4 기본 흐름

| 단계 | 행위자 | 동작 |
|------|--------|------|
| 1 | Pipeline | responseFinished 이벤트를 수신한다. |
| 2 | 시스템 | 이벤트 유형이 responseFinished인지 확인한다. responseCreated인 경우 알림 로직을 실행하지 않고 종료한다. |
| 3 | 시스템 | 해당 설문이 속한 Environment의 Organization을 조회한다. |
| 4 | 시스템 | 알림 수신 적격 사용자를 조회한다. 다음 조건을 모두 만족하는 사용자를 필터링한다: (a) 해당 환경의 Organization 멤버, (b) owner 또는 manager 역할이거나 해당 프로젝트에 팀을 통해 접근 가능한 사용자, (c) 해당 설문에 대한 alert 값이 true인 사용자. |
| 5 | 시스템 | 해당 설문의 현재까지의 응답 수를 조회한다. |
| 6 | 시스템 | 각 수신자에 대해 이메일 발송 Promise를 생성한다. 각 이메일에는 수신자의 locale 설정에 맞는 언어, 응답 수 정보, 구독 해제 링크가 포함된다. |
| 7 | 시스템 | 모든 이메일 발송 Promise를 병렬로 실행한다. |
| 8 | 시스템 | 개별 발송 결과를 로깅한다. |

#### 4.5.5 대안 흐름

| 조건 | 동작 |
|------|------|
| AF-05: 적격 수신자가 없는 경우 | 이메일 발송을 수행하지 않고 정상 종료한다. |
| AF-06: responseCreated 이벤트 | 알림 이메일 발송 로직을 실행하지 않는다. |

#### 4.5.6 예외 흐름

| 조건 | 동작 |
|------|------|
| EF-05: Pipeline API 인증 실패 | CRON_SECRET 인증 실패 시 미인증 응답(401)을 반환한다. 알림 발송이 수행되지 않는다. |
| EF-06: 개별 이메일 발송 실패 | 해당 이메일의 에러를 로깅하고, 나머지 수신자의 이메일 발송을 계속 진행한다. |
| EF-07: Organization 조회 실패 | 에러를 로깅하고 알림 발송을 건너뛴다. |

| 기능 설명 | 이메일 내 구독 해제 링크를 통한 설문별 알림 자동 비활성화와, Organization 자동 구독 해제 기능을 제공한다. |

#### 4.6.2 선행 조건

- (이메일 구독 해제) 사용자가 알림 이메일을 수신하고, 이메일 내 구독 해제 링크를 클릭해야 한다.
- (Organization 자동 구독 해제) 알림 설정 페이지에 접근한 상태여야 한다.

#### 4.6.3 후행 조건

- (이메일 구독 해제) 해당 설문의 alert 값이 false로 변경된다.
- (Organization 자동 구독 해제) 해당 Organization ID가 자동 구독 해제 목록에 추가된다.

#### 4.6.4 기본 흐름 - 이메일 내 구독 해제

| 단계 | 행위자 | 동작 |
|------|--------|------|
| 1 | 사용자 | 알림 이메일에 포함된 "Unsubscribe" 링크를 클릭한다. |
| 2 | 시스템 | URL 파라미터에서 설문 ID를 추출한다. |
| 3 | 시스템 | 사용자를 알림 설정 페이지로 리다이렉트한다. |
| 4 | 시스템 | 해당 설문의 alert 값을 false로 자동 변경한다. |
| 5 | 시스템 | 변경된 알림 설정을 User 모델에 저장한다. |
| 6 | 시스템 | 알림 설정 페이지에 변경된 상태를 반영하여 표시한다. |

#### 4.6.5 기본 흐름 - Organization 자동 구독 해제

| 단계 | 행위자 | 동작 |
|------|--------|------|
| 1 | 사용자 | 알림 설정 페이지에서 Organization의 "Auto-subscribe to new surveys" 토글을 비활성화한다. |
| 2 | 시스템 | 자동 구독 해제 Organization 목록에 해당 Organization ID를 추가한다. |
| 3 | 시스템 | 변경된 알림 설정을 User 모델에 저장한다. |
| 4 | 시스템 | 이후 해당 Organization에서 생성되는 새 설문에 대해 자동 구독이 적용되지 않는다. |

## 5. 데이터 모델

### 5.1 주요 엔티티 정의

#### User (알림 설정 관련 필드)

| 필드명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| id | string (PK) | 필수 | 사용자 고유 ID |
| notificationSettings | JSON | 필수 | 알림 설정 (alert + unsubscribedOrganizationIds) |
| locale | string | 선택 | 사용자 언어/지역 설정 (기본값: en-US) |

**notificationSettings JSON 구조**:

```json
{
  "alert": {
    "survey_id_1": true,
    "survey_id_2": false,
    "survey_id_3": true
  },
  "unsubscribedOrganizationIds": ["org_id_1"]
}
```

#### Survey (알림 관련 필드)

| 필드명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| id | string (PK) | 필수 | 설문 고유 ID |
| environmentId | string (FK) | 필수 | 소속 Environment ID |
| name | string | 필수 | 설문명 |
| status | enum | 필수 | 설문 상태 (draft, inProgress, paused, completed) |
| createdBy | string (FK) | 선택 | 설문 생성자 User ID |

#### Environment (알림 관련 필드)

| 필드명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| id | string (PK) | 필수 | Environment 고유 ID |
| type | enum | 필수 | 환경 유형 (production, development) |
| projectId | string (FK) | 필수 | 소속 Project ID |

#### Membership (알림 수신자 조회 관련)

| 필드명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| userId | string (FK) | 필수 | 사용자 ID |
| organizationId | string (FK) | 필수 | Organization ID |
| role | enum | 필수 | 역할 (owner, manager, member, billing) |

### 5.2 엔티티 간 관계

```
Organization (1) --- (*) Membership --- (*) User
     |
     v
Project (1) --- (2) Environment (production, development)
                       |
                       v
                  Survey (*) --- (*) Response
```

- **Organization - User**: Membership을 통한 다대다 관계. 역할(role) 정보 포함.
- **Organization - Project**: 1:N 관계. Organization 하위에 여러 프로젝트 존재.
- **Project - Environment**: 1:2 관계. production과 development 환경 고정.
- **Environment - Survey**: 1:N 관계. 환경 하위에 여러 설문 존재.
- **User.notificationSettings.alert**: 설문 ID를 키로 참조하는 논리적 관계 (JSON 내부).

### 5.3 데이터 흐름

```
[설문 생성]
  → 생성자 User의 notificationSettings.alert에 { surveyId: true } 추가
  → (조건: Organization 자동 구독 활성)

[알림 설정 변경]
  → User의 notificationSettings.alert[surveyId] 값 변경 (true/false)
  → 또는 unsubscribedOrganizationIds 배열 수정

[응답 완료]
  → Pipeline API 수신
  → Organization 멤버 조회 → 역할 필터 → alert 상태 필터
  → 적격 수신자에게 이메일 발송

[구독 해제]
  → URL 파라미터 파싱 → alert[surveyId] = false 저장
```

## 6. 인터페이스 명세

### 6.1 외부 시스템 연동

| 시스템 | 연동 방식 | 용도 |
|--------|----------|------|
| 이메일 서비스 | SMTP 또는 이메일 API | 알림 이메일 발송 |
| Pipeline API | 내부 HTTP API (CRON_SECRET 인증) | responseFinished 이벤트 수신 및 알림 트리거 |

### 6.2 API 명세

#### 6.2.1 알림 설정 조회

| 항목 | 내용 |
|------|------|
| 목적 | 현재 사용자의 알림 설정 조회 |
| 인증 | 세션 기반 인증 |
| 입력 | userId (세션에서 추출) |
| 출력 | `{ alert: Record<string, boolean>, unsubscribedOrganizationIds: string[] }` |

#### 6.2.2 알림 설정 갱신

| 항목 | 내용 |
|------|------|
| 목적 | 사용자의 알림 설정 업데이트 |
| 인증 | 세션 기반 인증 |
| 입력 | `{ alert?: Record<string, boolean>, unsubscribedOrganizationIds?: string[] }` |
| 출력 | 업데이트된 알림 설정 |
| 유효성 검증 | alert의 키는 유효한 설문 ID, 값은 boolean. unsubscribedOrganizationIds의 각 요소는 유효한 Organization ID. |

#### 6.2.3 Pipeline 알림 발송 (내부)

| 항목 | 내용 |
|------|------|
| 목적 | responseFinished 이벤트에 대한 알림 이메일 발송 |
| 인증 | CRON_SECRET 헤더 기반 내부 인증 |
| 입력 | `{ event: "responseFinished", surveyId: string, responseId: string, environmentId: string }` |
| 출력 | 발송 결과 (성공/실패 건수) |
| 인증 실패 시 | 401 Unauthorized |

## 8. 제약사항 및 가정

### 8.1 기술적 제약사항

| 항목 | 값 | 설명 |
|------|-----|------|
| 알림 채널 | 이메일만 | 인앱 알림, Slack/Discord 등 외부 채널 미지원 |
| 알림 설정 저장 | User 모델의 JSON 필드 | 별도 테이블이 아닌 JSON 필드에 저장 |
| Pipeline 인증 | CRON_SECRET 환경변수 | 내부 API 호출 시 반드시 인증 헤더 필요 |
| 이메일 발송 방식 | 병렬 (개별 실패 격리) | Promise.allSettled 또는 동등한 병렬 처리 |

### 8.2 비즈니스 제약사항

| 항목 | 값 | 설명 |
|------|-----|------|
| 알림 트리거 | responseFinished만 | responseCreated에서는 알림 미발송 |
| 자동 구독 대상 | 설문 생성자만 | Organization 전체 멤버 자동 구독 미지원 |
| 알림 대상 환경 | Production만 | Development 환경의 설문은 알림 미대상 |
| 제외 역할 | billing | billing 역할 사용자는 알림 설정 접근 불가 |
| 수신자 권한 조건 | owner/manager 또는 팀 멤버 | 역할 기반 및 팀 기반 접근 제어 |
| 라이선스 | Community (기본 기능) | 별도 Enterprise 라이선스 불필요 |

### 8.3 가정사항

| 가정 | 설명 |
|------|------|
| 이메일 서비스 가용성 | 외부 이메일 서비스(SMTP 또는 API)가 정상 동작함을 전제한다 |
| Pipeline 시스템 동작 | FSD-022에서 정의한 Pipeline 시스템이 정상 운영되고 있음을 전제한다 |
| User 모델 JSON 지원 | 사용 중인 데이터베이스가 JSON 필드 타입을 지원함을 전제한다 (PostgreSQL jsonb 등) |
| locale 설정 존재 | 사용자에게 locale이 미설정된 경우 기본 locale(en-US)이 적용됨을 전제한다 |
