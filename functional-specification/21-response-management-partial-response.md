## 2. 개요

### 2.1 목적

Inquiry 설문에 수집된 개별 응답(Response)을 카드 뷰 및 테이블 뷰로 조회하고, 필터링/태깅/삭제를 수행하는 기능을 정의한다. 완료된 응답뿐 아니라 부분 응답(Partial Response)도 실시간으로 확인할 수 있으며, 각 질문별 소요 시간(TTC: Time-to-Complete)을 추적한다.

### 2.2 범위

**포함 범위 (In-scope)**
- 응답 카드 뷰 렌더링
- 응답 테이블 뷰 렌더링
- 응답 필터 조건 (22가지 filter condition + 추가 criteria)
- 응답 태그(Tag) 관리 (생성, 연결, 삭제)
- 응답 메타데이터 표시 (source, URL, browser, OS, device, country, action, IP)
- 부분 응답 표시 및 질문별 스킵/중단 상태 시각화
- 응답 상태 (finished / in-progress) 표시
- 응답 삭제 (5분 제한 규칙 포함)
- 질문별 TTC 추적

**제외 범위 (Out-of-scope)**
- 응답 분석/요약(Summary) 기능 (FSD-025 참조)
- 응답 내보내기(Export) 기능 (FSD-025 참조)
- 웹훅/후속 메일 파이프라인 (FSD-022, FSD-023 참조)

### 2.3 대상 사용자

| 역할 | 설명 | 권한 수준 |
|------|------|----------|
| Survey Manager | 응답 목록 조회, 필터링, 태깅, 삭제 수행 | 읽기/쓰기 |
| Project Member (Read) | 읽기 전용으로 응답 확인 | 읽기 전용 |
| Anonymous Respondent | 설문 응답 제출 (부분 응답 포함) | 응답 제출만 가능 |
| Contact (인증 사용자) | userId 기반 식별 가능한 응답 제출 | 응답 제출만 가능 |

### 2.4 용어 정의

| 용어 | 정의 |
|------|------|
| Response | 설문에 대한 개별 응답 단위. 질문별 응답 데이터, 메타데이터, 태그 등을 포함 |
| Partial Response | 설문을 끝까지 완료하지 않은 부분 응답. `finished = false` 상태 |
| TTC (Time-to-Complete) | 각 질문별 응답 소요 시간(ms 단위). `_total` 키로 전체 합산 시간도 저장 |
| Skipped | 응답자가 중간 질문을 건너뛴 상태 |
| Aborted | 미완료 응답에서 마지막 부분의 미응답 질문 상태 |
| Tag | 응답에 부여하는 분류 라벨. Environment 단위로 고유 이름 관리 |
| Filter Condition | 응답 목록을 특정 조건으로 걸러내는 연산자 |
| Contact | userId 기반으로 식별 가능한 인증된 응답자 |
| Environment | Inquiry에서 설문이 배포되는 환경 단위 |
| CUID2 | 고유 식별자 생성 알고리즘. 응답 ID, 설문 ID 등에 사용 |
| Display | 설문이 사용자에게 노출된 이벤트 |
| Ending | 설문 완료 후 표시되는 종료 화면 |
| Quota | 응답 수량 제한 관리 단위 |

## 4. 상세 기능 명세

### 4.1 응답 카드 뷰

#### 4.1.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-035-01 |
| 기능명 | 응답 카드 뷰 |
| 관련 요구사항 ID | FR-035, AC-035-01 |
| 우선순위 | 높음 |
| 기능 설명 | 개별 응답을 카드 형태로 렌더링하여 질문별 응답 내용, 연락처 정보, 태그, 메타데이터를 시각적으로 표시한다 |

#### 4.1.2 선행 조건

- 사용자가 해당 Survey에 접근 권한(Survey Manager 또는 Project Member Read)을 보유해야 한다
- 해당 Survey에 1건 이상의 응답 데이터가 존재해야 한다

#### 4.1.3 후행 조건

- 응답 카드 목록이 화면에 렌더링된다
- 각 카드에 헤더, 바디, 풋터 영역이 올바르게 구성된다

#### 4.1.4 기본 흐름

| 단계 | 행위자 | 동작 |
|------|--------|------|
| 1 | 시스템 | 현재 Survey의 응답 목록을 페이지네이션 기반으로 조회한다 (기본 10건) |
| 2 | 시스템 | 각 응답을 카드 형태로 렌더링한다 |
| 3 | 시스템 | **헤더 영역**: 연락처 정보(아바타, 이름), 응답 시간(상대 시간), 삭제 버튼을 표시한다 |
| 4 | 시스템 | **바디 영역**: 질문별 응답 데이터, 스킵/중단 표시, 변수, 히든 필드를 표시한다 |
| 5 | 시스템 | **풋터 영역**: 태그 관리 UI, 메타데이터 아이콘을 표시한다 |
| 6 | 사용자 | 스크롤하여 추가 응답 카드를 로딩한다 |

#### 4.1.5 대안 흐름

| 조건 | 동작 |
|------|------|
| AF-01: 연락처 ID가 없는 경우 | 아바타 대신 기본 아이콘을 표시하고, 이름 대신 "Anonymous"를 표시한다 |
| AF-02: 연락처의 userId가 존재하는 경우 | 이름 옆에 userId ID 배지를 추가 표시한다 |
| AF-03: 읽기 전용 사용자인 경우 | 삭제 버튼과 태그 편집 UI를 숨긴다 |
| AF-04: 필터가 적용된 경우 | 필터 조건에 부합하는 응답만 카드로 렌더링한다 |

#### 4.1.6 예외 흐름

| 조건 | 동작 |
|------|------|
| EF-01: 응답 데이터가 0건인 경우 | 빈 상태(Empty State) 메시지를 표시한다 |
| EF-02: 응답 조회 API 실패 | 에러 메시지를 표시하고 재시도 옵션을 제공한다 |

#### 4.1.7 비즈니스 규칙

| 규칙 ID | 규칙 |
|---------|------|
| BR-01 | 응답 카드 헤더의 시간은 사용자 locale에 맞는 상대 시간(예: "5분 전", "2시간 전")으로 표시한다 |
| BR-02 | 완료된 응답(finished = true)에는 체크 아이콘과 "Completed" 배지를 표시한다 |
| BR-03 | 미완료 응답(finished = false)에는 스킵/중단 그룹화 시각화를 적용한다 |

#### 4.1.8 데이터 요구사항

**입력 데이터 (응답 조회 요청)**

| 필드명 | 타입 | 필수 | 유효성 검증 |
|--------|------|------|------------|
| surveyId | string (CUID2) | 필수 | CUID2 형식 검증 |
| page | number | 선택 | 양의 정수, 기본값 1 |
| pageSize | number | 선택 | 양의 정수, 기본값 10 |
| filterCriteria | FilterCriteria | 선택 | 유효한 필터 조건 객체 |

**출력 데이터 (응답 카드 렌더링)**

| 필드명 | 타입 | 설명 |
|--------|------|------|
| id | string (CUID2) | 응답 고유 식별자 |
| createdAt | DateTime | 생성 시간 |
| updatedAt | DateTime | 수정 시간 |
| surveyId | string (CUID2) | 설문 ID |
| displayId | string (CUID2) \| null | Display 이벤트 ID |
| contact | Contact \| null | 연락처 정보 |
| contactAttributes | Record<string, string> | 연락처 속성 |
| finished | boolean | 완료 여부 |
| endingId | string \| null | 종료 화면 ID |
| data | Record<string, ResponseValue> | 질문별 응답 데이터 |
| variables | Record<string, string \| number> | 변수 데이터 |
| ttc | Record<string, number> | 질문별 소요 시간 (ms) |
| tags | Tag[] | 태그 배열 |
| meta | ResponseMeta | 메타데이터 |
| singleUseId | string \| null | 일회성 링크 ID |
| language | string \| null | 응답 언어 |

**ResponseValue 타입**

| 타입 | 설명 | 예시 |
|------|------|------|
| string | 텍스트, 단일 선택 등 | `"서울"` |
| number | NPS, Rating 등 | `8` |
| string[] | 복수 선택, 파일 업로드 URL 등 | `["A", "B"]` |
| Record<string, string> | Matrix 질문 등 키-값 매핑 | `{"행1": "열2"}` |
| undefined | 미응답 | - |

#### 4.1.9 화면/UI 요구사항

**카드 레이아웃 구조**

| 영역 | 구성 요소 | 세부 사항 |
|------|----------|----------|
| 헤더 | 연락처 아바타/이름 | 연락처 존재 시 아바타 + 이름, 미존재 시 기본 아이콘 + "Anonymous" |
| 헤더 | 응답 시간 | locale 적용 상대 시간 |
| 헤더 | 삭제 버튼 | 읽기 전용 모드에서는 숨김 |
| 헤더 | userId 배지 | userId 존재 시에만 표시 |
| 바디 | 질문별 응답 | 질문 텍스트와 응답 값을 쌍으로 표시 |
| 바디 | 스킵/중단 표시 | 건너뛴 질문 그룹을 시각적으로 구분 |
| 바디 | 변수/히든 필드 | 설정된 변수 및 히든 필드 값 표시 |
| 풋터 | 태그 관리 | 태그 추가/삭제/검색 UI |
| 풋터 | 메타데이터 아이콘 | 디바이스, 브라우저 등 아이콘으로 표시 |

#### 4.1.10 비기능 요구사항

- 페이지네이션 기반 로딩으로 기본 10건씩 조회하여 초기 로딩 시간을 최소화한다

### 4.2 응답 테이블 뷰

#### 4.2.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-035-02 |
| 기능명 | 응답 테이블 뷰 |
| 관련 요구사항 ID | FR-035, AC-036-03 |
| 우선순위 | 높음 |
| 기능 설명 | TanStack Table 기반으로 응답 데이터를 테이블 형태로 렌더링하며, 열 순서 변경, 열 가시성 설정, 행 선택, 무한 스크롤 페이지네이션을 지원한다 |

#### 4.2.2 선행 조건

- 사용자가 해당 Survey에 접근 권한을 보유해야 한다
- 해당 Survey에 1건 이상의 응답 데이터가 존재해야 한다

#### 4.2.3 후행 조건

- 응답 테이블이 화면에 렌더링된다
- 사용자가 변경한 열 순서와 가시성 설정이 유지된다

#### 4.2.4 기본 흐름

| 단계 | 행위자 | 동작 |
|------|--------|------|
| 1 | 시스템 | 현재 Survey의 응답 목록을 무한 스크롤 기반으로 초기 로딩한다 |
| 2 | 시스템 | TanStack Table로 테이블을 렌더링한다 |
| 3 | 시스템 | 기본 열 구성(responseId, createdAt, status, tags, responseData 등)으로 표시한다 |
| 4 | 사용자 | 테이블 하단으로 스크롤하면 추가 데이터가 자동 로딩된다 |
| 5 | 사용자 | 열 헤더를 드래그하여 열 순서를 변경한다 |
| 6 | 사용자 | 열 가시성 메뉴에서 표시할 열을 선택/해제한다 |
| 7 | 사용자 | 행 체크박스를 선택하여 일괄 작업 대상을 지정한다 |

#### 4.2.5 대안 흐름

| 조건 | 동작 |
|------|------|
| AF-01: 선택된 행이 1건 이상인 경우 | 일괄 삭제 및 일괄 다운로드 버튼을 활성화한다 |
| AF-02: 읽기 전용 사용자인 경우 | 행 선택 체크박스와 일괄 삭제 버튼을 숨긴다 |

#### 4.2.6 예외 흐름

| 조건 | 동작 |
|------|------|
| EF-01: 무한 스크롤 데이터 로딩 실패 | 에러 메시지를 표시하고 재시도 옵션을 제공한다 |
| EF-02: 응답 데이터가 0건인 경우 | 빈 상태 메시지를 표시한다 |

#### 4.2.7 비즈니스 규칙

| 규칙 ID | 규칙 |
|---------|------|
| BR-04 | 열 순서 변경은 Drag-and-drop 방식으로만 수행한다 |
| BR-05 | 무한 스크롤 시 한 번에 10건씩 추가 로딩한다 |
| BR-06 | 일괄 삭제 시 미완료 응답의 5분 삭제 제한 규칙이 개별적으로 적용된다 |

- 선택된 행 수와 일괄 작업 버튼(삭제, 다운로드)을 테이블 상단에 표시한다

#### 4.2.10 비기능 요구사항

- 무한 스크롤 기반으로 데이터를 점진적으로 로딩하여 대량의 응답 데이터도 원활하게 표시한다

### 4.4 부분 응답 처리 및 시각화

#### 4.4.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-035-04 |
| 기능명 | 부분 응답 처리 및 시각화 |
| 관련 요구사항 ID | FR-035, AC-035-01 |
| 우선순위 | 높음 |
| 기능 설명 | 미완료 응답(Partial Response)의 스킵/중단 상태를 그룹화하여 시각적으로 구분 표시한다. 완료 여부와 질문 응답 유무를 조합하여 Completed, Skipped, Aborted 상태를 판별한다 |

#### 4.4.2 선행 조건

- 응답 데이터와 해당 Survey의 질문 목록이 로딩되어 있어야 한다
- 응답의 finished 필드와 data 필드가 존재해야 한다

#### 4.4.3 후행 조건

- 각 질문별로 Completed, Skipped, Aborted 상태가 시각적으로 구분되어 표시된다
- 연속 미응답 질문이 그룹으로 묶여 표시된다

#### 4.4.4 기본 흐름

| 단계 | 행위자 | 동작 |
|------|--------|------|
| 1 | 시스템 | 응답의 finished 값을 확인한다 |
| 2a | 시스템 | finished = true인 경우: 체크 아이콘과 "Completed" 배지를 표시한다 |
| 2b | 시스템 | finished = false인 경우: 부분 응답 시각화를 적용한다 |
| 3 | 시스템 | 각 질문의 응답 값이 유효한지 판별한다 (유효 값 판별 기준 적용) |
| 4 | 시스템 | 완료된 응답인 경우 순방향 스킵 그룹화 로직을 적용한다 |
| 5 | 시스템 | 미완료 응답인 경우 역방향 스킵 그룹화 로직을 적용한다 |
| 6 | 시스템 | 그룹화된 스킵/중단 질문을 시각적으로 구분하여 렌더링한다 |

#### 4.4.5 대안 흐름

| 조건 | 동작 |
|------|------|
| AF-01: 모든 질문에 유효 응답이 있는 완료 응답 | 스킵/중단 표시 없이 전체 "Completed" 상태로 표시한다 |
| AF-02: 미완료 응답에 일부 질문만 응답된 경우 | 마지막 미응답 질문부터 역방향으로 "aborted" 그룹을 생성하고, 중간 미응답은 "skipped"로 표시한다 |

#### 4.4.6 예외 흐름

| 조건 | 동작 |
|------|------|
| EF-01: 응답 data 필드가 비어있는 경우 | 모든 질문을 미응답으로 처리한다 |

#### 4.4.7 비즈니스 규칙

| 규칙 ID | 규칙 |
|---------|------|
| BR-10 | **유효 응답 값 판별 기준**: 다음 중 하나에 해당하면 유효한 응답 값이다 -- (1) 빈 문자열("")이 아닌 문자열 (2) 1개 이상의 항목이 있는 배열 (3) 숫자 타입 (4) 1개 이상의 키가 있는 객체 |
| BR-11 | **완료 응답 스킵 그룹화**: 순방향(첫 질문부터)으로 질문을 순회하며, 연속된 미응답 질문을 하나의 "skipped" 그룹으로 묶는다. 유효 응답이 있는 질문을 만나면 현재 그룹을 종료하고 새 그룹을 시작한다 |
| BR-12 | **미완료 응답 스킵 그룹화**: 역방향(마지막 질문부터)으로 질문을 순회하며, 마지막부터 연속된 미응답 질문을 "aborted" 그룹으로 생성한다. 그 이전의 미응답 질문은 "skipped" 그룹으로 처리한다 |

**상태 판별 로직 상세**

| 상태 | 조건 | 시각적 표현 |
|------|------|------------|

| Skipped | 중간 질문이 건너뛰어짐 (유효 값 없음) | "skipped" 상태 표시 |
| Aborted | 미완료 응답의 마지막 연속 미응답 구간 | "aborted" 상태 표시 |

#### 4.4.8 데이터 요구사항

**유효 값 판별 입력**

|------|------|------|

### 4.6 응답 삭제

#### 4.6.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-035-06 |
| 기능명 | 응답 삭제 |
| 관련 요구사항 ID | FR-035, AC-035-02 |
| 우선순위 | 높음 |
| 기능 설명 | 완료/미완료 응답을 삭제한다. 미완료 응답은 마지막 업데이트 후 5분 경과해야 삭제 가능하다. Quota가 연결된 응답 삭제 시 Quota 차감 옵션을 제공한다 |

#### 4.6.2 선행 조건

- 사용자가 Survey Manager 권한을 보유해야 한다 (읽기 전용 사용자는 삭제 불가)
- 삭제 대상 응답이 존재해야 한다

#### 4.6.3 후행 조건

- 해당 응답이 데이터베이스에서 삭제된다
- Quota 차감 옵션이 선택된 경우 관련 Quota 카운트가 감소한다
- 응답 목록에서 해당 카드/행이 제거된다

#### 4.6.4 기본 흐름

| 단계 | 행위자 | 동작 |
|------|--------|------|
| 1 | 사용자 | 응답 카드 또는 테이블 행의 삭제 버튼을 클릭한다 |
| 2 | 시스템 | 응답의 finished 값을 확인한다 |
| 3a | 시스템 | finished = true인 경우: 삭제 확인 다이얼로그를 표시한다 |
| 3b | 시스템 | finished = false인 경우: updatedAt과 현재 시간의 차이를 계산한다 |
| 4 | 시스템 | (3b 경로) 5분 이상 경과한 경우 삭제 확인 다이얼로그를 표시한다 |
| 5 | 시스템 | Quota가 연결된 응답인 경우, Quota 차감 여부 옵션을 함께 표시한다 |
| 6 | 사용자 | 삭제를 확인한다 |
| 7 | 시스템 | 응답을 삭제하고 목록을 갱신한다 |

#### 4.6.5 대안 흐름

| 조건 | 동작 |
|------|------|
| AF-01: 사용자가 삭제를 취소한 경우 | 다이얼로그를 닫고 아무 작업도 수행하지 않는다 |
| AF-02: 테이블 뷰에서 일괄 삭제 | 선택된 모든 행에 대해 개별적으로 삭제 규칙을 적용한다. 5분 미경과 미완료 응답은 삭제에서 제외한다 |

#### 4.6.6 예외 흐름

| 조건 | 동작 |
|------|------|
| EF-01: 미완료 응답이 5분 미경과인 경우 | 삭제 버튼을 비활성화하고 "This response is in progress" 툴팁을 표시한다 |
| EF-02: 삭제 API 호출 실패 | 에러 메시지를 표시하고 응답 목록 상태를 원복한다 |

#### 4.6.7 비즈니스 규칙

| 규칙 ID | 규칙 |
|---------|------|
| BR-16 | 완료된 응답(finished = true)은 즉시 삭제 가능하다 |
| BR-17 | 미완료 응답(finished = false)은 updatedAt 기준 5분(300,000ms) 이상 경과해야 삭제 가능하다 |
| BR-18 | 5분 미경과 미완료 응답의 삭제 버튼은 disabled 상태이며, "This response is in progress" 툴팁을 표시한다 |

#### 4.6.8 데이터 요구사항

#### 4.7.1 기능 개요

| 항목 | 내용 |

| 기능 설명 | 설문 응답자가 제출하는 신규 응답을 생성하고, 부분 응답을 업데이트한다 |

#### 4.7.2 선행 조건

- 유효한 environmentId와 surveyId가 제공되어야 한다
- 해당 Survey가 활성 상태여야 한다

#### 4.7.3 후행 조건

- 신규 응답이 데이터베이스에 저장된다
- 또는 기존 부분 응답이 업데이트된다

#### 4.7.4 기본 흐름 (신규 응답 생성)

| 단계 | 행위자 | 동작 |
|------|--------|------|
| 1 | 응답자 | 설문의 첫 번째 질문에 응답한다 |
| 2 | 시스템 | 신규 응답 레코드를 생성한다 (finished = false) |
| 3 | 시스템 | data, variables, ttc, meta 필드를 저장한다 |
| 4 | 응답자 | 후속 질문에 계속 응답한다 |
| 5 | 시스템 | 응답 데이터를 업데이트한다 |
| 6 | 응답자 | 설문을 완료한다 |
| 7 | 시스템 | finished = true로 업데이트하고 endingId를 저장한다 |

#### 4.7.5 대안 흐름

| 조건 | 동작 |
|------|------|
| AF-01: 응답자가 중간에 이탈한 경우 | finished = false 상태로 부분 응답이 유지된다 |
| AF-02: userId가 제공된 경우 | Contact 정보와 연결하여 응답을 저장한다 |
| AF-03: hiddenFields가 존재하는 경우 | 업데이트 시 히든 필드 값을 함께 저장한다 |

#### 4.7.6 예외 흐름

| 조건 | 동작 |
|------|------|
| EF-01: environmentId 또는 surveyId가 유효하지 않은 경우 | 400 에러를 반환한다 |
| EF-02: 데이터베이스 저장 실패 | 500 에러를 반환하고 재시도를 안내한다 |

#### 4.7.7 비즈니스 규칙

| BR-24 | meta 필드의 url, source, action은 업데이트 시 갱신 가능하다 |

#### 4.7.8 데이터 요구사항

**신규 응답 생성 입력**

| tagId | string (CUID2) | FK, NOT NULL | 태그 ID |

**PK**: (responseId, tagId) 복합 기본키

#### Contact

| 필드 | 타입 | 제약 조건 | 설명 |
|------|------|----------|------|
| id | string (CUID2) | PK, NOT NULL | 연락처 고유 식별자 |
| userId | string | NULLABLE | 사용자 식별 ID |
| attributes | JSON | NOT NULL, DEFAULT {} | 연락처 속성 |

### 5.2 엔티티 간 관계

```
Survey (1) ──── (N) Response
Response (N) ──── (N) Tag        [through ResponseTag]
Response (N) ──── (1) Contact    [optional]
Response (N) ──── (1) Display    [optional]
Tag (N) ──── (1) Environment
```

### 5.3 데이터 흐름

```
[응답자] ──(응답 제출)──> [Response 생성/업데이트]
                                |
                                v
                      [Response DB 저장]
                                |
                                v
                      [Survey Manager 조회]
                        /       |       \
                       v        v        v
                 [카드 뷰]  [테이블 뷰]  [필터 적용]
                       \        |       /
                        v       v      v
                     [태그 관리 / 삭제]
```

## 7. 비기능 요구사항

### 7.1 성능 요구사항

| 항목 | 요구사항 |
|------|---------|
| 페이지네이션 | 응답 목록은 기본 10건 단위로 페이지네이션하여 로딩한다 |
| 무한 스크롤 | 테이블 뷰에서 스크롤 시 10건씩 추가 로딩한다 |
| 초기 로딩 | 첫 페이지 응답 10건 조회는 합리적 시간 내에 완료되어야 한다 |

### 7.2 보안 요구사항

| 항목 | 요구사항 |
|------|---------|
| 접근 제어 | 읽기 전용 사용자(readOnly 플래그)는 태그 편집 및 응답 삭제가 차단된다 |
| 개인정보 보호 | Contact 개인정보는 외부 추적 도구(analytics, tracking 등)에서 제외 처리한다 |
| IP 주소 | IP 주소는 메타데이터로만 저장하며, 관리자 화면에서만 확인 가능하다 |

### 7.3 가용성 요구사항

| 항목 | 요구사항 |
|------|---------|
| 실시간성 | 부분 응답은 마지막 업데이트 시간 기준으로 상태를 실시간 반영한다 |
| 데이터 정합성 | 태그 작업 시 동시 실행을 방지하여 데이터 정합성을 보장한다 |
| Quota 연동 | 응답 삭제 시 Quota 차감 옵션을 통해 Quota 데이터와 정합성을 유지한다 |

## 8. 제약사항 및 가정

### 8.1 기술적 제약사항

| 항목 | 값 |
|------|-----|
| 응답 ID 형식 | CUID2 |
| 테이블 렌더링 | TanStack Table 라이브러리 사용 필수 |
| TTC 단위 | 밀리초(ms) |
| 지원 메타데이터 필드 | 8개 (source, url, browser, os, device, country, action, ipAddress) |

### 8.2 비즈니스 제약사항

| 항목 | 값 |
|------|-----|
| 미완료 응답 삭제 대기 시간 | 5분 (300,000ms) |
| 기본 페이지 크기 | 10건 |
| 태그 이름 유일성 범위 | Environment 단위 |
| 태그 하이라이트 지속 시간 | 2초 (2,000ms) |
| 필터 연산자 수 | 22 + 추가 criteria |
| 다중 필터 조합 방식 | AND 연산 |

### 8.3 가정사항

| 항목 | 내용 |
|------|------|
| Survey 모듈 | 설문 및 질문 데이터 조회 API가 이미 구현되어 있다고 가정 |
| Contact 모듈 | 연락처 정보 및 속성 조회 API가 이미 구현되어 있다고 가정 |
| Environment 모듈 | Environment 설정 조회 API가 이미 구현되어 있다고 가정 |
| 인증/권한 | 사용자 인증 및 역할 기반 접근 제어가 이미 구현되어 있다고 가정 |
| 시간 기준 | 서버와 클라이언트 간 시간 동기화가 되어 있다고 가정 (5분 삭제 제한 판별) |
