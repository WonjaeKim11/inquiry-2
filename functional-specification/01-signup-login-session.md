## 2. 개요

### 2.1 목적

Inquiry 플랫폼의 사용자 인증 시스템에 대한 기능 명세를 정의한다. 이메일/비밀번호 기반의 Credentials 인증과 이메일 토큰 기반 검증을 지원하며, Next-Auth 4.24.12 (patched)를 사용하여 JWT 세션 관리를 구현한다. 보안 관점에서 Timing Attack 방지, bcrypt DoS 방지, IP 기반 Rate Limiting을 적용한다.

### 2.2 범위

**In-scope:**
- 이메일/비밀번호 기반 회원가입 (Credentials Provider)
- 이메일 검증 토큰 기반 로그인 (Token Provider)
- 비밀번호 재설정 (Forgot Password / Reset)
- JWT 세션 관리
- IP 기반 Rate Limiting
- Turnstile CAPTCHA 연동 (Cloudflare)
- 초대(Invite) 기반 회원가입 흐름

**Out-of-scope:**
- SSO/OAuth 인증 (FSD-002에서 다룸)
- 2FA/TOTP (FSD-002에서 다룸)
- 조직(Organization) 관리 (FSD-003에서 다룸)

### 2.3 대상 사용자

| 역할 | 설명 |
|------|------|
| 신규 사용자 | 회원가입 및 이메일 인증을 통해 계정을 생성하는 사용자 |
| 기존 사용자 | 이메일/비밀번호로 로그인하는 사용자 |
| 초대받은 사용자 | 조직 초대를 통해 가입하는 사용자 |
| 시스템 관리자 | Rate Limit, 이메일 검증 정책 등을 관리 |

### 2.4 용어 정의

| 용어 | 정의 |
|------|------|
| Credentials Provider | Next-Auth에서 이메일/비밀번호 기반 인증을 처리하는 인증 공급자 |
| Token Provider | 이메일 토큰 기반으로 사용자 검증을 수행하는 인증 공급자 |
| CONTROL_HASH | Timing Attack 방지를 위해 사용되는 더미 bcrypt 해시값. bcrypt cost factor 12로 생성 |
| Turnstile | Cloudflare에서 제공하는 CAPTCHA 대체 봇 감지 서비스 |
| JWT (JSON Web Token) | 세션 관리에 사용되는 토큰 형식 |
| bcrypt | 비밀번호 해싱에 사용되는 단방향 암호화 알고리즘 |
| Rate Limiting | 특정 시간 내 허용되는 요청 횟수를 제한하는 보안 기법 |
| User Enumeration | 시스템 응답 차이를 통해 사용자 존재 여부를 파악하는 공격 기법 |
| Multi-Org | 하나의 사용자가 여러 조직에 소속될 수 있는 기능 모드 |
| Membership | 사용자와 조직 간의 소속 관계를 나타내는 엔티티 |
| Brevo | 마케팅 자동화 플랫폼 (이전 Sendinblue). 고객 데이터 연동에 사용 |
| Audit Log | 시스템 이벤트를 기록하는 감사 로그 |
| Server Action | Next.js의 서버 사이드 함수 호출 방식 |

## 4. 상세 기능 명세

### 4.1 비밀번호 유효성 검사

#### 4.1.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-001 |
| 기능명 | 비밀번호 유효성 검사 규칙 |
| 관련 요구사항 ID | FR-001-01 |
| 우선순위 | 필수 |
| 기능 설명 | 비밀번호 입력 시 보안 기본 요건과 bcrypt DoS 방지를 위한 유효성 검사를 수행한다 |

#### 4.1.2 선행 조건

- 사용자가 비밀번호 입력 필드에 값을 입력한 상태

#### 4.1.3 후행 조건

- 유효성 검사 통과 시: 다음 처리 단계로 진행
- 유효성 검사 실패 시: 에러 메시지 반환, 처리 중단

#### 4.1.4 기본 흐름

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | 비밀번호 문자열 길이가 8자 이상인지 확인 |
| 2 | 시스템 | 비밀번호 문자열 길이가 128자 이하인지 확인 |
| 3 | 시스템 | 대문자가 1자 이상 포함되어 있는지 확인 |
| 4 | 시스템 | 숫자가 1자 이상 포함되어 있는지 확인 |
| 5 | 시스템 | 모든 조건 충족 시 유효성 검사 통과 |

#### 4.1.5 대안 흐름

- 해당 없음

#### 4.1.6 예외 흐름

| 조건 | 에러 메시지 | 처리 |
|------|------------|------|
| 8자 미만 | "비밀번호는 최소 8자 이상이어야 합니다" | 요청 거부 |
| 128자 초과 | "비밀번호는 최대 128자까지 가능합니다" | 요청 거부 |
| 대문자 미포함 | "비밀번호에 대문자가 1자 이상 포함되어야 합니다" | 요청 거부 |
| 숫자 미포함 | "비밀번호에 숫자가 1자 이상 포함되어야 합니다" | 요청 거부 |

#### 4.1.7 비즈니스 규칙

| 규칙 ID | 규칙 | 근거 |
|---------|------|------|
| BR-001-01 | 최소 길이 8자 | 보안 기본 요건 |
| BR-001-02 | 최대 길이 128자 | bcrypt DoS 방지. bcrypt는 72 bytes까지 처리하나, 입력 전처리 과정의 DoS를 방지하기 위해 128자로 제한 |
| BR-001-03 | 대문자 1자 이상 필수 | 비밀번호 복잡도 보장 |
| BR-001-04 | 숫자 1자 이상 필수 | 비밀번호 복잡도 보장 |

#### 4.1.8 데이터 요구사항

**입력 데이터:**

| 필드명 | 타입 | 필수 | 유효성 검증 규칙 |
|--------|------|------|-----------------|
| password | String | O | 최소 8자, 최대 128자, 대문자 1자 이상, 숫자 1자 이상 |

**출력 데이터:**

| 필드명 | 타입 | 설명 |
|--------|------|------|
| isValid | Boolean | 유효성 검사 통과 여부 |
| errors | String[] | 유효성 검사 실패 시 에러 메시지 목록 |

#### 4.4.2 선행 조건

- 사용자가 회원가입 페이지에 접근한 상태
- 시스템이 정상 동작 중 (DB 연결, 이메일 서비스 가용)

#### 4.4.3 후행 조건

- 성공 시:
  - DB에 새 사용자 레코드 생성 (비밀번호는 bcrypt 해시로 저장)
  - 이메일 검증 메일 발송 (비활성화 설정이 아닌 경우)
  - 초대 토큰이 있는 경우: Membership 생성, Team Membership 생성, 초대 레코드 삭제
  - 초대 토큰이 없고 Multi-Org 활성화 시: 새 Organization 생성 (owner 역할)
  - Audit Log에 사용자 생성 이벤트 기록
- 실패 시:
  - 이미 존재하는 이메일: 에러를 노출하지 않고 성공 응답 반환 (User Enumeration 방지)
  - Rate Limit 초과: 요청 거부

#### 4.4.4 기본 흐름

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 사용자 | 회원가입 페이지에서 이름, 이메일, 비밀번호를 입력하고 제출 |
| 2 | 시스템 | IP 기반 Rate Limit 확인 (30회/시간). 초과 시 예외 흐름 EX-004-01로 이동 |
| 3 | 시스템 | Turnstile CAPTCHA 검증 (환경에 Turnstile이 설정된 경우). 실패 시 예외 흐름 EX-004-02로 이동 |
| 4 | 시스템 | 입력값 유효성 검사 수행 (FN-001 비밀번호, FN-002 이름, FN-003 이메일). 실패 시 예외 흐름 EX-004-03으로 이동 |
| 5 | 시스템 | 비밀번호를 bcrypt로 해싱 |
| 6 | 시스템 | 이메일을 소문자로 변환 |
| 7 | 시스템 | DB에 사용자 생성 시도 |
| 8 | 시스템 | 초대 토큰 존재 여부 확인. 있으면 단계 9로, 없으면 단계 10으로 진행 |
| 9 | 시스템 | 초대 수락 처리: Membership 생성 (초대에 명시된 역할), Team Membership 생성, 초대 레코드 삭제. 단계 11로 진행 |
| 10 | 시스템 | Multi-Org 활성화 여부 확인. 활성화 시 새 Organization 생성 (owner 역할). 단계 11로 진행 |

| 13 | 시스템 | Audit Log에 사용자 생성 이벤트 기록 |
| 14 | 시스템 | 성공 응답 반환 |

### 4.5 로그인 (Credentials Provider)

#### 4.5.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-005 |
| 기능명 | 로그인 (Credentials Provider) |
| 관련 요구사항 ID | FR-001-05 |
| 우선순위 | 필수 |
| 기능 설명 | 이메일/비밀번호를 사용한 사용자 인증을 수행한다. Timing Attack 방지 및 bcrypt DoS 방지를 적용한다 |

#### 4.5.2 선행 조건

- 사용자가 로그인 페이지(/auth/login)에 접근한 상태
- 사용자 계정이 DB에 존재 (비존재 시에도 일관된 응답 시간 보장)

#### 4.5.3 후행 조건

- 성공 시:
  - JWT 세션 생성 (사용자 ID, 이메일, 이메일 검증 여부 포함)
  - 마지막 로그인 시각(lastLoginAt) 업데이트
- 실패 시:
  - 적절한 에러 메시지 반환
  - 응답 시간은 성공/실패 여부와 관계없이 일정

#### 4.5.4 기본 흐름

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 사용자 | 로그인 페이지에서 이메일, 비밀번호를 입력하고 제출 |
| 2 | 시스템 | IP 기반 Rate Limit 확인 (10회/15분). 초과 시 예외 흐름 EX-005-01로 이동 |
| 3 | 시스템 | 비밀번호 길이 사전 검증: 128자 초과 시 즉시 "Invalid credentials" 반환 (bcrypt DoS 방지) |
| 4 | 시스템 | 이메일로 DB에서 사용자 조회 |
| 5 | 시스템 | Timing Attack 방지 처리: 사용자가 존재하면 실제 비밀번호 해시와 비교, 존재하지 않으면 CONTROL_HASH(bcrypt cost factor 12로 생성된 더미 해시)와 비교 수행. 두 경우 모두 동일한 연산 시간을 소모 |
| 6 | 시스템 | 사용자 상태 검증 및 비밀번호 일치 확인. 이상 없으면 단계 7로, 이상 있으면 예외 흐름으로 이동 |
| 7 | 시스템 | 인증 성공: 사용자 ID, 이메일, 이메일 검증 여부를 반환 |
| 8 | 시스템 | JWT 세션 생성 |
| 9 | 시스템 | 마지막 로그인 시각 업데이트 |

#### 4.5.5 대안 흐름

| ID | 조건 | 흐름 |
|----|------|------|
| ALT-005-01 | 2FA 활성화 사용자 | 단계 6 이후, 2FA 추가 검증 단계로 진행 (FSD-002 참조). 2FA 검증 성공 시 단계 7로 이동 |

#### 4.5.6 예외 흐름

| ID | 조건 | 에러 메시지 | 처리 |
|----|------|------------|------|
| EX-005-01 | Rate Limit 초과 (10회/15분) | "Too many requests" | HTTP 429 반환 |
| EX-005-02 | 128자 초과 비밀번호 | "Invalid credentials" | 즉시 거부 (bcrypt 연산 수행하지 않음) |
| EX-005-03 | 사용자 미존재 | "Invalid credentials" | CONTROL_HASH 비교 후 동일 에러 반환 |
| EX-005-04 | 비밀번호 미설정 (SSO 사용자) | "User has no password stored" | 요청 거부 |
| EX-005-05 | 계정 비활성화 (isActive = false) | "Your account is currently inactive" | 요청 거부 |
| EX-005-06 | 비밀번호 불일치 | "Invalid credentials" | 요청 거부 |
| EX-005-07 | 이메일 미검증 (이메일 검증 비활성화 설정이 아닌 경우) | 로그인 차단 | SignIn Callback에서 거부 |

#### 4.5.7 비즈니스 규칙

| 규칙 ID | 규칙 | 조건 | 동작 |
|---------|------|------|------|
| BR-005-01 | Timing Attack 방지 | 모든 로그인 시도 시 | 사용자 존재 여부와 관계없이 bcrypt 비교 수행. 미존재 시 CONTROL_HASH(bcrypt cost factor 12) 사용 |
| BR-005-02 | bcrypt DoS 방지 | 비밀번호 입력 시 | 128자 초과 시 bcrypt 연산 없이 즉시 거부 |
| BR-005-03 | Rate Limit | 동일 IP에서 15분 내 10회 초과 | 요청 거부 |
| BR-005-04 | 사용자 미존재/비밀번호 불일치 동일 메시지 | 사용자 미존재 또는 비밀번호 불일치 시 | 동일하게 "Invalid credentials" 반환 (User Enumeration 방지) |
| BR-005-05 | 이메일 검증 필수 | 이메일 검증 비활성화 설정이 아닌 경우 | emailVerified가 null이면 로그인 차단 |

#### 4.5.8 데이터 요구사항

**입력 데이터:**

| 필드명 | 타입 | 필수 | 유효성 검증 규칙 | 설명 |
|--------|------|------|-----------------|------|
| email | String | O | 표준 이메일 형식 | 사용자 이메일 |
| password | String | O | 최대 128자 (사전 검증) | 사용자 비밀번호 |

**출력 데이터 (인증 성공 시):**

| 필드명 | 타입 | 설명 |
|--------|------|------|
| id | String | 사용자 고유 식별자 |
| email | String | 사용자 이메일 |
| emailVerified | DateTime | 이메일 검증 완료 시각 (null 가능) |

#### 4.5.9 화면/UI 요구사항

- 로그인 폼: 이메일, 비밀번호 입력 필드
- 제출 버튼
- "비밀번호 찾기" 링크
- 회원가입 페이지 링크
- 에러 메시지 표시 영역

#### 4.5.10 비기능 요구사항

- Rate Limiting: 10회/15분 (IP 기반)
- Timing Attack 방지: constant-time 비밀번호 검증
- bcrypt DoS 방지: 128자 초과 입력 사전 차단

### 5.2 엔티티 간 관계

```
User 1──N Membership (사용자-조직 관계)
User 1──N Session (세션)
Invite 1──1 User (초대-사용자 매핑)
```

- 한 사용자는 여러 Organization에 소속 가능 (Membership을 통해)
- 한 사용자는 여러 활성 세션을 가질 수 있음
- 초대 토큰은 가입 완료 시 삭제됨

### 5.3 데이터 흐름

```
[회원가입]
사용자 입력 → 유효성 검사 → bcrypt 해싱 → User 레코드 생성
                                           → Membership 생성 (초대 시)
                                           → Organization 생성 (일반 가입 시)
                                           → 이메일 검증 메일 발송

[로그인]
사용자 입력 → Rate Limit 확인 → 사용자 조회 → bcrypt 비교 → JWT 세션 생성
                                                           → lastLoginAt 업데이트

[이메일 검증]
토큰 클릭 → JWT 검증 → 사용자 조회 → emailVerified 업데이트 → Brevo 고객 생성

[비밀번호 재설정]
토큰 + 새 비밀번호 → bcrypt 해싱 → JWT 검증 → password 업데이트
                                               → Audit Log 기록
                                               → 알림 이메일 발송
```

| 로그인 응답 시간 | constant-time | 사용자 존재 여부와 관계없이 동일한 응답 시간 보장 |

### 7.2 보안 요구사항

| 항목 | 설명 | 관련 NFR |
|------|------|---------|
| Rate Limiting | IP 기반. 로그인 10회/15분, 회원가입 30회/시간, 비밀번호 재설정 5회/시간, 이메일 검증 10회/시간 | NFR-001 |
| Timing Attack 방지 | CONTROL_HASH(bcrypt cost factor 12)를 사용한 constant-time 비밀번호 검증 | NFR-002 |
| bcrypt DoS 방지 | 비밀번호 최대 128자 제한. 128자 초과 입력 시 bcrypt 연산 없이 즉시 거부 | NFR-003 |
| User Enumeration 방지 | 회원가입 시 기존 이메일 에러 미노출. 로그인 시 사용자 미존재/비밀번호 불일치 동일 메시지 | - |
| 비밀번호 해싱 | bcrypt 단방향 해싱. 평문 비밀번호 저장 금지 | - |
| Audit Logging | 사용자 생성, 비밀번호 재설정, 로그인 시도/성공/실패 이벤트 기록 | NFR-004 |
