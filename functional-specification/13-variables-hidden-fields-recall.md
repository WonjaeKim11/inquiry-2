## 2. 개요

### 2.1 목적

본 문서는 Inquiry 설문 시스템에서 데이터를 동적으로 관리하고 활용하기 위한 세 가지 핵심 메커니즘(변수, 히든 필드, 리콜)에 대한 상세 기능 명세를 정의한다. 이 세 기능은 조건부 로직 엔진(FSD-012)과 긴밀하게 연동되어 동적 설문 경험을 구현하는 기반이 된다.

### 2.2 범위

**포함 범위 (In-scope)**:
- Survey 변수(text/number) 정의 및 기본값 관리
- 변수 이름 유효성 검증 규칙
- Hidden Field 생성, 삭제, 유효성 검증
- 금지 ID 목록 및 제한 규칙
- Recall 패턴(`#recall:id/fallback:val#`) 파싱 및 치환
- Recall에서 응답값, 변수값, 히든 필드값 해석
- 날짜, 배열 등 특수 값의 포매팅
- 설문 에디터에서의 Recall UI 표시

**제외 범위 (Out-of-scope)**:
- 변수의 외부 API 동기화
- Hidden Field 값의 서버 사이드 동적 생성
- Recall의 실시간 미리보기 (에디터에서는 `@라벨` 형태로 표시)

### 2.3 대상 사용자

| 역할 | 관심 사항 |
|------|----------|
| 설문 작성자 | 스코어링, 개인화 메시지, 외부 데이터 활용 |
| 마케터 | UTM 파라미터 등 외부 데이터를 히든 필드로 수집 |
| 데이터 분석가 | 변수 기반 점수 계산 및 응답 분류 |
| 프론트엔드 개발자 | Recall 패턴 파싱 및 렌더링 구현 |

### 2.4 용어 정의

| 용어 | 정의 |
|------|------|
| Variable (변수) | 설문 내부에서 계산 로직으로 값을 누적/변경할 수 있는 텍스트 또는 숫자 타입의 동적 데이터 |
| Hidden Field (히든 필드) | URL 파라미터 또는 SDK를 통해 외부에서 주입되는 비가시적 데이터 필드. 응답자에게 보이지 않으나 응답 데이터에 포함됨 |
| Recall (리콜) | 이전 질문의 응답, 변수값, 히든 필드값을 후속 질문의 텍스트에 동적으로 삽입하는 기능 |
| Recall 패턴 | `#recall:{id}/fallback:{value}#` 형태의 문자열 패턴으로, 런타임에 실제 값으로 치환됨 |
| Fallback 값 | Recall 대상의 실제 값이 존재하지 않을 때 대신 표시되는 대체 텍스트 |
| CUID2 | 충돌 방지를 위한 고유 식별자 생성 알고리즘. 변수 ID에 사용됨 |
| FORBIDDEN_IDS | 시스템 예약어로 지정된 11개의 금지 ID 목록 |
| Recall Item | Recall 참조 대상의 메타 정보 (id, label, type)를 담는 구조체 |
| Element | 설문 내 질문 요소. 응답값을 가지며 Recall의 참조 대상이 될 수 있음 |
| Block | 설문의 논리적 단위. 하나 이상의 Element를 포함하며 조건부 로직의 단위가 됨 |
| Calculate 액션 | 조건부 로직 엔진(FSD-012)에서 변수값을 계산/업데이트하는 액션 타입 |

## 4. 상세 기능 명세

### 4.1 변수 정의 및 기본값 관리

#### 4.1.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-013-01 |
| 기능명 | 변수 정의 및 기본값 관리 |
| 관련 요구사항 ID | FR-019, AC-019-01, AC-019-02 |
| 우선순위 | 높음 |
| 기능 설명 | 설문 내에서 사용할 변수를 정의하고 타입에 따른 기본값을 설정한다. 변수는 discriminated union 구조로 number와 text 두 가지 타입을 지원한다. |

#### 4.1.2 선행 조건 (Preconditions)

- 사용자가 설문 에디터에 접근 권한을 가지고 있어야 한다.
- 설문이 생성되어 있어야 한다.

#### 4.1.3 후행 조건 (Postconditions)

- 설문 데이터 모델의 `variables` 배열에 새 변수가 추가된다.
- 변수의 기본값이 타입에 따라 자동 설정된다 (number: `0`, text: `""`).

#### 4.1.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 사용자 | 설문 에디터에서 변수 추가 버튼을 클릭한다. |
| 2 | 시스템 | 변수 타입 선택 UI를 표시한다 (number 또는 text). |
| 3 | 사용자 | 변수 타입을 선택하고 변수 이름을 입력한다. |
| 4 | 시스템 | 변수 이름에 대해 유효성 검증을 수행한다 (FN-013-02 참조). |
| 5 | 시스템 | CUID2 형식의 고유 ID를 자동 생성한다. |
| 6 | 시스템 | 타입에 따라 기본값을 설정한다: number 타입이면 `0`, text 타입이면 빈 문자열 `""`. |
| 7 | 시스템 | 변수를 설문의 `variables` 배열에 추가한다. |
| 8 | 사용자 | (선택) 기본값을 원하는 값으로 변경한다. |

#### 4.1.5 대안 흐름 (Alternative Flow)

| ID | 조건 | 동작 |
|----|------|------|
| AF-01-01 | 사용자가 기본값을 직접 지정하는 경우 | 시스템은 타입에 맞는 값인지 검증한다. number 타입이면 숫자 값만, text 타입이면 문자열 값만 허용한다. |

#### 4.1.6 예외 흐름 (Exception Flow)

| ID | 조건 | 동작 |
|----|------|------|
| EF-01-01 | 변수 이름 유효성 검증 실패 | FN-013-02의 예외 흐름을 따른다. 변수는 추가되지 않는다. |
| EF-01-02 | number 타입에 문자열 기본값 입력 | 시스템이 타입 불일치 에러를 표시한다. |

#### 4.1.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 규칙 | 조건 | 동작 |
|---------|------|------|------|
| BR-01-01 | 변수 타입은 "number" 또는 "text"만 허용 | 변수 생성 시 | 두 타입 외의 값은 거부한다. |
| BR-01-02 | number 변수의 기본값은 0 | number 타입 변수 생성 시 | value 필드를 `0`으로 초기화한다. |
| BR-01-03 | text 변수의 기본값은 빈 문자열 | text 타입 변수 생성 시 | value 필드를 `""`으로 초기화한다. |
| BR-01-04 | 변수 ID는 CUID2 형식 | 변수 생성 시 | 시스템이 CUID2 알고리즘으로 ID를 자동 생성한다. |

#### 4.1.8 데이터 요구사항

**변수 데이터 모델 (Variable)**:

| 필드명 | 타입 | 필수 | 유효성 검증 규칙 | 설명 |
|--------|------|------|-----------------|------|
| id | string | Y | CUID2 형식 | 변수 고유 식별자 |
| name | string | Y | 소문자 알파벳, 숫자, 밑줄만 허용 (정규식: `^[a-z0-9_]+$`) | 변수 이름 |
| type | "number" \| "text" | Y | 리터럴 타입 | 변수 타입 (discriminated union 구분자) |
| value | number \| string | Y | type="number"이면 number, type="text"이면 string | 변수의 현재 값 (기본값 포함) |

**출력 데이터**: 설문 데이터 모델 내 `variables` 배열에 저장된 변수 목록.

#### 4.1.9 화면/UI 요구사항

- 설문 에디터 내 변수 관리 패널에서 변수 목록을 조회, 추가, 수정, 삭제할 수 있어야 한다.
- 변수 타입 선택 시 드롭다운 또는 라디오 버튼으로 "number"와 "text"를 선택할 수 있어야 한다.
- 각 변수의 현재 값(기본값)을 인라인으로 편집할 수 있어야 한다.

#### 4.1.10 비기능 요구사항

- NFR-013-01: 변수 ID는 CUID2 형식을 준수해야 한다.
- NFR-013-02: 변수 이름은 소문자, 숫자, 밑줄만 허용해야 한다.

### 4.2 변수 이름 유효성 검증

#### 4.2.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-013-02 |
| 기능명 | 변수 이름 유효성 검증 |
| 관련 요구사항 ID | FR-019, AC-019-03 |
| 우선순위 | 높음 |
| 기능 설명 | 변수 이름이 허용된 문자 패턴을 준수하는지 검증한다. 소문자 알파벳, 숫자, 밑줄만 허용하며 대문자, 공백, 특수문자, 하이픈은 불허한다. |

#### 4.2.2 선행 조건 (Preconditions)

- 사용자가 변수 이름을 입력 또는 수정하려는 상태여야 한다.

#### 4.2.3 후행 조건 (Postconditions)

- 유효한 이름인 경우: 변수 이름이 저장된다.
- 유효하지 않은 이름인 경우: 에러 메시지가 표시되고 변수 이름이 저장되지 않는다.

#### 4.2.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 사용자 | 변수 이름 입력 필드에 값을 입력한다. |
| 2 | 시스템 | 입력값을 정규식 `^[a-z0-9_]+$`로 검증한다. |
| 3 | 시스템 | 검증 통과 시 변수 이름을 저장한다. |

#### 4.2.5 대안 흐름 (Alternative Flow)

해당 없음.

#### 4.2.6 예외 흐름 (Exception Flow)

| ID | 조건 | 동작 |
|----|------|------|
| EF-02-01 | 대문자가 포함된 경우 | 유효성 검증 실패. 에러 메시지를 표시한다. |
| EF-02-02 | 공백이 포함된 경우 | 유효성 검증 실패. 에러 메시지를 표시한다. |
| EF-02-03 | 하이픈(-) 또는 특수문자가 포함된 경우 | 유효성 검증 실패. 에러 메시지를 표시한다. |
| EF-02-04 | 빈 문자열인 경우 | 유효성 검증 실패. 에러 메시지를 표시한다. |

#### 4.2.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 규칙 | 조건 | 동작 |
|---------|------|------|------|
| BR-02-01 | 변수 이름 허용 패턴 | 변수 이름 입력/수정 시 | 소문자 알파벳(`a-z`), 숫자(`0-9`), 밑줄(`_`)만 허용한다. |
| BR-02-02 | 변수 이름 불허 문자 | 변수 이름에 대문자, 공백, 특수문자, 하이픈이 포함된 경우 | 검증을 실패 처리한다. |

#### 4.2.8 데이터 요구사항

**입력 데이터**:

| 필드명 | 타입 | 필수 | 유효성 검증 규칙 |
|--------|------|------|-----------------|
| name | string | Y | 정규식 `^[a-z0-9_]+$` 부합 여부 |

**출력 데이터**: 검증 결과 (통과/실패) 및 실패 시 에러 메시지.

| 기능 설명 | 설문 저장 시 설문 내 모든 변수의 ID가 유일한지 검증한다. 중복 ID가 발견되면 저장을 차단하고 에러를 반환한다. |

- 사용자가 설문을 저장하려는 상태여야 한다.

#### 4.3.3 후행 조건 (Postconditions)

- 검증 통과: 설문 저장이 계속 진행된다.
- 검증 실패: 설문 저장이 중단되고 에러 메시지가 반환된다.

#### 4.3.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 사용자 | 설문 저장을 요청한다. |
| 2 | 시스템 | 설문의 `variables` 배열에서 모든 변수의 `id` 값을 추출한다. |
| 3 | 시스템 | 추출된 ID 목록에서 중복 여부를 검사한다. |
| 4 | 시스템 | 중복이 없으면 다음 검증 단계로 진행한다. |

#### 4.3.5 대안 흐름 (Alternative Flow)

해당 없음.

#### 4.3.6 예외 흐름 (Exception Flow)

#### 4.3.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 규칙 | 조건 | 동작 |
|---------|------|------|------|
| BR-03-01 | 변수 ID 유일성 | 설문 저장 시 | 동일 설문 내 모든 변수 ID는 고유해야 한다. 중복 발견 시 저장을 거부한다. |

#### 4.3.8 데이터 요구사항

**입력 데이터**: 설문의 `variables` 배열 (각 변수의 `id` 필드).

**출력 데이터**: 검증 결과 (통과/실패). 실패 시 에러 메시지: `"Variable IDs must be unique"`.

### 4.4 변수값 업데이트 (Calculate 액션 연동)

#### 4.4.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-013-04 |
| 기능명 | 변수값 업데이트 (Calculate 액션 연동) |
| 관련 요구사항 ID | FR-019, FSD-012 (FR-018) |
| 우선순위 | 높음 |
| 기능 설명 | 변수값은 조건부 로직 엔진의 Calculate 액션을 통해서만 업데이트된다. 숫자 변수는 5가지 연산(add, subtract, multiply, divide, assign), 텍스트 변수는 2가지 연산(assign, concat)을 지원한다. |

#### 4.4.2 선행 조건 (Preconditions)

- 설문에 변수가 정의되어 있어야 한다.
- 조건부 로직에 Calculate 액션이 설정되어 있어야 한다.
- 설문이 실행 중이어야 한다 (런타임).

#### 4.4.3 후행 조건 (Postconditions)

- 지정된 변수의 `value` 필드가 연산 결과로 업데이트된다.
- 업데이트된 변수값은 후속 Recall 및 조건 평가에서 즉시 사용 가능하다.

#### 4.4.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | 조건부 로직 엔진이 Block의 로직 조건을 평가한다. |
| 2 | 시스템 | 조건이 충족되면 해당 로직의 액션 배열을 순회한다. |
| 3 | 시스템 | Calculate 타입 액션을 만나면 대상 변수 ID로 변수를 조회한다. |
| 4 | 시스템 | 값 소스(static, 다른 변수, element 응답값, hidden field 값)에서 연산 값을 추출한다. |
| 5 | 시스템 | 변수 타입에 맞는 연산을 수행한다. |
| 6 | 시스템 | 연산 결과를 변수의 `value` 필드에 저장한다. |

#### 4.4.5 대안 흐름 (Alternative Flow)

해당 없음.

#### 4.4.6 예외 흐름 (Exception Flow)

| ID | 조건 | 동작 |
|----|------|------|
| EF-04-01 | divide 연산에서 제수(divisor)가 0인 경우 | 연산 결과를 undefined로 처리한다. 변수값은 변경되지 않는다. |
| EF-04-02 | 대상 변수 ID가 존재하지 않는 경우 | 해당 Calculate 액션을 무시하고 다음 액션으로 진행한다. |

#### 4.4.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 규칙 | 조건 | 동작 |
|---------|------|------|------|
| BR-04-01 | 숫자 변수 연산 | type="number"인 변수 대상 | add, subtract, multiply, divide, assign 연산 허용 |
| BR-04-02 | 텍스트 변수 연산 | type="text"인 변수 대상 | assign, concat 연산만 허용 |
| BR-04-03 | 값 소스 | Calculate 액션 실행 시 | static 값, 다른 변수값, element 응답값, hidden field 값을 소스로 사용 가능 |
| BR-04-04 | 0 나누기 방지 | divide 연산에서 static 값이 0인 경우 | 설문 저장 시 유효성 검증에서 차단 (FSD-012 NFR-012-05) |

#### 4.4.8 데이터 요구사항

**숫자 변수 연산자**:

| 연산자 | 설명 | 연산 | 제약 |
|--------|------|------|------|
| add | 더하기 | value = value + operand | 없음 |
| subtract | 빼기 | value = value - operand | 없음 |
| multiply | 곱하기 | value = value * operand | 없음 |
| divide | 나누기 | value = value / operand | operand가 0이면 undefined |
| assign | 할당 | value = operand | 없음 |

**텍스트 변수 연산자**:

| 연산자 | 설명 | 연산 |
|--------|------|------|
| assign | 할당 | value = operand |
| concat | 연결 | value = value + operand |

### 4.5 히든 필드 정의 및 활성화 관리

### 4.6 히든 필드 ID 유효성 검증

#### 4.6.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-013-06 |
| 기능명 | 히든 필드 ID 유효성 검증 |
| 관련 요구사항 ID | FR-020, AC-020-01, AC-020-02 |
| 우선순위 | 높음 |
| 기능 설명 | 히든 필드 ID의 유효성을 다단계 규칙으로 검증한다. 빈 문자열, 중복, 금지 ID, 공백, 허용 외 문자 등을 검사한다. 이 검증 로직은 히든 필드뿐 아니라 Element, Variable의 ID 검증에도 공통으로 적용된다. |

#### 4.6.2 선행 조건 (Preconditions)

- 사용자가 히든 필드 ID를 입력하려는 상태여야 한다.
- 시스템이 기존 element, hidden field, variable의 ID 목록을 조회 가능해야 한다.

#### 4.6.3 후행 조건 (Postconditions)

- 유효한 ID: 히든 필드가 정상적으로 추가된다.
- 유효하지 않은 ID: 적절한 에러 메시지가 표시되고 히든 필드가 추가되지 않는다.

#### 4.6.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 사용자 | 히든 필드 ID를 입력한다. |
| 2 | 시스템 | 빈 문자열 여부를 검사한다. |
| 3 | 시스템 | 기존 ID(questions, hidden fields, variables)와 중복 여부를 대소문자 무시로 비교한다. |
| 4 | 시스템 | FORBIDDEN_IDS 목록에 포함 여부를 검사한다. |
| 5 | 시스템 | 공백 포함 여부를 검사한다. |
| 6 | 시스템 | 허용 문자 패턴(영문, 숫자, 하이픈, 밑줄)을 검사한다. |
| 7 | 시스템 | 모든 검증 통과 시 ID를 수락한다. |

#### 4.6.5 대안 흐름 (Alternative Flow)

해당 없음.

#### 4.6.6 예외 흐름 (Exception Flow)

| ID | 조건 | 에러 메시지 |
|----|------|------------|
| EF-06-01 | 빈 문자열 | "Please enter a {type} Id." |
| EF-06-02 | 기존 ID와 중복 (대소문자 무시) | "{type} ID already exists in questions, hidden fields, or variables" |
| EF-06-03 | FORBIDDEN_IDS에 포함 | "{type} ID is not allowed." |
| EF-06-04 | 공백 포함 | "{type} ID cannot contain spaces." |
| EF-06-05 | 영숫자/하이픈/밑줄 외 문자 포함 | 영문, 숫자, 하이픈, 언더스코어만 허용 |

> `{type}`은 검증 대상의 유형에 따라 "Hidden field", "Variable", "Element" 등으로 치환된다.

#### 4.6.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 규칙 | 조건 | 동작 |
|---------|------|------|------|
| BR-06-01 | FORBIDDEN_IDS 목록 | ID 검증 시 | 다음 11개 ID는 시스템 예약어로 사용 불가: `userId`, `source`, `suid`, `end`, `start`, `welcomeCard`, `hidden`, `verifiedEmail`, `multiLanguage`, `embed`, `verify` |
| BR-06-02 | ID 중복 검사 범위 | ID 검증 시 | questions(elements), hidden fields, variables의 모든 ID를 대소문자 무시하여 비교한다. |
| BR-06-03 | 허용 문자 패턴 | ID 검증 시 | 영문 대소문자(a-z, A-Z), 숫자(0-9), 하이픈(-), 밑줄(_)만 허용한다. |
| BR-06-04 | 검증 순서 | ID 검증 시 | 빈 문자열 -> 중복 -> 금지 ID -> 공백 -> 문자 패턴 순서로 검증하며, 첫 번째 실패에서 중단한다. |

#### 4.6.8 데이터 요구사항

**입력 데이터**:

| 필드명 | 타입 | 필수 | 유효성 검증 규칙 |
|--------|------|------|-----------------|
| id | string | Y | 비어있지 않음, 중복 없음, 금지 ID 아님, 공백 없음, 허용 문자만 포함 |
| type | string | Y | 검증 대상 유형 ("Hidden field" 등) |
| existingIds | string[] | Y | 기존에 등록된 모든 ID 목록 (questions + hiddenFields + variables) |

**출력 데이터**: 검증 결과 (통과/실패) 및 실패 시 에러 메시지.

| EF-07-03 | 쿼터 조건에서 사용 중 | 토스트 에러 메시지를 표시하고 삭제를 차단한다. |
| EF-07-04 | Follow-up 액션에서 사용 중 | 토스트 에러 메시지를 표시하고 삭제를 차단한다. |

#### 4.7.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 규칙 | 조건 | 동작 |
|---------|------|------|------|
| BR-07-01 | 참조 무결성 보장 | 히든 필드 삭제 시 | 로직, 리콜, 쿼터, Follow-up 4개 영역 모두에서 미참조 상태여야 삭제 가능하다. |
| BR-07-02 | 검사 순서 | 히든 필드 삭제 시 | 로직 -> 리콜 -> 쿼터 -> Follow-up 순서로 검사하며, 첫 번째 참조 발견 시 즉시 삭제를 차단한다. |
| BR-07-03 | 에러 표시 방식 | 참조가 발견된 경우 | 토스트(Toast) UI로 에러 메시지를 표시한다. |

#### 4.7.8 데이터 요구사항

**출력 데이터**: 삭제 성공 또는 실패. 실패 시 어떤 영역에서 사용 중인지 나타내는 에러 메시지.

### 4.8 히든 필드 값 설정 (URL/SDK)

#### 4.8.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-013-08 |
| 기능명 | 히든 필드 값 설정 (URL/SDK) |
| 관련 요구사항 ID | FR-020 |
| 우선순위 | 높음 |
| 기능 설명 | 히든 필드의 값은 Link Survey에서는 URL 쿼리 파라미터로, App/Website Survey에서는 SDK를 통해 프로그래밍 방식으로 주입된다. 주입된 값은 응답 데이터에 저장되어 조건부 로직 평가 및 리콜에서 사용 가능하다. |

#### 4.8.2 선행 조건 (Preconditions)

- 설문에 히든 필드가 정의되고 활성화되어 있어야 한다 (enabled = true).
- Link Survey: 설문 URL에 쿼리 파라미터가 포함되어야 한다.
- App/Website Survey: SDK가 초기화되어 히든 필드 값을 전달해야 한다.

#### 4.8.3 후행 조건 (Postconditions)

- 히든 필드의 값이 응답 데이터에 포함되어 저장된다.
- 설정된 값은 조건부 로직 평가의 피연산자로 사용 가능하다.
- 설정된 값은 리콜 패턴에서 참조 가능하다.

#### 4.8.4 기본 흐름 (Basic Flow)

**URL 파라미터 방식 (Link Survey)**:

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 응답자 | 히든 필드가 포함된 URL로 설문에 접속한다. (예: `https://survey.example.com/s/abc?utm_source=email&campaign=summer`) |
| 2 | 시스템 | URL의 쿼리 파라미터를 파싱한다. |
| 3 | 시스템 | 설문에 정의된 `fieldIds`와 파라미터 키를 매칭한다. |
| 4 | 시스템 | 매칭된 파라미터의 값을 해당 히든 필드의 값으로 설정한다. |
| 5 | 시스템 | 설정된 값을 응답 데이터에 포함시켜 저장한다. |

**SDK 방식 (App/Website Survey)**:

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 개발자 | SDK 초기화 또는 설문 호출 시 히든 필드 값을 프로그래밍 방식으로 전달한다. |
| 2 | 시스템 | 전달된 히든 필드 데이터를 설문의 `fieldIds`와 매칭한다. |
| 3 | 시스템 | 매칭된 값을 해당 히든 필드의 값으로 설정한다. |
| 4 | 시스템 | 설정된 값을 응답 데이터에 포함시켜 저장한다. |

#### 4.8.5 대안 흐름 (Alternative Flow)

| ID | 조건 | 동작 |
|----|------|------|
| AF-08-01 | URL 파라미터에 설문에 정의되지 않은 키가 포함된 경우 | 해당 파라미터는 무시한다. |
| AF-08-02 | 설문에 정의된 히든 필드에 대응하는 URL 파라미터가 없는 경우 | 해당 히든 필드의 값은 비어있는 상태로 유지된다. |

#### 4.8.6 예외 흐름 (Exception Flow)

해당 없음. 값이 전달되지 않은 히든 필드는 빈 값으로 처리된다.

#### 4.8.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 규칙 | 조건 | 동작 |
|---------|------|------|------|
| BR-08-01 | URL 파라미터 매칭 | Link Survey 접속 시 | URL 쿼리 파라미터의 키가 `fieldIds`에 포함된 경우에만 값을 설정한다. |
| BR-08-02 | 값 저장 | 히든 필드 값 설정 시 | 값은 응답 데이터에 저장되어 로직 평가 및 리콜에서 사용 가능하다. |
| BR-08-03 | 값 타입 | 히든 필드 값 설정 시 | 히든 필드의 값은 문자열(string) 타입이다. |

#### 4.8.8 데이터 요구사항

**입력 데이터**:

| 소스 | 형태 | 예시 |
|------|------|------|
| URL 파라미터 | `?{fieldId}={value}` | `?utm_source=email&campaign=summer` |
| SDK | 프로그래밍 방식 객체 전달 | `{ utm_source: "email", campaign: "summer" }` |

**출력 데이터**: 응답 데이터 내 히든 필드 값 (key-value 쌍).

### 4.9 Recall 패턴 파싱

#### 4.9.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-013-09 |
| 기능명 | Recall 패턴 파싱 |
| 관련 요구사항 ID | FR-021, AC-021-01 |
| 우선순위 | 높음 |
| 기능 설명 | 텍스트 내에 포함된 `#recall:{id}/fallback:{value}#` 패턴을 인식하고 파싱하여 ID, fallback 값 등의 정보를 추출한다. 4가지 파싱 기능을 제공한다: 첫 번째 ID 추출, 모든 ID 추출, fallback 값 추출, 전체 recall 정보 추출. |

#### 4.9.2 선행 조건 (Preconditions)

- 파싱 대상 텍스트가 존재해야 한다.

#### 4.9.3 후행 조건 (Postconditions)

- 텍스트 내 recall 패턴의 ID 및 fallback 값이 추출된다.
- recall 패턴이 없는 경우 빈 결과가 반환된다.

#### 4.9.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | 대상 텍스트를 입력받는다. |
| 2 | 시스템 | 정규식으로 `#recall:{id}/fallback:{value}#` 패턴을 검색한다. |
| 3 | 시스템 | 매칭된 패턴에서 `{id}` 부분 (영문 대소문자, 숫자, 밑줄, 하이픈)과 `{value}` 부분을 추출한다. |
| 4 | 시스템 | 요청된 파싱 유형에 따라 결과를 반환한다. |

**4가지 파싱 유형**:

| 파싱 유형 | 설명 | 반환값 |
|-----------|------|--------|
| 첫 번째 recall ID 추출 | 텍스트에서 첫 번째 recall 참조 ID를 추출 | string 또는 null |
| 모든 recall ID 추출 | 텍스트에서 모든 recall 참조 ID를 추출 | string[] |
| fallback 값 추출 | recall 패턴에서 fallback 값을 추출 | string 또는 null |
| 전체 recall 정보 추출 | ID와 fallback 값을 포함한 전체 recall 정보 추출 | { id: string, fallback: string }[] |

#### 4.9.5 대안 흐름 (Alternative Flow)

| ID | 조건 | 동작 |
|----|------|------|
| AF-09-01 | 텍스트에 recall 패턴이 없는 경우 | 빈 결과를 반환한다 (ID 추출: null/빈 배열, fallback: null). |

#### 4.9.6 예외 흐름 (Exception Flow)

| ID | 조건 | 동작 |
|----|------|------|
| EF-09-01 | 파싱 중 while 루프에서 recall 정보가 null인 경우 | 즉시 루프를 중단한다 (NFR-013-04). |

#### 4.9.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 규칙 | 조건 | 동작 |
|---------|------|------|------|
| BR-09-01 | Recall 패턴 형식 | 텍스트 파싱 시 | `#recall:{id}/fallback:{value}#` 형식만 유효한 recall 패턴으로 인식한다. |
| BR-09-02 | ID 허용 문자 | recall 패턴의 {id} 부분 | 영문 대소문자(a-z, A-Z), 숫자(0-9), 밑줄(_), 하이픈(-)만 허용한다. |
| BR-09-03 | 파싱 안전성 | while 루프에서 recall 정보 추출 시 | null이 반환되면 즉시 중단하여 무한 루프를 방지한다. |

#### 4.9.8 데이터 요구사항

**입력 데이터**:

| 필드명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| text | string | Y | recall 패턴이 포함된 텍스트 |

**출력 데이터**:

| 파싱 유형 | 반환 타입 | 예시 |
|-----------|----------|------|
| 첫 번째 ID | string \| null | `"abc123"` |
| 모든 ID | string[] | `["abc123", "my_field"]` |
| fallback 값 | string \| null | `"고객님"` |
| 전체 정보 | { id: string, fallback: string }[] | `[{ id: "abc123", fallback: "고객님" }]` |

### 4.10 Recall 값 해석 및 치환

#### 4.10.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-013-10 |
| 기능명 | Recall 값 해석 및 치환 |
| 관련 요구사항 ID | FR-021, AC-021-02, AC-021-03, AC-021-04, AC-021-05 |
| 우선순위 | 높음 |
| 기능 설명 | 설문 실행 시 텍스트 내 recall 패턴을 실제 값으로 치환한다. 값 해석 우선순위(변수 -> 응답 데이터 -> fallback)를 따르며, 날짜 및 배열 등 특수 값에 대한 포매팅을 적용한다. |

#### 4.10.2 선행 조건 (Preconditions)

- 설문이 실행 중이어야 한다 (런타임).
- recall 패턴이 포함된 텍스트가 렌더링 대상이어야 한다.
- 변수 데이터, 응답 데이터, 히든 필드 데이터가 조회 가능해야 한다.

#### 4.10.3 후행 조건 (Postconditions)

- 텍스트 내 모든 recall 패턴이 실제 값 또는 fallback 값으로 치환된다.

#### 4.10.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | 텍스트에서 recall 패턴을 파싱한다 (FN-013-09 사용). |
| 2 | 시스템 | 추출된 각 recall ID에 대해 다음 우선순위로 값을 조회한다: |
| 2-1 | 시스템 | **1순위 - 변수(Variables)**: 해당 ID를 가진 변수의 현재 값을 확인한다. |
| 2-2 | 시스템 | **2순위 - 응답 데이터(Response Data)**: 해당 ID를 가진 element의 응답값 또는 히든 필드값을 확인한다. |
| 2-3 | 시스템 | **3순위 - Fallback 값**: 값이 없으면 recall 패턴에 정의된 fallback 값을 사용한다. |
| 3 | 시스템 | 조회된 값에 대해 특수 포매팅을 적용한다 (날짜, 배열 등). |
| 4 | 시스템 | 원본 텍스트의 recall 패턴을 치환된 값으로 대체한다. |

#### 4.10.5 대안 흐름 (Alternative Flow)

| ID | 조건 | 동작 |
|----|------|------|
| AF-10-01 | 값이 날짜 문자열인 경우 | "1st January 2024" 형태의 서수 날짜 형식으로 변환한다. |
| AF-10-02 | 값이 배열인 경우 | 빈 값을 제거한 후 쉼표(,)로 연결하여 단일 문자열로 변환한다. |
| AF-10-03 | fallback 값에 "nbsp"가 포함된 경우 | "nbsp"를 공백 문자(` `)로 치환한다. |

#### 4.10.6 예외 흐름 (Exception Flow)

| ID | 조건 | 동작 |
|----|------|------|
| EF-10-01 | 변수, 응답 데이터, fallback 모두 값이 없는 경우 | recall 패턴을 빈 문자열로 치환한다. |

#### 4.10.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 규칙 | 조건 | 동작 |
|---------|------|------|------|
| BR-10-01 | 값 해석 우선순위 | recall 패턴 치환 시 | 1순위: 변수(Variables) -> 2순위: 응답 데이터(Response Data) -> 3순위: Fallback 값 |
| BR-10-02 | 날짜 포매팅 | 값이 날짜 문자열인 경우 | "1st January 2024" 형태로 변환한다. 서수 접미사(st, nd, rd, th)를 일(day)에 적용한다. |
| BR-10-03 | 배열 포매팅 | 값이 배열인 경우 | 빈 값(빈 문자열, null, undefined)을 제거한 후 쉼표(`, `)로 연결한다. |
| BR-10-04 | nbsp 치환 | fallback 값에 "nbsp" 문자열이 있는 경우 | 공백 문자(` `)로 치환한다. |

#### 4.10.8 데이터 요구사항

**입력 데이터**:

| 필드명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| text | string | Y | recall 패턴이 포함된 텍스트 |
| variables | Variable[] | Y | 설문에 정의된 변수 목록 (현재 값 포함) |
| responseData | Record<string, any> | Y | 응답자의 현재 응답 데이터 |

**출력 데이터**: recall 패턴이 실제 값으로 치환된 문자열.

### 4.11 Recall 에디터 표시

#### 4.11.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-013-11 |
| 기능명 | Recall 에디터 표시 |
| 관련 요구사항 ID | FR-021, AC-021-07 |
| 우선순위 | 중간 |
| 기능 설명 | 설문 에디터에서 recall 패턴(`#recall:id/fallback:val#`)을 사용자 친화적 형태로 변환하여 표시한다. 슬래시 모드(`/라벨명\`)와 앳 모드(`@라벨명`)를 지원한다. |

#### 4.11.2 선행 조건 (Preconditions)

- 설문 에디터가 활성화되어 있어야 한다.
- 텍스트에 recall 패턴이 포함되어 있어야 한다.

#### 4.11.3 후행 조건 (Postconditions)

- 에디터에서 recall 패턴이 읽기 쉬운 라벨 형태로 표시된다.

#### 4.11.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 시스템 | 텍스트에서 recall 패턴을 파싱한다. |
| 2 | 시스템 | 각 recall ID의 참조 대상 유형을 확인한다. |
| 3 | 시스템 | 라벨 해석 우선순위에 따라 표시용 라벨을 결정한다: |
| 3-1 | 시스템 | Hidden Field인 경우: field ID를 그대로 라벨로 사용한다. |
| 3-2 | 시스템 | Survey Element인 경우: 해당 질문의 headline을 라벨로 사용한다 (HTML 태그 제거 후). |
| 3-3 | 시스템 | Variable인 경우: 변수의 name을 라벨로 사용한다. |
| 4 | 시스템 | 표시 모드에 따라 변환한다: |
| 4-1 | 시스템 | 슬래시 모드: `/라벨명\` 형태로 변환한다. |
| 4-2 | 시스템 | 앳 모드: `@라벨명` 형태로 변환한다. |

### 4.12 Recall 에디터-저장 형식 변환

#### 4.12.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-013-12 |
| 기능명 | Recall 에디터-저장 형식 변환 |
| 관련 요구사항 ID | FR-021 |
| 우선순위 | 중간 |
| 기능 설명 | 에디터에서 사용자가 입력/표시하는 `@라벨명` 형태의 텍스트를 저장 시 `#recall:id/fallback:val#` 패턴으로 역변환한다. 이를 통해 에디터 UI와 저장 데이터 간의 형식 차이를 해소한다. |

#### 4.12.2 선행 조건 (Preconditions)

- 에디터에서 `@라벨명` 형태의 recall 참조가 포함된 텍스트가 존재해야 한다.
- 설문 데이터(히든 필드, elements, variables)가 조회 가능해야 한다.

#### 4.12.3 후행 조건 (Postconditions)

- `@라벨명` 형태가 `#recall:id/fallback:val#` 패턴으로 변환되어 저장된다.

#### 4.12.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|
| 1 | 사용자 | 설문을 저장한다. |
| 2 | 시스템 | 텍스트 내 `@라벨명` 형태의 recall 참조를 탐색한다. |
| 3 | 시스템 | 각 라벨에 대응하는 recall 대상의 ID를 조회한다. |
| 4 | 시스템 | `#recall:{id}/fallback:{fallback_value}#` 형식으로 변환한다. |
| 5 | 시스템 | 변환된 텍스트를 저장한다. |

#### 4.12.5 대안 흐름 (Alternative Flow)

해당 없음.

#### 4.12.6 예외 흐름 (Exception Flow)

| ID | 조건 | 동작 |
|----|------|------|
| EF-12-01 | 라벨에 대응하는 대상을 찾을 수 없는 경우 | 해당 recall 참조를 제거하거나 원본 텍스트로 유지한다. |

#### 4.12.7 비즈니스 규칙 (Business Rules)

| 규칙 ID | 규칙 | 조건 | 동작 |
|---------|------|------|------|
| BR-12-01 | 양방향 변환 | 에디터 로드/저장 시 | 저장 형식(`#recall:id/fallback:val#`) <-> 에디터 형식(`@라벨명`) 간 양방향 변환이 가능해야 한다. |

#### 4.12.8 데이터 요구사항

**입력 데이터**: `@라벨명` 형태가 포함된 에디터 텍스트, 설문의 히든 필드/elements/variables 데이터.

**출력 데이터**: `#recall:id/fallback:val#` 형식으로 변환된 저장용 텍스트.

### 4.13 빈 Fallback 값 검증

#### 4.13.1 기능 개요

| 항목 | 내용 |

| 관련 요구사항 ID | FR-021, AC-021-06 |
| 우선순위 | 중간 |
| 기능 설명 | headline과 subheader에서 fallback 값이 비어있는 recall 패턴을 탐지하여 사용자에게 경고한다. 빈 fallback은 실행 시 빈 텍스트로 표시되어 사용자 경험을 저해할 수 있으므로 사전에 경고한다. |

#### 4.13.3 후행 조건 (Postconditions)

- 빈 fallback을 가진 recall 패턴이 포함된 element가 식별되어 사용자에게 경고가 표시된다.

#### 4.13.4 기본 흐름 (Basic Flow)

| 단계 | 주체 | 동작 |
|------|------|------|

| 2 | 시스템 | 각 element의 headline과 subheader에서 recall 패턴을 파싱한다. |
| 3 | 시스템 | 파싱된 recall 패턴 중 fallback 값이 빈 문자열인 것을 탐지한다. |
| 4 | 시스템 | 빈 fallback이 발견된 element를 반환한다. |
| 5 | 시스템 | 사용자에게 해당 element에 대한 경고를 표시한다. |

#### 4.13.5 대안 흐름 (Alternative Flow)

| ID | 조건 | 동작 |
|----|------|------|
| AF-13-01 | 빈 fallback을 가진 recall 패턴이 없는 경우 | 경고를 표시하지 않는다. |

#### 4.13.6 예외 흐름 (Exception Flow)

#### 4.13.7 비즈니스 규칙 (Business Rules)

### 4.14 중첩 Recall 처리

#### 4.14.1 기능 개요

| 항목 | 내용 |
|------|------|
| 기능 ID | FN-013-14 |
| 기능명 | 중첩 Recall 처리 |

#### 4.14.3 후행 조건 (Postconditions)

- 중첩된 recall 패턴이 `"___"`로 대체되어 무한 루프 없이 처리가 완료된다.

| 단계 | 주체 | 동작 |
|------|------|------|

| 2 | 시스템 | 조회된 라벨 내에 또 다른 recall 패턴이 존재하는지 확인한다. |
| 3 | 시스템 | 중첩 recall 패턴이 발견되면 해당 패턴을 `"___"`(밑줄 3개)로 대체한다. |
| 4 | 시스템 | 대체된 텍스트를 최종 결과로 반환한다. |

#### 4.14.5 대안 흐름 (Alternative Flow)

| ID | 조건 | 동작 |
|----|------|------|
| AF-14-01 | 중첩 recall이 없는 경우 | 정상적으로 recall 값 치환을 수행한다 (FN-013-10 기본 흐름). |

#### 4.14.6 예외 흐름 (Exception Flow)

**출력 데이터**: 중첩 recall이 `"___"`로 대체된 최종 텍스트.

### 4.15 Recall 텍스트 축약

#### 4.15.1 기능 개요

| 4 | 시스템 | 뒤 10자를 추출하여 연결한다. |
| 5 | 시스템 | 최종 축약된 텍스트를 반환한다. |

## 5. 데이터 모델

### 5.1 주요 엔티티 정의

#### 5.1.1 Variable (변수)

```
Variable (Discriminated Union)
├── NumberVariable
│   ├── id: string (CUID2)
│   ├── name: string (소문자, 숫자, 밑줄)
│   ├── type: "number"
│   └── value: number (기본값: 0)
└── TextVariable
    ├── id: string (CUID2)
    ├── name: string (소문자, 숫자, 밑줄)
    ├── type: "text"
    └── value: string (기본값: "")
```

#### 5.1.2 HiddenFields (히든 필드)

```
HiddenFields
├── enabled: boolean
└── fieldIds: string[]
    └── 각 ID: 영문, 숫자, 하이픈, 밑줄 (FORBIDDEN_IDS 제외)
```

#### 5.1.3 RecallItem (리콜 아이템)

```
RecallItem
├── id: string (참조 대상 ID)
├── label: string (표시용 라벨)
└── type: "hiddenField" | "element" | "variable"
```

#### 5.1.4 FORBIDDEN_IDS (금지 ID 목록)

```
FORBIDDEN_IDS = [
  "userId", "source", "suid", "end", "start",
  "welcomeCard", "hidden", "verifiedEmail",
  "multiLanguage", "embed", "verify"
]  // 총 11개
```

### 5.2 엔티티 간 관계

```
Survey (설문)
├── variables: Variable[]          -- 1:N 관계
├── hiddenFields: HiddenFields     -- 1:1 관계
├── blocks: Block[]
│   └── elements: Element[]        -- 블록 내 질문 요소
│       └── headline/subheader에 Recall 패턴 포함 가능
└── welcomeCard / endingCards
    └── headline/subheader에 Recall 패턴 포함 가능

Recall Pattern → Variable (참조)
Recall Pattern → Element 응답값 (참조)
Recall Pattern → HiddenField 값 (참조)

Calculate Action (FSD-012) → Variable (값 업데이트)
Logic Condition (FSD-012) → Variable, HiddenField (피연산자로 참조)
```

### 5.3 데이터 흐름

```
[외부 데이터 주입]
    │
    ├── URL Query Params ──→ Hidden Field 값 설정
    └── SDK 전달          ──→ Hidden Field 값 설정
                                    │
                                    v
[설문 실행 (런타임)]
    │
    ├── 응답자 입력 ──→ Element 응답값 저장
    │                         │
    ├── 조건부 로직 평가 ──→ Calculate 액션 ──→ Variable 값 업데이트
    │                                              │
    └── Recall 치환 엔진 <────────────────────────┘
         │
         ├── 1순위: Variable 값 조회
         ├── 2순위: Response Data (Element 응답/Hidden Field) 조회
         └── 3순위: Fallback 값 사용
              │
              v
         [치환된 텍스트 렌더링]
```

## 6. 인터페이스 명세

### 6.1 외부 시스템 연동

| 연동 대상 | 연동 방식 | 데이터 흐름 | 설명 |
|-----------|----------|------------|------|
| URL Query Parameter | HTTP GET | 외부 -> Hidden Field | Link Survey 접속 시 URL 파라미터로 히든 필드 값 주입 |
| JavaScript SDK | API 호출 | 외부 -> Hidden Field | App/Website Survey에서 SDK를 통해 히든 필드 값 주입 |
| 조건부 로직 엔진 (FSD-012) | 내부 모듈 | 양방향 | Calculate 액션으로 변수값 업데이트, 변수/히든 필드를 조건 피연산자로 사용 |
| 설문 에디터 | 내부 모듈 | 양방향 | Recall UI 표시/편집, 변수/히든 필드 관리 |

### 6.2 API 명세

본 기능은 독립적인 외부 API를 제공하지 않는다. 변수, 히든 필드, 리콜은 설문 데이터 모델의 일부로서 설문 CRUD API를 통해 관리된다.

**설문 저장 시 검증 항목**:
- 변수 ID 유일성 검증
- 변수 이름 유효성 검증
- 히든 필드 ID 유효성 검증
- 빈 Fallback 값 경고

| 변수값 조회 | O(1) | 변수 ID 기반 직접 조회 |

### 7.2 보안 요구사항

| 항목 | 기준 | 설명 |
|------|------|------|
| FORBIDDEN_IDS 강제 | 시스템 예약어 보호 | 11개 금지 ID를 통해 시스템 내부 키와의 충돌 방지 |
| HTML 태그 제거 | XSS 방지 | Recall 라벨 생성 시 Element headline에서 HTML 태그를 제거하여 스크립트 삽입 방지 (NFR-013-06) |

### 7.3 가용성 요구사항

| 항목 | 기준 | 설명 |
|------|------|------|
| Recall 파싱 안전성 | 무한 루프 방지 | while 루프에서 null 반환 시 즉시 중단 (NFR-013-04) |
| 중첩 Recall 안전성 | 무한 루프 방지 | 중첩 recall을 `"___"`로 대체하여 재귀 해석 차단 (NFR-013-05) |

## 8. 제약사항 및 가정

### 8.1 기술적 제약사항

| 항목 | 제약 값 | 설명 |
|------|--------|------|
| 변수 ID 형식 | CUID2 | 충돌 방지를 위한 고유 식별자 형식 |
| 변수 이름 패턴 | `^[a-z0-9_]+$` | 소문자, 숫자, 밑줄만 허용 |
| Hidden Field ID 패턴 | `^[a-zA-Z0-9_-]+$` | 영문, 숫자, 하이픈, 밑줄만 허용 |
| Hidden Field ID 중복 검사 | 대소문자 무시 (case-insensitive) | 기존 모든 ID와 비교 |
| Recall ID 패턴 | `^[a-zA-Z0-9_-]+$` | 영문 대소문자, 숫자, 밑줄, 하이픈 허용 |
| 금지 ID 개수 | 11개 | userId, source, suid, end, start, welcomeCard, hidden, verifiedEmail, multiLanguage, embed, verify |
| Recall 텍스트 축약 기준 | 25자 | 25자 이상 시 축약 |
| Recall 축약 형태 | 앞 10자 + "..." + 뒤 10자 | 총 23자로 축약 |
| 숫자 변수 기본값 | 0 | 생성 시 자동 설정 |
| 텍스트 변수 기본값 | 빈 문자열 (`""`) | 생성 시 자동 설정 |

### 8.2 비즈니스 제약사항

| 항목 | 제약 | 설명 |
|------|------|------|
| 변수값 업데이트 경로 | Calculate 액션만 가능 | 변수값은 조건부 로직의 Calculate 액션을 통해서만 업데이트 가능. 직접 수정 불가. |
| 히든 필드 삭제 | 참조 무결성 필수 | 로직, 리콜, 쿼터, Follow-up에서 미사용 상태여야 삭제 가능 |
| 라이선스 | Community | 본 기능은 Community 라이선스에 포함 (무료 사용 가능) |

### 8.3 가정사항

| 항목 | 가정 내용 |
|------|----------|
| CUID2 유일성 | CUID2 알고리즘이 동일 설문 내에서 중복 없는 고유 ID를 생성한다고 가정한다. |
| 클라이언트 사이드 Recall 처리 | Recall 패턴의 파싱 및 치환은 클라이언트 사이드(설문 런타임)에서 수행된다고 가정한다. |
| URL 파라미터 인코딩 | URL 쿼리 파라미터는 표준 URL 인코딩/디코딩을 따른다고 가정한다. |
| 단일 설문 컨텍스트 | 변수와 히든 필드의 스코프는 단일 설문 내로 한정된다고 가정한다. |
